### 3. 论文主结论降调

今天最关键的写作决策是：不再把 TC-PPO 写成“全面提升 Vanilla PPO 的稳定性方法”，而是写成：

- 一个物理启发的接触一致性门控机制；
- 可以揭示任务间 FTC 分布差异；
- 对阈值非常敏感；
- 在不同任务上进入不同工作模式；
- 当前主要贡献是诊断与机制验证，而不是全面性能领先。

对应到 [[Mode A-B-C Taxonomy]]：

- **Mode C**：FTC 自然高于阈值，gate 不触发，TC-PPO ≈ Vanilla PPO。
- **Mode B**：FTC 自然偏低，gate 长期激活，探索被压制，可能低于 Vanilla PPO，例如 Truck。
- **Mode A**：理想状态是 gate 在接触混乱时短暂激活、恢复后关闭；但目前没有完整闭环证据支撑“已解决”。

这使论文叙事从“成功故事”转为“诚实的方法论验证 + 泛化诊断”。

## 新洞察

### 1. TC-PPO 当前最大的价值是“诊断”，不是“提升”

真实数据说明，TC-PPO 目前并没有形成“全面优于 Vanilla PPO”的证据。相反，它的价值更像是：

- 用 [[FTC Metric]] 暴露不同任务的接触规律性；
- 用 gate 激活模式识别任务属于 Mode B 还是 Mode C；
- 说明全局阈值的局限；
- 为 future per-task calibration 提供清晰动机。

这使论文从“性能论文”转向“机制诊断 + 方法论论文”，反而更可信。

### 2. 全局 τ 阈值无法同时适配 Truck 和 Walk

从今天整理的数据看：

- Truck 的自然 FTC 很低，约在 `0.33–0.39`，导致 τ_high 如果固定为 `0.42`，gate 很难关闭，进入 Mode B。
- Walk 的 FTC 更高，在 calibrated 设置下可不触发或少触发，表现接近 Vanilla。
- Bookshelf Hard / Bookshelf Simple 的 FTC 更高，属于典型 Mode C。

这说明单一全局阈值存在结构性矛盾：为了避免 walk / bookshelf 误触发，τ 不能太高；为了让 truck 能恢复关闭 gate，τ_high 又不能太高。两者无法由同一个固定值同时满足。

### 3. 动态 τ 不应直接写成 τ = k·FTC + b

晚上讨论了动态 τ 的设想。用户提出 τ 是否可以类似 `y = kx + b`，即随 FTC 线性变化。

分析后形成初步判断：

- 直接让 `τ = k·FTC + b` 有问题，因为 τ 如果实时跟着当前 FTC 变化，可能导致阈值和观测量一起漂移，gate 判定失去稳定参考。
- 更合理的是在 warm-up 阶段估计每个任务的 FTC 稳态分布，然后根据该分布设阈值。
- 推荐形式不是简单的 `k·FTC+b`，而是统计自适应：

\[
\tau_{\mathrm{low}} = \mu_{\mathrm{FTC}} - 2\sigma_{\mathrm{FTC}}
\]

\[
\tau_{\mathrm{high}} = \mu_{\mathrm{FTC}} - 0.5\sigma_{\mathrm{FTC}}
\]

这种形式仍是线性的，但变量是任务级 FTC 分布的均值和方差，而不是当前瞬时 FTC。

其优点：

- 对高 FTC 任务自动设更高阈值。
- 对低 FTC 任务，如 truck，自动降低 τ_high，避免 gate 永久锁死。
- 方差大的任务会有更宽容的触发边界。
- 比固定 τ 更符合 [[Mode A-B-C Taxonomy]] 的目标。

### 4. Figure 5 是解释论文新叙事的关键图

Figure 5 从占位图变成真实 FTC boxplot 后，论文叙事更完整。它直接支持：

- 为什么某些任务是 Mode C；
- 为什么 truck / powerlift / kitchen 等任务更接近低 FTC 区域；
- 为什么全局 τ 是启发式而不是最终答案；
- 为什么需要 per-task adaptive calibration。

这张图比单纯列 reward 更能解释 TC-PPO 的机制行为。
