### 2. 按文档重写扰动实验脚本

随后根据用户提供的《噪声+扰动.docx》和《RL训练资源利用优化指南》重新检查脚本，发现原脚本偏差较大：

- 原先 `force_scale=2.0` 远低于文档想表达的实际力强度。
- 原计划没有严格覆盖 P0 / P1 / P2 矩阵。
- 没有完整 pre-flight checklist。
- 没有按文档要求控制并发资源。
- P1 / P2 的分组设计与文档不一致。

工程上确认了一个关键事实：

- `PerturbationWrapper` 实际作用 body 是 **pelvis**，质量约 **5.39 kg**。
- 因此实际力和 `force_scale` 的关系为：

\[
F_{\mathrm{N}} = \text{force\_scale} \times 5.39
\]

据此得到：

- Stair 150N → `force_scale ≈ 27.83`
- Slide 200N → `force_scale ≈ 37.11`

脚本层面完成并验证：

- `run_robust_experiments.sh`
- `run_all_batches.sh`
- 加入 `MUJOCO_EGL_DEVICE_ID` + `CUDA_VISIBLE_DEVICES`
- 使用真实 warmup 路径，不再依赖失效的 `LATEST_WARMUP`
- 加入 pre-flight checklist
- 固定 `n_envs=4`
- 每 run 5M steps
- 初始布局为每批 6 进程，GPU0/1/2 各 2 个

但随后方法论上又进一步修正：直接用 150N/200N 做主实验仍然可能过强或过弱，因此决定先做 adaptive scan。

### 3. 重新定义“崩溃阈值”：从 15% drop 改为双指标

围绕“什么叫策略崩溃”进行了文献方法论讨论。结论：

- 鲁棒 RL / sim-to-real 文献中并没有广泛接受的 canonical 标准，如“15% reward drop = collapse”。
- 更常见的做法是：
  - success / failure rate
  - episode termination / fall rate
  - worst-percentile / CVaR
  - reward curve under perturbation
  - 对照 baseline 是否接近失效状态

因此今天决定把 [[FTC Metric]] 相关鲁棒性实验的崩溃判据改成更稳健的双指标：

1. **主指标**：survival / success rate 明显下降，例如 survival rate < 50%。
2. **辅指标**：reward drop 约 20–35%，作为“moderate-to-severe but not impossible”的区间。

这比原先“15% drop”更能防止 reviewer 质疑。

### 4. Adaptive Two-stage Scan：粗扫 + 形态判断 + 细扫

用户指出：噪声强度不一定越大 reward 越低，可能存在非单调曲线。这个判断非常重要，今天正式纳入实验规划。

已经观察到的 Slide 旧 scan 数据支持这个担忧：

| force_scale | 实际力约 N | 早先 reward 趋势 |
|---|---:|---|
| 0.5 | 2.7N | 约 27.2 |
| 1.0 | 5.4N | 约 27.3 |
| 2.0 | 10.8N | 约 28.5，上升 |
| 3.0 | 16.2N | 约 27.7 |

这说明小扰动可能具有 noise-as-regularization 效应，而不是单调破坏。

最终确定 [[Critical Perturbation Threshold Testing]] / Phase 0 采用 adaptive two-stage scan：

- Stage 1：Vanilla PPO 粗扫大档位。
- Stage 2：根据曲线形态补点。

曲线形态分类：

- A：严格单调下降 → 在 20–35% drop 区间补点。
- B：早期反升后下降 → 先确认峰值，再在下降区找 F\*。
- C：平台后悬崖 → 在 cliff 前后补点。
- D：全程无崩溃 → 扩大上限。
- E：第一点就崩 → 向下细扫。

最终 Phase 0 Stage 1 扫描设计：

| 任务 | 噪声 | scan 点 |
|---|---|---|
| Stair | force | 30 / 70 / 110 / 150 N |
| Slide | force | 40 / 90 / 140 / 190 / 240 N |
| Stair | action noise | 0.05 / 0.13 / 0.21 / 0.30 |
| Slide | action noise | 0.13 / 0.21 / 0.30 |
| Stair | delay | 1 / 2 / 3 / 4 / 5 steps |
| Slide | delay | 2 / 3 / 4 / 5 steps |

合计约 **25 vanilla runs × 2M steps × 1 seed**。

## 新洞察

### 1. “扰动实验是否启动”必须有多层验证

今天最大的工程教训：

> 只看到 PID 或 nohup 启动不代表实验跑对了。

以后启动训练必须至少检查：

1. 进程 30s 后仍存活。
2. GPU 显存和 util 符合预期。
3. 日志中出现正确 env / algo / seed / perturbation tag。
4. 参数真正进入 `PerturbationWrapper`。
5. reward 区间与任务难度一致。
6. 目录命名能追溯扰动强度，因为 `run_config.json` 不保存扰动参数。

这应写入 [[TC-PPO Algorithm]] 实验规范。

### 2. force_scale 必须转成物理单位，否则论文不可解释

旧设计直接讨论 `force_scale=0.5/1/2/3`，可复现性和物理解释都弱。今天明确：

\[
F_{\mathrm{N}} = \text{force\_scale} \times m_{\mathrm{pelvis}}
\]

且当前 H1hand wrapper 实际使用 pelvis，约 5.39kg。

这使得实验可以用 30N / 70N / 110N / 150N 等物理量描述，论文更可信。

### 3. 噪声-性能曲线不一定单调

旧假设是扰动越强 reward 越低。今天确认这不可靠：

- Slide 旧 scan 中 `force_scale=2.0` reward 反而更高。
- Stair action noise 早期也出现 σ=0.13 高于 σ=0.05/0.21 的现象。

这支持 noise-as-regularization / contact-rich dithering / plateau-then-cliff 等机制。后续应把 [[Mode A-B-C Taxonomy]] 和 [[FTC Metric]] 与曲线形态关联，而不是只寻找单调下降点。

### 4. Slide × external force 不崩不一定是失败

如果 Slide 在 240N 甚至更高 external force 下仍不崩，不能简单说实验失败。可能说明：

- Slide 任务对 horizontal pulse 不敏感。
- Stair 是 contact-rich locomotion，external pulse 更容易破坏 foot placement / support polygon。
- Slide 的主要脆弱性可能在 action noise / delay，而不是外力。
- 这可以转化为 “task-structural robustness asymmetry”：不同任务对不同扰动模态敏感，[[FTC Metric]] 应预测这种结构差异。

这比“Slide force 没用”更适合论文叙事。

### 5. Kitchen PBRS 是今天最明确的正结果

Kitchen 从 0.0 到 0.375 是实质突破。即使后续 plateau，也足以支持：

- PBRS 可以帮助 [[TC-PPO Algorithm]] 跨过稀疏奖励冷启动。
- Kitchen 可能成为论文中 “hard sparse reward” 的补充证据。
- 但不要急着 20M：先看 10M 后半程斜率。

### 6. 多模型协作可以服务 TC-PPO 工作流，但要分层

今天形成的实践原则：

- Claude / Opus 负责最终决策、代码仓库操作、论文一致性。
- DeepSeek 负责数学、统计、公式、低成本推理草稿。
- Kimi 负责中文长文档、导师汇报、中文润色。
- 本地工具负责敏感内容和自动化。
- 不要把所有模型“塞进 Claude 内核”，而是通过 MCP / shell / API 做后端工具路由。
