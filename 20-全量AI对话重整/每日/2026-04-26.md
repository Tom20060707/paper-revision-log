## 关键对话与决策

### 1. PBRS 0.375 的机制解释被重构

最初对 Server C Kitchen 任务的 `0.375` 进行了机制分析。关键结论是：

- `0.375` 是 shaped reward，不是真实 task completion。
- 它可以由 PBRS 的望远镜求和解释：
  - `0.375 = 0.2 × 1.875`
  - 其中 `0.2` 是 PBRS 系数 λ，`1.875` 是势能差相关的数值。
- 如果真实任务奖励 `task_reward ≡ 0`，则 shaped return 主要来自：
  - agent 改变了势能函数的某些距离项；
  - 但没有完成任何真实子任务。

当天后续进一步修正了这个解释：这不应简单写成“PBRS 失败的负面结果”，而应写成 **diagnostic finding**。因为 0.375 说明学习确实发生了，但学习到的是 proxy objective，而非真实物理操作。

### 2. Kitchen benchmark 被纠正为 HumanoidBench，而不是 FrankaKitchen

用户指出之前分析混用了 FrankaKitchen 背景，这是错误的。当天确认：

- 论文使用的是 **HumanoidBench `h1hand-kitchen-v0`**。
- 机器人是 H1 全身人形机器人，有手，不是 Franka 机械臂。
- Kitchen 任务包含 4 个子任务：
  1. 打开微波炉门；
  2. 移动水壶；
  3. 打开燃烧器/灶台开关；
  4. 打开灯开关。
- 任务终止步长为 500。
- 最大真实奖励为 4。
- Table V & VI 中 DreamerV3、TD-MPC2、SAC、PPO 在 Kitchen 上均为 0.0。

这个纠正确保后续论文和报告不再使用 Franka 机械臂、末端执行器短臂工作空间等错误物理背景。

### 3. 势能函数中的“距离”定义再次被纠正

用户强调：不能把“机器人手靠近物体”当作子任务完成。今天明确区分了三种距离/状态：

- **机器人手到物体的距离**：只说明接近，不说明完成操作。
- **物体状态到目标状态的距离**：Kitchen completion threshold 实际依赖这个。
- **真实接触操作是否发生**：需要门角度变化、旋钮旋转、开关状态改变、水壶位置变化等物理结果。

因此论文和报告中的势能函数符号从容易误解的 `d_next(s)` 改为更明确的：

- `d_obj^{(k)}(s)`：第 `k` 个当前子任务中，物体状态到目标状态的距离。

并明确：

- 子任务完成条件和势能项都应使用物体状态距离，而不是手部距离。
- 非接触阶段物体状态通常不变，因此 object-only potential 对机器人动作的有效训练信号可能消失。
- 这不是简单的“梯度为零”问题，更准确地说是 **zero-signal barrier** 或 **contact dynamics barrier**。

### 4. 形成“两阶段”解释：接近阶段 + 接触操作阶段

今天形成了一个对 [[TC-PPO Algorithm]] 有帮助的新叙事：

- **阶段 1：Approach / pre-contact**
  - 机器人需要靠近目标物体。
  - 这时可以用 PBRS 的 approach potential 提供稠密引导。
  - 例如基于手/末端执行器到目标物体的距离。
- **阶段 2：Manipulation / contact-rich interaction**
  - 机器人必须通过接触力、力矩、摩擦和状态改变完成真实操作。
  - 例如打开微波炉门需要绕铰链施加力矩；开关需要越过状态阈值；移动水壶需要实际改变物体位置。
  - 这部分与 [[FTC Metric]] 和 [[Mode A-B-C Taxonomy]] 的时序一致性、接触阶段建模更相关。

这个两阶段视角把 PBRS 的 0.375 重新解释为：agent 学到了部分接近/状态扰动信号，但没有进入有效接触操作阶段。它反而支持 TC-PPO 需要显式处理接触时序和任务阶段切换。

### 5. 对 `ENFORCE_TASK_ORDER=True` 的批判被纳入论文

用户提出：Kitchen 四个子任务之间没有天然逻辑依赖，强制顺序可能是人为制造难度。当天用 Kimi/DeepSeek 分析后，形成较温和、可写入论文的表述：

- 四个操作在现实中多数是并行独立的 manipulation primitives。
- 强制顺序把独立任务压缩成单一路径，造成 reward sparsity amplification。
- 若每个子任务完成概率约为 `p ≪ 1`：
  - 强制顺序下完整链条探索概率呈 `O(p^4)`；
  - 独立并行奖励可提供更强的梯度信号。
- 论文中应避免直接攻击 benchmark 作者，而以 “benchmark design implications” 和 “evaluation protocol bias” 的方式表述。

## 新洞察

### 1. 0.375 不是“失败数据”，而是诊断信号

今天最重要的认知更新是：`0.375` 不应简单解释为负面失败。它说明：

- agent 确实优化了某个 proxy；
- 但 proxy 与真实操作完成之间存在因果断层；
- 这恰好揭示了 naive PBRS 与 contact-rich manipulation 的不匹配。

因此它可以作为论文中的一个有价值发现：PBRS 是一个 diagnostic probe，揭示了 HumanoidBench Kitchen 中“接近/状态扰动”与“真实任务完成”的分离。

### 2. Object-only potential 与 approach potential 必须严格区分

今天 Kimi 评估指出论文中的一个潜在硬伤：

- PBRS 失败分析使用的是 object-only potential；
- Two-Phase Framework 中使用的是 approach potential；
- 如果不明确区分，审稿人会问：到底是 PBRS 失败，还是你们势能函数设计错了？

因此论文中必须写清楚：

- naive object-only potential：用于诊断，暴露零信号/接触壁垒。
- approach potential：用于接近阶段，可提供有效连续梯度。
- manipulation/contact 阶段：需要 [[FTC Metric]] 或 TC-PPO 的时序/接触机制。

### 3. Kitchen 的问题不只是“难”，而是任务定义会引导研究方向

今天形成了更深层 benchmark 观点：

- Kitchen 当然难，但“难”和“定义合理”不是一回事。
- 四个子任务之间没有强物理依赖，却被强制排序，会人为放大奖励稀疏性。
- 这种评估协议可能偏向分层 RL 或强先验方法，而不利于端到端方法。
- 写论文时不能用攻击性语言，但可以作为 benchmark design implication 提出。

### 4. 接触操作不能只用距离阈值解释

距离阈值在一些任务中合理，例如导航、放置、简单抓取。但对于：

- 旋转开关；
- 打开门；
- 拨动灯开关；
- 移动带接触约束的物体；

距离接近不充分，也不必要。真正需要的是：

- 角度变化；
- 位置变化；
- 开关状态变化；
- 力/力矩和接触事件；
- 任务阶段切换。

这与 [[Sim-to-Real Robustness]] 和 [[FTC Metric]] 的研究目标直接相关：真实机器人学习不能只看空间接近，还要看接触时序是否稳定、物理效果是否真实发生。

### 5. 国产模型路由的真实稳定用法

今天对 Kimi 的认识被更新：

- Kimi 不是不行，而是 K2.6 作为推理模型需要足够大 `max_tokens`。
- 若 `max_tokens` 小，推理过程消耗完 token，`content` 就为空。
- 后续应使用：
  - Kimi：`max_tokens >= 8000`
  - 复杂论文任务：`max_tokens` 可到 12000
- 对 LaTeX 代码生成，DeepSeek 仍可能更稳定；但 Kimi 可以先负责分析与表达质量评估。
