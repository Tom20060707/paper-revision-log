# 2026-04-18 研究日志

## 核心进展

今天主要推进了 [[TC-PPO Algorithm]] 论文中实验表格的真实性核查、版面压缩和叙事聚焦，尤其围绕 [[FTC Metric]]、percentile threshold 选择、以及 HumanoidBench 表格是否服务论文主线做了多轮修正。

最重要的进展有三点：

1. **确认并修正 FTC percentile 表格数据来源**
   - 发现早先表格中 P55–P90 的部分数值并非全部来自完整 JSON。
   - 重新核查后确认：P55–P80 可由已有 JSON / 原始记录支持；P85/P90 后续从同一批 warmup 实验保存的 `ftc_percentiles_history.npy` 时间序列重新计算。
   - 最终 Table I 中 P55/P65/P70/P75/P80/P85/P90 均改为基于 Server C truck warmup 原始数据计算，不再使用推算值。

2. **明确 P75 选择的论文逻辑**
   - P80 的 Δτ 最接近设计目标 0.07，但其对应 FTC percentile 超出 truck 稳定期可达天花板。
   - P75 的 Δτ 略小，但被论证为更现实、更可达的折中点。
   - 这强化了论文的核心论点：不是简单追求更大 temporal contrast，而是在 [[FTC Metric]] 的可达稳定区间内选择合适门控阈值。

3. **大幅压缩和重构 HumanoidBench 相关表格**
   - 删除原本意义较弱的 Mode C 小表，将 walk / hurdle 的零干扰结论并入正文。
   - 将大表从全宽 `table*` 改为单栏 `table`，删除 SAC、TD-MPC2、DreamerV3 等 off-policy baseline 列。
   - 将 `PPO (ours)` 统一改为 `Vanilla PPO`，避免与 TC-PPO 混淆。
   - 修正 “512×4 eliminates collapse” 这类歧义表头，改为明确表达 HB-PPO collapse、Vanilla PPO recover 的因果关系。

## 对话精炼

### 1. FTC percentile 表格真实性核查

一开始发现 Table I 的 percentile 数据存在可信度问题：并不是所有 seed 都有完整 P55–P90 的 JSON 记录。原始记录显示：

- 8 个 seed 都有 P5、P45。
- 只有 5 个 seed 有完整 P55–P80 数据。
- P85、P90 最初并不是从 JSON 直接读取，而是基于已有数据做过估算。

随后决定不重新跑实验，而是读取 warmup 过程中保存的原始 `ftc_percentiles_history.npy` 时间序列。该文件来自同一批 truck warmup 实验，每 10 个 rollout 保存一次 FTC 值，因此可以从原始时间序列直接计算 P85/P90/P95 等 percentile。这样避免了重新跑实验，也避免使用推算值。

最终确认 Table I 的数值全部来自真实原始数据，来源为 Server C 上 2026-04-11 truck warmup 的 5 个 seed。P85/P90 虽然不是原始 JSON 中预先保存的字段，但它们与 P75 等 percentile 来自同一个 FTC 时间序列，计算方式等价，因此可以视为真实数据而非虚构或外推。

### 2. P75 vs P80 的关键判断

重新计算后发现：

- P75 的 Δτ = **0.060±0.014**，不是早先错误表格中的 0.045。
- P80 的 Δτ = **0.072±0.010**，最接近设计目标 0.07。
- 但 P80 对应 Pb 均值为 0.421，超出 truck 稳定期 FTC 天花板约 0.39。
- P65 均值 0.396 已经略超出 0.39，P70/P75/P80 更高。

这里出现一个重要张力：**P80 在 temporal contrast 上更理想，但在稳定期可达性上不可行；P75 虽然 Δτ 略小，但更适合作为论文中的实际折中选择。**

这一点不只是表格修正，而是强化了论文对于 [[FTC Metric]] 的解释：FTC threshold 不是单纯越高越好，必须同时满足可达性、稳定性和对 Mode A/B 的区分能力。

### 3. Table III 版面与叙事重构

用户指出原 Table III 占据半页 A4，且在双栏论文中变成全宽表格，caption 和 footnote 都过长，严重影响排版和美观。

第一轮处理是压缩 caption 和 footnote，但仍保留 `table*`。随后进一步讨论后认为：表中 SAC、TD-MPC2、DreamerV3 等 off-policy 方法并不服务论文核心，甚至可能造成反效果。因为这些方法在若干任务上得分更高，而论文重点并不是证明 TC-PPO 击败所有 off-policy baseline，而是证明 [[FTC Metric]] 与门控机制能解释并缓解特定任务中的训练崩塌。

因此最终决策是：

- 删除 SAC、TD-MPC2、DreamerV3 列。
- 表格只保留 HB-PPO / Vanilla PPO / TC-PPO。
- `table*` 改为单栏 `table`。
- 正文中涉及 DreamerV3 / TD-MPC2 具体数字的段落删除或改写，仅保留指向 HumanoidBench 原文的简短说明。
- 表格更聚焦于 on-policy PPO 系列中的 collapse / recover / gated stabilization 关系。

这个改动让表格从“泛泛 benchmark 对比”回到论文主线：[[TC-PPO Algorithm]] 相对 HB-PPO 与 Vanilla PPO 的稳定性贡献。

### 4. 删除原 Table II，将 Mode C 结果并入正文

晚上继续检查表格时，用户指出一个小表只包含 walk 和 hurdle 两个任务，信息量很低，不值得单独占版面。

该表原本想证明的是：在 [[Mode A-B-C Taxonomy]] 中的 Mode C 任务上，TC-PPO 与 Vanilla PPO 得分基本一致，即门控未触发时没有额外负担或副作用。但这个结论可以用一句正文表达，不需要单独建表。

最终删除 Table II，并将其核心结论并入正文：

- walk：$36.1\pm0.4$ vs. $36.3\pm0.5$，Mode C。
- hurdle：$14.4\pm1.1$ vs. $18.1\pm3.9$，Mode C。
- 结论：gate 不激活时没有明显 overhead。

删除后论文页数从 31 页降到 30 页，版面压力得到缓解。

### 5. 术语和表头严谨性修正

用户进一步指出两个表述问题：

1. `PPO (ours)` 容易误导读者，以为是 TC-PPO 或本文算法的一部分。
2. `[512×4] eliminates collapse` 的表头与下方 HB-PPO collapsed 结果形成视觉矛盾，容易被理解为 512×4 仍 collapse。

最终修正为：

- `PPO (ours)` → `Vanilla PPO`
- 分组标题改为更明确的：
  - `HB-PPO [64×2] collapses; Vanilla PPO [512×4] recovers`

这一修改提升了论文表格的可读性和严谨性，避免审稿人误解 baseline 设置。

## 新洞察

1. **数据真实性比表格完整性更重要**
   - 早先为了让表格看起来完整，P85/P90 曾被处理成估算值，这是论文风险。
   - 今天的修正说明：只要原始时间序列存在，就应直接从原始数据重新计算，而不是依赖表格补全或推算。
   - 对 [[FTC Metric]] 这类核心指标，任何 percentile 数值都必须能追溯到原始实验文件。

2. **P75 的合理性来自“可达折中”，不是数值最优**
   - P80 的 Δτ 更接近 0.07，但它可能超出稳定可达区间。
   - P75 的价值在于兼顾 temporal contrast 和训练稳定性。
   - 这使得 threshold selection 从经验选择变成有约束的设计决策。

3. **HumanoidBench 表格不应变成泛化 benchmark 排名**
   - TC-PPO 的论文主线不是“打败 SAC/Dreamer/TD-MPC2”。
   - 如果 off-policy baseline 得分更高，放在主表里反而削弱论点。
   - 更好的叙事是：在 on-policy PPO 家族内部，TC-PPO 如何通过 FTC gate 缓解 collapse、保持 Mode C 零干扰。

4. **小表格如果只能证明一句话，就应该进正文**
   - walk / hurdle 的 Mode C 结果单独成表不划算。
   - 删除小表不仅省版面，还让论文结构更紧凑。
   - 这对双栏会议论文尤其重要。

5. **baseline 命名必须避免“算法归属”误导**
   - `PPO (ours)` 容易让读者误会这是本文算法的一部分。
   - `Vanilla PPO` 更清楚地区分普通 PPO baseline 与 [[TC-PPO Algorithm]]。
   - 表头应直接表达比较逻辑，避免让读者从布局中反推含义。
