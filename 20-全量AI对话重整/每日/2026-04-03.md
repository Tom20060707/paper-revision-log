# 2026-04-03 研究日记：TC-PPO 数据归属、脚本清理与消融缺口确认

## 今日主线

今天的核心不是继续“写论文”，而是对 [[TC-PPO Algorithm]] 论文叙事的实验基础做了一次关键审计：**之前以为已经有 HB-PPO → Vanilla-PPO → TC-PPO 三层对比，但实际数据归属比想象中复杂，尤其是 Vanilla-PPO [512×4, no FTC] 这一层存在严重缺口。**

主要推进了四件事：

1. **重新核查 Server A / Server B / Server C 的实验身份**
   - 重点不再看文件名或模型名，而是看实际脚本、命令行参数、`algo` 分支、网络结构、[[FTC Metric]] 是否启用。
   - 明确提出：不能因为目录或模型名叫 `tc_ppo` / `vanilla_ppo` 就直接判断算法，必须看实际代码逻辑。

2. **确认当前论文三层叙事存在消融缺口**
   - 原计划三层：
     1. HB-PPO：官方 HumanoidBench PPO，小网络 `[64,64]`，无 FTC；
     2. Vanilla-PPO：大网络 `[512×4]`，无 FTC；
     3. TC-PPO：大网络 `[512×4]` + [[FTC Metric]] gating。
   - 但目前已确认：真正缺的是 **Vanilla-PPO [512×4, no FTC] 的系统性 10M 多种子数据**。
   - 这会直接影响 [[Mode A-B-C Taxonomy]] 或 collapse narrative 的可信度。

3. **整理并清理 PPO 脚本**
   - 在本机建立 `PPO脚本合集/`，只保留 3 个核心脚本：
     - `1_HB_PPO.py`
     - `2_Vanilla_PPO.py`
     - `3_TC_PPO.py`
   - 删除或准备删除与当前实验无关的旧草稿 / 独立实现，避免之后继续混淆。

4. **开始收集三台服务器的全量实验信息**
   - 用户准备分别发送 Server A、Server B、Server C 的所有 `summary.json` / CSV / 日志输出。
   - 下一步需要基于“实际参数”而不是“脚本名称”判断：
     - 用了什么网络；
     - 是否启用 FTC；
     - `ent_coef`、`target_kl`、batch size、learning rate 等是否一致；
     - 每个任务实际跑了多少步、几个 seed、效果如何。

### 2. 官方 HB-PPO baseline 需要重新核查

对 HumanoidBench 官方 PPO 的理解进一步明确：

- 官方 `run_sb3_ppo.py` 使用 SB3 `PPO("MlpPolicy", env, batch_size=512)`；
- 没有显式设置 `net_arch`；
- 因此网络结构是 SB3 默认 MLP：`[64,64]`；
- 学习率确认是 `3e-5`；
- batch size 是 `512`。

这支持当前论文中的关键 confound 发现：**官方 HumanoidBench PPO baseline 很可能被小网络容量限制严重影响。**

但也发现：之前把 Server A 的某些结果误填入 HB-PPO 列，是严重错误，已经要求修正。

### 3. Vanilla-PPO 数据缺口被明确为 P0 问题

此前曾误以为：

- Server A = Vanilla-PPO [512×4, no FTC]；
- Server B = TC-PPO [512×4 + FTC]。

但经过多轮检查，结论变得更谨慎：

- Server A / Server B 中很多成功结果很可能并不是无 FTC 的 Vanilla-PPO；
- 有些 `vanilla_ppo` 尝试失败原因是：
  - Server A：并行过多导致 exit 143 / OOM / SIGTERM，属于实验操作失败，不是算法自然 collapse；
  - Server B：调用 `run_ppo_paper.py`，但脚本不存在，属于路径/脚本错误。
- 因此不能把这些失败当成 Vanilla-PPO 的性能证据。

决策：**论文中不能声称 Vanilla-PPO [512×4, no FTC] 已经在 collapse 任务上实测崩溃，除非后续补跑。**

### 4. Server C 的 100M 实验暴露了长训练风险

从 Server C 输出中看到：

| 服务器 | 任务 | algo | steps | 结果 |
|---|---|---|---|---|
| Server C A800 | walk | `tc_ppo` | 100M | final 约 12.2，best 约 29.8 |
| Server C A800 | walk | `vanilla_ppo` | 100M | final 约 66.4，best 约 70.6 |
| Server C A800 | truck | `tc_ppo` | 接近 100M | final 约 774.5，best 约 827.2 |
| Server C A800 | truck | `vanilla_ppo` | 接近 100M | final 约 1065.2，best 约 1083.8 |

这些数字来自当日对 `summary.json` / 日志的初步读取，仍需等用户发全量 Server C 数据后统一确认。

初步洞察：

- 在 100M 长训练下，TC-PPO 可能因为 FTC 阈值过于激进而长期处于 stabilizing mode；
- 对 walk / truck 这类任务，[[FTC Metric]] 低于阈值不一定等价于“即将崩溃”；
- 如果 TC-PPO 长期把 `ent_coef` 锁高、`target_kl` 收紧，可能反而抑制策略更新；
- 这不是立即否定 TC-PPO，而是提示：**FTC gating 的阈值和适用任务族需要重新定义。**

这对论文影响很大：TC-PPO 不能写成“普遍优于 PPO”，而应写成更审慎的 **collapse-aware / contact-instability-aware intervention**。

## 新洞察

### 1. “脚本名”不是算法身份，必须看实际参数路径

今天最大的教训是：不能根据 `tc_ppo_h1_xxx`、`vanilla_ppo_xxx` 这类名称判断算法。真正要看：

- 是否传入 `--algo vanilla_ppo` 或 `--algo tc_ppo`；
- `algo` 如何影响 callback；
- 是否进入 FTC 分支；
- 网络 `net_arch` 是 `[64,64]`、`[512×4]` 还是其他；
- `ent_coef`、`target_kl` 是否随 FTC 改变；
- 日志中是否有 `tc_ppo/ftc`、`stabilizing`、`collapse_depth` 等字段。

这应成为以后所有实验审计的标准流程。

### 2. 当前论文不能把“网络容量”和“FTC”混在一起讲

如果只有：

```text
HB-PPO [64×2, no FTC]  vs  TC-PPO [512×4 + FTC]
```

那么提升可能来自：

- 网络变大；
- `ent_coef` 从 0 变成 0.01；
- `target_kl` 从 None 变成 0.2；
- FTC gating；
- 训练脚本其他细节；
- 随机种子或运行环境差异。

因此论文必须补齐至少一个关键 baseline：

```text
Vanilla-PPO [512×4, no FTC]
```

否则不能声称“FTC 防止了 collapse”，最多只能说“当前 TC-PPO configuration 相对官方 HB-PPO 有提升”。

### 3. FTC 不一定是普适正向指标

100M Server C 结果提示：FTC gating 在一些任务上可能是负向约束。

可能机制：

- walk / truck 的 FTC 值长期低于阈值；
- TC-PPO 长期进入 stabilizing；
- `ent_coef` 提高、`target_kl` 收紧；
- 策略更新被过度限制；
- 长训练后出现性能退化。

这说明 [[FTC Metric]] 可能更适合“检测 contact-pattern collapse”，但不一定适合作为所有任务的统一控制信号。论文应该把贡献限定为 **collapse-sensitive adaptive trust region**，而不是泛化成全任务 universal improvement。

### 4. Vanilla-PPO 补跑优先级高于继续扩展 TC-PPO 任务

现在继续跑更多 TC-PPO 任务的边际价值低于补 Vanilla-PPO。因为没有 Vanilla-PPO，就无法回答审稿人最核心的问题：

> 是大网络起作用，还是 FTC 起作用？

因此实验优先级应该从“多跑任务”转为“补关键消融”。

最关键补跑：

- insert_normal；

建议至少 3 seeds，每个 10M steps。

目的：

### P1：补充“只调超参数、不加 FTC”的控制组

考虑到官方 HB-PPO 与当前脚本还存在：

- `ent_coef`: 官方默认可能为 0.0，当前脚本为 0.01；
- `target_kl`: 官方默认可能为 None，当前脚本为 0.2。

需要至少设计一个控制组：

```text
Large PPO [512×4, ent_coef=0.01, target_kl=0.2, no FTC]
```

这才是 TC-PPO 最公平的 baseline。

更严格版本可选：

```text
Large PPO [512×4, ent_coef=0.0, target_kl=None, no FTC]
```

但优先级低于前者。

### P1：论文表格重构

当前论文表格应避免把没有实测的 Vanilla-PPO 数字写成结果。建议表格分层：

1. 官方 HB-PPO baseline 表；
2. 当前 TC-PPO 实测表；
3. Vanilla-PPO 补跑结果表；
4. 如果缺失，用 `---` 并脚注说明“pending”。

### P2：重新定义 TC-PPO 的适用范围

根据 Server C 100M 结果，需要在 Discussion / Limitations 中加入：

- FTC 阈值 `τ_low=0.8` 可能对 locomotion 长训练过于激进；
- 长训练中长期 stabilizing 可能限制策略改进；
- 未来需要 task-adaptive threshold 或 learned gating；
- [[FTC Metric]] 应与 reward trend、KL trend、contact phase 分布联合使用。
