### 2. BookshelfHard 结果：不是 Mode A，而是 Mode C

Server C `bookshelf_hard` 10M steps 结果：

| 算法 | seed0 | seed1 | seed2 | 均值 |
|---|---:|---:|---:|---:|
| TC-PPO final_reward | 154.58 | 175.66 | 182.80 | 171.01 |
| TC-PPO best_reward | 189.77 | 197.19 | 194.00 | 193.65 |
| Vanilla final_reward | 待完整原始表确认 | 待完整原始表确认 | 待完整原始表确认 | 167.6±27.9 |

关键判断：

- `bookshelf_hard` FTC 约 0.54，高于 `τ_high=0.42`；
- gate 从未触发；
- 因此该任务是 **Mode C**，不是 Mode A；
- TC-PPO 与 Vanilla 结果接近，说明在 gate 不触发时 TC-PPO 近似退化为 Vanilla，这对机制 specificity 是好证据。

同时发现 Vanilla 的 FTC=1.000 可能来自空接触序列边界条件，不能解释为“稳定步态”。

### 3. Mode A 是否触发：重新定义后确认“未完整观测”

用户随后给出 Truck 的 gate 触发轨迹：

- 约 5,242,880 steps 时 FTC 从 0.389 → 0.351 → 0.344；
- 低于 `τ_low=0.35` 后进入 stabilizing；
- 此后 FTC 在 0.344–0.352 间震荡；
- 始终无法恢复到 `τ_high=0.42`；
- 后半段长期 `stabilizing=1`，`ent_coef=0.02`，`target_kl=0.05`，策略更新被压制。

最终决策：

- Truck 是 **gate triggered but never closes**，应归为 **Mode B**；
- **Mode A 必须定义为 “open → close” 的完整 hysteresis cycle**；
- 当前没有任何任务已确认出现完整 Mode A；
- 论文中不能声称 Mode A 已被实证验证。

这直接导致 Section 3.2、Section 6.3、Limitations、概念图 caption 的叙事修改。

### 4. 阈值根因分析：全局 `τ_high=0.42` 对 Truck 不适配

今天形成一个重要洞察：

> Truck 的自然稳定 FTC 约 0.36–0.37，本身低于全局 `τ_high=0.42`。因此一旦 gate 打开，即使策略恢复到该任务自身的正常接触节律，也无法满足退出条件。

这说明全局阈值存在结构性限制：

- 对 FTC 高的任务：gate 永远不触发 → Mode C；
- 对 FTC 低的任务：gate 一旦触发就关不上 → Mode B；
- 理想 Mode A 需要 **per-task calibration**，而不是跨任务固定 `τ_high`。

该结论被写入论文讨论与限制部分。

## 新洞察

### 1. Mode A 的实证标准必须严格：触发不等于 Mode A

今天澄清了一个关键概念：

- `stabilizing=True` 只能说明 gate 打开；
- **Mode A 必须是 gate 打开后又关闭**；
- Truck 虽然触发，但未关闭，因此是 Mode B；
- 论文不能用“触发了”来证明 Mode A。

这对 [[Mode A-B-C Taxonomy]] 很关键。后续所有实验汇报应记录完整时序：

```text
FTC trajectory
stabilizing_active: False/True transitions
activation step
deactivation step
frac_on
final reward
```

没有 `True → False` 证据，就不能声称 Mode A 被观测。

### 2. 全局阈值不适合所有任务，per-task calibration 是必要方向

`τ_low=0.35, τ_high=0.42` 的全局阈值可以解释很多任务，但今天确认它存在结构性限制：

- Truck 的稳定 FTC 约 0.36–0.37；
- 这低于全局 `τ_high=0.42`；
- 因此 Truck 一旦进入 stabilizing，理论上就很难退出；
- 这不是实现 bug，而是阈值定义与任务物理接触节律不匹配。

后续可以把论文未来工作明确为：

> Learn or estimate per-task FTC baselines, then set task-conditioned hysteresis thresholds.

### 3. Config A 数据不能简单当作标准 TC-PPO

若一个 run 从第一个 checkpoint 或极早期就 `stabilizing=1`，则它本质上是：

- 高 entropy；
- 低 target_kl；
- 固定保守 PPO；

而不是动态 FTC-conditioned control。

因此 Config A 可以作为 failure/diagnostic evidence，但不能直接作为 TC-PPO 主结果吹嘘。论文中必须标注，避免审稿人认为算法机制被夸大。

### 4. 图例错误比数字错误更危险

今天发现图中 “TC-PPO (ours)” 实际对应 Vanilla-PPO [512×4]。这类错误会严重损害论文可信度，因为它会让审稿人误以为：

- TC-PPO 在图中表现很好；
- 但实际是 Vanilla scale-up 的结果。

修正后，论文叙事更清晰：

- Vanilla-PPO [512×4]：证明 scale-up 解决部分 collapse；
- TC-PPO：用于 FTC gating 机制研究；
- HB-PPO [64×2]：官方 benchmark baseline，不是自己跑的。
