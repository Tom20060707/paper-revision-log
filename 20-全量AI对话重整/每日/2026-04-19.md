## 新洞察

### 1. “贡献多”不等于“论文强”

今天最重要的写作洞察是：

> 贡献不能平均打光。主贡献需要被放在读者第一眼能看到的位置，其他内容应变成支撑证据、控制实验、附录或未来工作。

TC-PPO 现在真正的主线应是：

> 接触时序不稳定会导致 PPO collapse → FTC 能 reward-agnostic 地检测这种不稳定 → TC-PPO 用 FTC 做选择性门控 → P5/P75 + soft gating 让这个机制在多任务中可用。

Layer-1/Layer-2、GCTC、Type-I/II Mode C、网络容量分析都很有价值，但它们不应该和主贡献平铺。

### 2. 论文不应过度声称性能提升

当前实验最强的 reward gain 是 slide 的 +9.6%。其他 Mode A 任务中，sit-hard、stair 基本持平，powerlift 均值还更低。因此如果把论文包装成“TC-PPO 显著提升 reward”，会被审稿人抓住。

更稳妥定位：

- TC-PPO 是 contact-rich RL 的诊断与选择性稳定化框架；
- reward gain 是次级证据；
- variance reduction、collapse early warning、Mode taxonomy、failure decomposition 是更核心的价值。

尤其 powerlift 的 σ 从 8.3 降到 5.5，是稳定性角度的重要证据，不应只看均值 87.4 vs 99.7。

### 3. 鲁棒性实验可能比改算法更适合当前阶段

用户提出“Boss 升级，角色装备也应升级”的直觉，即任务加噪声后算法也应升级。讨论后的结论更谨慎：

- 加噪声非常合理；
- 但短期不一定需要改 TC-PPO 底层算法；
- 因为真实扰动本身就会表现为 contact timing irregularity，FTC 理论上天然会响应。

更干净的叙事是：

> clean benchmark 中很多任务是 Mode C，所以 TC-PPO 看起来不 universal；但在扰动/Sim-to-Real 条件下，contact irregularity 增强，更多任务转为 Mode A，TC-PPO 的选择性门控优势被放大。

这比“为了困难任务临时改算法”更容易说服审稿人，因为它验证的是原算法机制的适用边界，而不是引入未经充分验证的新组件。

### 4. 加噪声不需要下载新环境

一个具体工程结论是：MuJoCo/HumanoidBench 中大多数扰动可以直接通过 wrapper 或底层 model/data 修改完成，不需要额外下载数据或新环境。

可行接口包括：

- `data.xfrc_applied[torso_id]`：外力脉冲；
- `model.geom_friction[floor_id]`：地面摩擦随机化；
- action 上加 Gaussian noise：执行器/力矩噪声；
- observation 上加 Gaussian noise：传感器噪声；
- `model.body_mass[id]`：质量随机化；
- action buffer：动作延迟。

这为后续 [[Sim-to-Real Robustness]] 实验提供了低成本路径。

### 风险

1. **扰动过强导致所有方法都崩**
   - 如果噪声强度太高，TC-PPO 也无法恢复，会变成负面证据。
   - 需要设计 mild / medium / hard 三档扰动，先找可恢复区间。

2. **扰动环境可能被审稿人质疑为人为构造**
   - 需要把扰动解释为 sim-to-real domain randomization，而不是“为了让 Vanilla 崩溃而改任务”。
   - 最好使用标准、可复现、物理合理的扰动。

3. **改底层算法会拖慢投稿**
   - Lyapunov FTC margin、per-sample FTC weighting、GAE re-weighting 等想法有潜力，但会引入新的 bias/ablation/重跑压力。
   - 当前投稿阶段除非有明确时间和算力，否则不建议加入主算法。

4. **论文定位仍需克制**
   - 不能声称 TC-PPO universal；
   - 不能声称 FTC 是 collapse 的充分必要条件；
   - 应表述为 leading indicator / diagnostic signal / selective stabilizer。

5. **每日研究日志质量风险**
   - 之前日志过于模板化，缺少真实推进内容。
   - 之后需要记录：具体决策、文件名、实验名、数字、失败原因、下一步，而不是笼统写“讨论了论文修改”。
