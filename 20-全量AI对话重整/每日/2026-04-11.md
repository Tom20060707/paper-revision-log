## 关键对话与决策

### 1. 自适应阈值不是 Future Work，而是当前方法配置

早上首先明确了一个关键写作定位：  
[[Adaptive Threshold Calibration]] 不应写成 future work，而应作为当前实验配置的一部分写入论文。

最终采用的设定是：

- 先用 Vanilla-PPO 做 warm-up；
- 收集 FTC 分布 $\mathcal{F}$；
- 用 percentile 设定阈值：
  - $\tau_{\text{low}} = P_5(\mathcal{F})$
  - $\tau_{\text{high}} = P_{45}(\mathcal{F})$

这个写法解决了“τ_low=0.35、τ_high=0.42 是否是手调参数”的质疑：它们被重新解释为从 [[FTC Metric]] 分布中校准出的语义稳定 percentile，而非任意常数。

同时决定：跑完 warm-up 后，可以补充 per-task calibration table，用真实 P5/P45 数值支撑论文自洽性。

### 7. truck percentile sweep 的关键实验判断

关于 percentile sweep，今天形成了一个重要认识：

原先提出的 P20/P60、P35/P70 方案不够合理，因为：

- $\tau_{\text{low}}$ 升高会让 gate 更容易开；
- 但 $\tau_{\text{high}}$ 也升高会让 gate 更难关；
- 这可能把实验推向新的 [[Mode A-B-C Taxonomy|Mode B]]，而不是 Mode A。

更合理的 sweep 是：

1. P5 / P45：论文主规则；
2. P10 / P40：更容易开，也更容易关；
3. P15 / P35：进一步收窄滞回区间，用于寻找 Mode A。

这个 sweep 的定位也明确了：  
它不是主结果，而是用于诊断“如何让 truck 从 Mode B 进入 Mode A”的小范围搜索。

### 8. 发现 soft gating 与 Mode A 检测之间的定义冲突

晚上检查 Server C 当前训练状态时发现：

- 6 个 warm-up 已完成；
- TC-PPO 阶段已经进入；
- 训练大约跑到 4.9M steps；
- GPU 使用约 28GB，利用率一度接近 95%；
- 尚未出现 Mode A。

同时观察到当前 P5/P45 在 truck 上大约为：

- $\tau_{\text{low}} \approx 0.3501$
- $\tau_{\text{high}} \approx 0.3758$
- 区间宽度约 `0.0257`

FTC 在约 `0.33–0.39` 区间震荡，说明 truck 的 warm-up FTC 分布本身偏低且窄。

更关键的是发现一个定义层面的冲突：  
当前 6 个实验都开启了 `--soft_gating`，而代码中的 `MODE A ACHIEVED` 检测只在 hard gating 的 `stabilizing_active True → False` 逻辑中触发。也就是说：

- hard gating 有离散开/关状态，因此可以严格定义 Mode A；
- soft gating 是连续 $g_t$，没有二值 gate close 事件；
- 因此 soft gating 实验不能直接产生传统意义上的 Mode A 证据。

这带来一个新决策：  
论文中应区分：

- **Hard-gating Mode A**：离散 gate ON → recovery → gate OFF；
- **Soft-gating analogue**：$g_t$ 上升后随 FTC 恢复而下降，可作为连续版恢复轨迹，但不能简单称为同一个 Mode A，除非重新定义判据。

## 新洞察

### 1. Percentile calibration 的风险不在“能不能算”，而在任务分布形状

今天 truck 的观察显示：P5/P45 在理论上语义优雅，但在 FTC 分布本身很窄、整体偏低的任务上，P5 与 P45 可能非常接近。  
例如当前 truck 观测到：

- P5 约 0.3501；
- P45 约 0.3758；
- 宽度仅约 0.0257。

这意味着 hysteresis band 太窄，可能导致门控状态不稳定或难以形成清晰的恢复证据。  
因此论文中需要强调 percentile 方法的优点，同时也要在 limitations 中承认：percentile pair 的选择仍需验证其对不同 FTC 分布形状的鲁棒性。

### 2. Mode A 是 hard gating 的离散闭环，不应直接套到 soft gating

[[Mode A-B-C Taxonomy]] 最初定义依赖 `stabilizing_active` 的离散开关：

- FTC 跌破 $\tau_{\text{low}}$ → gate ON；
- FTC 恢复超过 $\tau_{\text{high}}$ 且满足 $W=2$ → gate OFF；
- 完整 ON→OFF 即 Mode A。

而 [[Soft Gating]] 使用连续 $g_t$，天然没有二值 gate close 事件。  
因此不能用同一个日志条件检测 soft gating 的 Mode A。更严谨的写法是：

- hard gating：报告 Mode A / B / C；
- soft gating：报告 $g_t$ 随 FTC 恢复而下降的 continuous recovery analogue。

这个区分对论文非常重要，否则会造成概念自相矛盾。

### 3. 主结果必须只承载一种语义

今天对 Table I、Table II、Fig. 4、附录结果的整理说明：  
一张表不能同时承担“算法比较”“诊断失败”“超参不匹配补充”的多重语义。否则审稿人会认为作者在混用不公平结果。

后续论文图表应遵循：

- 主文主表：只放公平可比结果；
- 诊断图：只说明机制失败或模式；
- Appendix：放 Config A、服务器差异、pilot runs 等补充信息。

### 4. 网络扩容应作为控制变量，而不是贡献抢主线

如果把 `[512×4]` 写得太像贡献，会削弱 [[TC-PPO Algorithm]] 的主线。  
更好的叙事是：

- 大网络先建立 stronger PPO baseline；
- TC-PPO 在 stronger baseline 上测试是否能进一步稳定接触时序；
- 因此网络扩容是控制实验和前置条件，不是最终算法贡献。

### 5. 个人知识库应服务“写作记忆”，而不是单纯收藏资料

今天搭建 Obsidian/qmd 后进一步明确：  
知识库最有价值的用法不是保存网页，而是持续沉淀：

- 某个实验为什么这么设计；
- 某个参数为什么改；
- 某篇论文对 Related Work 的一句话定位；
- 某个 reviewer 风险如何回应；
- 某个术语如 [[FTC Metric]]、[[Mode A-B-C Taxonomy]] 的最终定义。

这类内容如果不写进 wiki，会在多轮论文修改中快速丢失。
