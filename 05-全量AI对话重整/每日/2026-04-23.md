   对 `main.tex` 做了 overclaim 修正：Abstract、Contribution 4、Section IV-D、Conclusion 等位置重新对齐“已验证结果”和“设计/形式化贡献”的边界。随后又统一修改 Fig. 1–7 的视觉表达，强调“单策略 + 训练侧调度 + Mode A/B/C 诊断”的论文主线。

## 关键对话与决策

### 1. 不把当前论文改成 HRL，而是守住“结构化训练”边界

今天讨论了 HRL、RPL、BUDS、SPiRL、CRISP、TAMP 等路线。共同结论是：

- HRL / BUDS / SPiRL / CRISP 属于 **结构化控制**：
  - 显式分解任务；
  - 有高层子目标或 skill selection；
  - 可能有多个策略或 skill library。
- 当前 [[TC-PPO Algorithm]] + PBRS 属于 **结构化训练**：
  - 仍然使用单一策略；
  - FTC 不生成子目标，而是调度训练阶段；
  - PBRS 不替代控制结构，只提供进度相关密集梯度。

最终决策：

> 当前论文不加入“最小分层系统”或 HRL 层。  
> 当前论文只声称 TC-PPO + PBRS 是 phase-aware training scheduler，而不是结构化控制器。

这个决策避免了两个风险：

- 论文贡献点被稀释成 “FTC + HRL”，需要重新和 BUDS / SPiRL / CRISP 对标；
- 工程周期增加 1–2 周，影响 Phase K 正式实验。

### 2. Kitchen 亮点的更准确表述

今天明确了 Kitchen 的论文价值不是“已经实现完整结构化控制”，而是：

> 在 HumanoidBench Kitchen 这种稀疏奖励、长时程、接触丰富任务中，FTC-guided PBRS 能否把训练从纯预警推进到首个子任务完成。

因此 Kitchen 的目标被收敛为：

- 不是立即追求 0→3 或 0→4；
- 首要目标是 **0→1**：
  - K-A / Vanilla 保持 `success_subtasks = 0`；
  - K-C / TC-PPO + PBRS 能完成至少 1 个子任务。

这成为 Phase K 的核心判据。

### 3. PBRS 原始设计存在致命稀疏性，必须改成连续势函数

今天识别出一个关键漏洞：如果势函数只写成

\[
\Phi_{\text{prog}}(s)=\text{completed subtasks}
\]

那么在 Kitchen 训练中大部分时间：

\[
\Phi(s_t)=0,\quad \Phi(s_{t+1})=0
\]

导致 PBRS shaping 项仍然为 0。也就是说，虽然形式上加入了 PBRS，但训练早期几乎没有任何梯度改善。

因此决定将 Kitchen PBRS 改成连续进度势函数：

- 暴露每个子任务的连续距离；
- 用距离到目标状态的变化提供密集 shaping；
- 再叠加 `success_subtasks` 作为离散完成奖励。

这使 PBRS 从“完成瞬间才有信号”变成“接近目标过程也有信号”。

### 4. protect 模式下 PBRS 必须关闭

今天还修正了 `m_t` 的定义：

\[
m_t =
\begin{cases}
0, & FTC_t < \tau_{\text{low}} \quad \text{protect} \\
1, & FTC_t \ge \tau_{\text{low}} \land \text{Plateau}(K) \quad \text{push} \\
0, & \text{otherwise}
\end{cases}
\]

关键决策：

- protect 模式下不应该同时 push；
- 如果 FTC 已经低于安全阈值，应优先稳定训练；
- 只有 FTC 稳定但 reward / progress 平台化时，才打开 PBRS。

这让 [[FTC Metric]] 的三模式逻辑更清晰：

- protect：稳定性优先；
- normal：常规 PPO；
- push：进度推进。

### 2. Kitchen 连续距离信号暴露

在 HumanoidBench Kitchen task wrapper 的 `get_reward()` 中发现：

```python
distance = np.linalg.norm(
    next_obj_obs[..., element_idx] - self.goal[element_idx]
)
```

原本该距离只用于判断：

```python
complete = distance < 0.3
```

今天在 Server B 和 Server C 上将每个子任务的 distance 暴露到 `reward_dict` / `info` 中。

Server B 验证得到四个子任务距离字段：

- `dist_microwave = 0.7500`
- `dist_kettle = 0.4065`
- `dist_light_switch = 0.6919`
- `dist_bottom_burner = 0.8801`

这使 Kitchen 可以构造类似：

\[
\Phi_{\text{prog}}(s)
=
-\sum_i d_i(s)
+
10 \cdot \text{success\_subtasks}
\]

的连续势函数。

## 新洞察

### 1. PBRS 的关键不是“是否使用子任务计数”，而是“势函数是否在首个成功前有梯度”

今天最重要的技术洞察是：  
`success_subtasks` 本身虽然是 progress signal，但它仍是稀疏信号。对于 Kitchen 这种从 0 开始几乎无法完成第一个子任务的环境，仅靠完成计数构造 PBRS 不会改变早期学习动态。

真正有效的是：

- 子任务目标距离；
- 对象状态到目标状态的连续误差；
- 手/物体/目标之间的几何进展；
- 完成计数作为离散 bonus，而不是唯一势函数。

这为 [[FTC Metric]] + PBRS 的实验设计提供了更强的依据。

### 2. Kitchen 的论文价值来自“稀疏长时程下的阶段推进”，不是最终任务全完成

Kitchen 不应被包装成“我们解决了长时程结构化控制”。更稳妥也更有说服力的表述是：

> TC-PPO + PBRS 将 FTC 从 stability warning signal 推进为 phase-aware training scheduler，并在 Kitchen 这种稀疏长时程任务中尝试推动首个子目标突破。

如果 Phase K 能证明 0→1，就足以构成亮点；如果不能，也可以作为清晰的 future work / limitation。

### 3. protect 和 push 不能同时打开

今天对三模式机制的理解更清楚了：

- protect 是防崩溃；
- push 是突破平台；
- normal 是常规优化。

如果 FTC 已经低到需要 protect，说明策略更新已不稳定，此时打开 PBRS 只会把推进和保护混在一起，破坏机制解释性。因此 `m_t` 必须在 protect 中为 0。

### 4. 论文贡献必须按“证据等级”写

今天论文修改中最重要的写作原则：

- 已跑完并有统计结果的贡献，可以写成 empirical finding；
- 已设计、有公式、有理论支撑但未完成实验验证的贡献，只能写成 design / formalization；
- Kitchen 仍为 future validation 时，不能在 Abstract 或 Contributions 里暗示已经有效。

这直接降低了审稿人抓 overclaim 的风险。

### 5. 图表不是装饰，而是论文主线的第二套语言

今天图表修改后，整篇论文的视觉叙事更一致：

- 蓝色：Vanilla / controlled baseline；
- 橙色：miscalibrated / over-regularized / hard gating；
- 绿色：Mode A / successful gating / push or recovery；
- 黄色：target zone 或阈值带；
- 红色：collapse / not comparable warning。

这种颜色语义统一能帮助审稿人快速理解 [[Mode A-B-C Taxonomy]] 和 [[FTC Metric]] 的因果角色。
