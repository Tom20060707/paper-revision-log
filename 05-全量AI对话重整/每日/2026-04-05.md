# 2026-04-05 研究日记

## 今日主线

今天主要围绕 [[TC-PPO Algorithm]] 的门控阈值、[[FTC Metric]] 实验有效性、以及 [[Mode A-B-C Taxonomy]] 中 **Mode A 证据缺失** 展开。

核心推进有四件事：

1. **厘清了 GPU 显存异常的真实原因**  
   最初误以为 PyTorch 模型没有正确分配到多卡，后来确认模型本身很小，主进程约 **484 MiB** 是正常值；真正导致 GPU:0 显存堆积到约 **13 GB** 的是 MuJoCo EGL worker 全部落在 GPU:0。修复方向是显式设置 `EGL_DEVICE_ID`。

2. **更新并修正论文中的 FTC 范围与 64env scaling 结果**  
   新数据表明 `bookshelf_simple` 的 FTC 达到 **0.546**，超过此前 `bookshelf_hard` 的最大值，因此全文 FTC 上界需要从 **0.522** 更新到 **0.546**。同时加入了 Server C 的 64env scaling 结果。

3. **围绕 Mode A 设计了新的搜索实验策略**  
   讨论并否定了简单降低 `tau_high` 的做法，因为这会改变论文标准、降低证据含金量。最终坚持使用校准阈值：
   - `tau_low = 0.35`
   - `tau_high = 0.42`

4. **发现并反复核查历史实验中的阈值记录问题**  
   今天最大风险点是：历史实验实际使用的 `tau_low/tau_high` 一度非常混乱。多次核查后形成阶段性结论：不同服务器、不同脚本、不同实验批次的阈值不能简单混为一谈，必须按实验脚本和日志来源逐项确认。日志未打印 tau 参数是一个重要盲点，后续必须修复。

## 关键对话与决策

### 1. GPU 显存问题：PyTorch 不是瓶颈，EGL worker 才是瓶颈

一开始看到 GPU:1/2 只有约 **484 MiB**，曾怀疑代码里存在 `cuda:0` 硬编码。但用户随后指出：

- PPO 网络是 `[512×512×4]` 量级，参数约 **800K**
- 参数、优化器状态和 CUDA runtime 加起来约 **460–484 MiB**
- 因此主训练进程占用 **484 MiB** 是合理的

真正的问题来自 MuJoCo EGL worker：

- 每个 EGL worker 大约 **1042–1047 MiB**
- `--envs 4` 时每个任务约 4 个 EGL worker
- cabinet / powerlift / truck 共 3 个任务时，12 个 EGL worker 堆到 GPU:0，合计约 **12.5 GB**
- 这与监控图中 GPU:0 约 **13 GB** 完全吻合

决策：

- 不修改 PyTorch device 逻辑。
- 多卡运行时重点设置 `EGL_DEVICE_ID`，让 MuJoCo 渲染上下文分散到对应 GPU。
- 下一批任务启动后，预期显存应变成三条约 **4–5 GB** 的并行线。

### 2. Server C 64env 实验完成，并给出 scaling 结果

Server C（A800）汇报所有实验已完成，GPU 空闲。关键结果：

| 任务 | 算法 | seeds | final reward | best reward | FTC |
|---|---|---:|---:|---:|---:|
| truck | vanilla_ppo | 0 | 785.03 | 795.07 | 1.00 |
| truck | vanilla_ppo | 1 | 785.27 | 794.14 | 1.00 |
| truck | vanilla_ppo | 2 | 783.58 | 800.12 | 1.00 |
| truck | tc_ppo | 0 | 754.40 | 767.48 | 0.35 |
| truck | tc_ppo | 1 | 755.42 | 766.38 | 0.35 |
| truck | tc_ppo | 2 | 754.24 | 764.18 | 0.35 |
| powerlift | vanilla_ppo | 0 | 45.83 | 54.59 | 1.00 |
| powerlift | tc_ppo | 0 | 45.83 | 54.59 | 0.40 |

整理后的结论：

- truck 64env：
  - Vanilla PPO：约 **784.6±0.9**
  - TC-PPO：约 **754.7±0.6**
  - FTC(TC)：约 **0.35**
  - 解释为 [[Mode A-B-C Taxonomy]] 中的 **Mode B**：gate 打开后关不上，TC-PPO 对 truck 有负面影响。

- powerlift 64env：
  - Vanilla 与 TC-PPO final reward 均为 **45.83**
  - TC-PPO FTC 约 **0.40**
  - 倾向解释为 **Mode C** 或接近未有效触发状态：TC 约束生效有限，未提升 reward。

### 4. Mode A 搜索策略：不能降低标准，必须保持 τ=(0.35, 0.42)

围绕 Mode A 搜索，讨论了多种方案。

Mode A 的定义：

> FTC 跌破 `tau_low=0.35` → gate 打开 → TC-PPO 干预 → FTC 恢复超过 `tau_high=0.42` → gate 关闭。

必要条件：

1. 任务稳定期 FTC 必须高于 **0.42**，否则 gate 不可能关闭。
2. 训练过程中 FTC 必须短暂跌破 **0.35**，否则 gate 不会打开。
3. TC-PPO 干预后 FTC 必须恢复到 **0.42** 以上，才能形成完整闭环。

曾讨论是否将 `tau_high` 降到 0.38 或 0.40 来提高 Mode A 命中率，但最终否定。理由是：

- `tau_high=0.42` 是论文核心校准标准。
- 降低阈值相当于降低证明难度，会削弱 Mode A 证据含金量。
- 用户用“600 分上 985 改成 500 分”的类比明确指出：不能通过降低标准制造 Mode A。

最终决策：

- `tau_low=0.35` 不变。
- `tau_high=0.42` 不变。
- 通过任务选择、随机种子数量和训练长度提高 Mode A 命中概率，而不是改阈值。

### 5. n_envs 的取舍：大并行环境不一定更利于 Mode A

一度考虑在 A800 80GB 上将 `n_envs` 提高到 64 或 128，以充分利用显存。但进一步分析后认为：

- `n_envs` 越大，FTC 统计越平滑。
- 64 个环境中个别环境失稳，对总体 FTC 的影响只有约 1/64。
- Mode A 需要捕捉 FTC 的瞬时跌落，因此过大的 `n_envs` 反而可能降低触发概率。

因此更合理的策略是：

- 使用较小 `n_envs=4`，保持 FTC 对失稳更敏感。
- 增加 seeds 数量，提高随机初始化覆盖率。
- 选择稳定 FTC 高于 0.42、但训练过程可能波动的任务。

后续又讨论了是否可将 seeds 扩大到 50、100、200 甚至更多。结论：

- 理论上 seeds 越多越好。
- 但 1000、10000 seeds 会带来不可接受的时间成本。
- A800 80GB 下，一批大约可并行几十个 n_envs=4 实验。
- **50 seeds 级别**可能是较好的性价比上限：一批或少数批次可完成，几小时内得到结果。

### 6. Mode A 候选任务重新排序

最初优先考虑：

- `bookshelf_hard`
- `bookshelf_simple`
- `insert_small`
- `insert_normal`

后续发现 `train_real.py` 的 argparse choices 中一度不支持 insert 系列，需要额外修改。随后又重新审视任务是否真的最容易触发 Mode A。

候选任务逻辑：

- FTC 太低：gate 关不上，容易 Mode B。
- FTC 太高且训练稳定：可能从不跌破 0.35，容易 Mode C。
- 最有希望的是：稳定 FTC 略高于 0.42，但训练中有波动和崩溃风险。

已知 FTC 数据中，可能更有希望的任务包括：

| 任务 | 稳定 FTC | 判断 |
|---|---:|---|
| bookshelf_simple | 0.546 | FTC 天花板最高，但可能太稳定 |
| bookshelf_hard | 约 0.514 | 高 FTC，reward 方差较大，是强候选 |
| stair | 约 0.472 | 稳定 FTC 适中，可能更容易跌破 |
| basketball | 约 0.471 | 接触复杂，可能有波动 |
| maze | 约 0.455 | 候选 |
| slide | 约 0.460 | 候选 |
| spoon | 约 0.453 | 候选 |
| insert_small | 约 0.456 | 候选，但脚本支持需确认 |

阶段性判断：

- `bookshelf_hard/simple` 是最容易“关门”的任务。
- `stair/basketball/insert_small` 可能更容易“开门”。
- 最优实验不应只押 bookshelf，应覆盖“高 FTC + 中等 FTC 高波动”任务组合。

### 7. 阈值历史核查：今天最大风险点

今天反复围绕 Server B / Server C 历史实验实际使用的 `tau_low/tau_high` 发生混乱。最终明确需要区分：

- 当前脚本值
- 历史脚本值
- 显式命令行参数
- train 函数内部默认值
- argparse 默认值
- 旧版 wandb config 中的 `ftc_threshold_low/high`
- 当前实现中的 `tau_low/tau_high`

几个关键事实：

1. 有些日志不打印 tau 参数，无法直接从日志头部反推。
2. `summary.json` 只记录 reward / FTC / stabilizing_active，不记录 tau。
3. `config.json` 中旧字段 `ftc_threshold_low/high` 可能属于旧实现，不一定等同于当前 `tau_low/tau_high`。
4. 仅凭当前脚本 grep 不能证明历史实验参数，因为脚本可能已经被修改。
5. 未来必须在日志和 summary/config 中显式写入 `tau_low`、`tau_high`，否则实验可追溯性不足。

阶段性结论需要谨慎表述：

- Server C 的 64env scaling 实验可由 `run_scaling_64env.sh` 显式参数确认：`tau_low=0.35, tau_high=0.42`。
- `bookshelf` 新日志显示没有打印 tau，但启动逻辑倾向使用当前默认 `0.35/0.42`；仍应视为“需用脚本版本和启动时间进一步确认”。
- Server B 历史 sweep 是否是 `0.8/0.9` 诊断失败实验，还是后续已修正的 `0.35/0.42`，需要基于对应启动脚本、git diff 或备份文件进一步确认，不能再凭口头记忆下最终结论。

## 新洞察

### 1. Mode A 不是简单靠“更多环境”就能出现

今天最重要的方法论洞察是：

> 找 Mode A 不是追求训练最稳定，而是要捕捉“可恢复的不稳定”。

因此：

- `n_envs=64` 有利于稳定训练和 scaling，但不一定有利于 Mode A。
- `n_envs=4` 更容易暴露单个环境失稳对 FTC 的影响。
- Mode A 搜索应更像“故障注入/随机覆盖”，而不是单纯大规模稳定训练。

### 2. 任务选择要同时满足“能开门”和“能关门”

Mode A 需要两端都满足：

- 能开门：FTC 必须短暂低于 0.35。
- 能关门：FTC 必须恢复高于 0.42。

因此候选任务不是 FTC 越高越好，也不是越难越好，而是要处在一个“可跌落、可恢复”的区间。

这意味着：

- truck 这类 FTC 天花板低的任务容易 Mode B。
- powerlift 这类可能不触发的任务容易 Mode C。
- bookshelf_hard、stair、basketball、insert_small 这类任务更值得继续搜索。

### 3. 历史实验参数可追溯性比想象中更关键

今天围绕 `tau_high=0.9 / 0.50 / 0.42` 的混乱说明：

- 如果日志不记录关键超参数，后续论文写作会非常危险。
- 仅看当前脚本默认值无法证明历史实验实际值。
- wandb/config 的旧字段也可能误导。
- 以后所有 TC-PPO 实验必须把 `tau_low/tau_high` 写入：
  - stdout 开头
  - config.json
  - summary.json
  - 最终论文数据表的 metadata

这不仅是工程问题，也是论文可信度问题。

### 4. “失败数据”也可能是论文证据，但必须定义清楚

如果某批实验确实用了过高阈值，例如旧版 `0.8/0.9`，它不一定完全作废。它可以作为：

- 直接借用人类步态阈值的失败案例
- [[FTC Metric]] 校准必要性的证据
- [[Mode A-B-C Taxonomy]] 中 Mode B 的诊断性案例

但前提是论文必须明确说明该批实验是“诊断性失败实验”，不能和校准后的 `0.35/0.42` 主结果混用。

### 风险

- **历史数据混用风险**  
  如果不同 tau 的实验被放在同一表格里，论文结论会被质疑。

- **Mode A 搜索失败风险**  
  即使用正确阈值和大量 seeds，也不保证能观测到完整闭环。需要准备备用叙事：Mode A 是理论目标，已观测到 Mode B/C，Mode A 作为未来工作或补充实验。

- **A800 显存利用与实验效率的矛盾**  
  `n_envs=4` 更利于 Mode A，但显存利用率低。可以通过增加 seeds 和任务数填满批次，而不是增大 envs。

- **脚本参数名混乱风险**  
  之后所有提示词必须基于当前脚本实际参数名，避免再生成不可运行命令。
