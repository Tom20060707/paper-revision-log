### 4. `Emergency stop` / `no legal \end found` 根因分析

用户提供 `output.log` 后，重点排查为什么仍然无法编译。日志中关键证据：

## 新洞察

### 1. “编译失败”不一定是源码失败，可能是入口文件失败

今天最大的工程教训是：用户看到的 `Emergency stop` 不一定来自目标论文文件。日志中 `**main.tex` 是关键证据，说明平台或编辑器实际编译的是 `main.tex`。这解释了为什么本地 V4 能编译，而用户平台仍失败。

后续处理 LaTeX 问题时，应优先确认：

1. 日志第一行实际编译入口。
2. 日志中第一处 `!` 报错。
3. 是否运行了 BibTeX / Biber。
4. 是否打开了旧 PDF。
5. 是否项目中存在多个 `.tex` 主文件。

### 2. References “消失”通常是 BibTeX 流程或主文件设置问题

`.bbl` 已有 47 条 `\bibitem`，说明 bibliography 数据生成是成功的。如果 PDF 没显示 References，更可能是：

- 用户没跑完整编译链；
- 编译的是模板 `main.tex`；
- 平台缓存旧 PDF；
- 主文件没有使用 `ref_v4_45.bib`；
- `\bibliography{...}` 和 `.bib` 文件名大小写不一致。

这对论文工程复现非常关键。之后应避免只说“已编译成功”，而应同时给出：

- 主文件名；
- bibliography 数据库；
- `.bbl` 条目数；
- PDF 中是否能搜索到 `References`；
- PDF 中是否能搜索到关键文献名，如 `Schulman`、`Haarnoja`。

### 3. `\nocite{*}` 是短期满足“参考文献数量”的办法，但有审稿风险

为了“不改正文，只改 references”，今天采用了 `\nocite{*}` 输出约 45 条参考文献。这满足数量要求，但从审稿视角有风险：

- 文献虽然出现在 References，但正文没有逐条引用。
- 审稿人可能认为引用堆砌。
- 更好的最终版本应把关键文献自然嵌入 Related Work / Method / Experiments，而不是只靠 `\nocite{*}`。

明天需要考虑是否把 5 条核心文献合理写入正文引用位置，尤其是：

- PPO 放在算法背景。
- SAC 放在 off-policy RL comparison。
- DreamerV3 / TD-MPC2 放在 world-model RL comparison。
- HumanoidBench 放在 humanoid benchmark discussion。

### 4. Figure 10 的“有无代码”判断需要以 label 和源码为准

今天纠正了一个重要误判：`fig:ftc_specificity` 并非没有代码，它包含完整 `tikzpicture` 与 `groupplot`。以后整理论文图源时，不能依赖印象或外部回答，应直接搜索 label 并检查 figure 环境。

这也说明论文图源当前是混合模式：

- 部分图是 TikZ/PGFPlots 内嵌。
- 部分图通过外部文件引用。
- 部分外部 `\input{fig_*.tex}` 已被删除。

因此后续如果要重构图表，必须先建立一张 “figure label → source type → file/line range” 索引表。

### 5. 对 [[FTC Metric]] 的论文叙事正在更清晰

从 Figure 2、Figure 10、Figure 11 的定位可见，论文围绕 [[FTC Metric]] 的证据链大致是：
