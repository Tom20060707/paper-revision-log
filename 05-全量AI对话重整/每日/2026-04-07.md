## 关键对话与决策

### 1. Server B 崩溃原因定位：CPU/MuJoCo 瓶颈，而非 GPU 瓶颈

一开始观察到 Server B 的 GPU 显存占用约 4000–5000 MiB，但 GPU 计算利用率接近 0%。分析认为 PPO + MuJoCo 的训练瓶颈主要在 CPU 环境仿真：

- 大约 **95% 时间**在 MuJoCo 环境仿真；
- 大约 **5% 时间**在神经网络更新；
- 多进程同时跑时，GPU 被短暂轮流使用，因此监控上看起来利用率很低。

随后 Server B 出现严重资源问题。用户反馈 AutoDL 重启后服务器曾崩溃，CPU 利用率达到约 **5000%**，内存接近爆满，GPU/显存无法正常加载。

检查后确认：

- AutoDL 机器配置：
  - CPU：54 核；
  - 内存：180 GB；
  - GPU：NVIDIA GeForce RTX 4090 D × 3；
  - 系统盘：30G；
  - 数据盘 `/root/autodl-tmp`：100G。
- 重启后训练进程基本消失，只剩系统进程。
- 上次爆掉的根本原因是训练进程太多：  
  **27 个进程 × 每进程 4 envs = 108 个 MuJoCo 子进程**，远超 54 核 CPU 的舒适区。

决策：

- 以后 Server B 不再使用过多并行进程；
- 推荐保守上限为每卡 3 个进程，共 9 个进程；
- 若每进程 `--envs 4`，则约 36 个 MuJoCo 子进程，给 54 核 CPU 留出余量；
- 后续若启动补跑，优先控制在 8–9 个进程以内。

## 论文/实验/代码/文档产物

### 1. [[Mode A-B-C Taxonomy]] 的实证状态被重新定性

今日最重要的科研结论：**当前全局阈值 `τ_low=0.35, τ_high=0.42` 下，两台服务器的数据没有直接观测到 Mode A。**

Mode A 的必要特征：

1. 同一个 run 内 `stabilizing_active` 从 `False → True`；
2. 随后 FTC 恢复到 `τ_high` 以上并满足连续窗口条件；
3. `stabilizing_active` 从 `True → False`。

Server B 的历史汇总：

| 实验 | gate 激活时刻 | 激活时 FTC | 后续 FTC | 最终状态 |
|---|---:|---:|---:|---|
| truck_tc_64_s0 | 5,111,808 steps | 0.344 | 0.344→0.352 | stab=1 |
| truck_tc_64_s1 | 3,801,088 steps | 0.347 | 0.347→0.349 | stab=1 |
| truck_tc_64_s2 | 3,801,088 steps | 0.349 | 0.349→0.351 | stab=1 |

解释：

- truck 确实触发 gate；
- 但 FTC 只恢复到约 0.349–0.352，远低于 `τ_high=0.42`；
- gate 从未关闭；
- 因此 truck 是 [[Mode B]]，不是 Mode A。

其他 Server B 任务：

- walk、stair、bookshelf_hard、powerlift、insert_small、run 等均未触发 stabilizing，属于 [[Mode C]] 或未完成/无效实验；
- basketball crash/OOM，未形成有效运行。

Server C 也未找到 True→False 的 Mode A 序列。用户汇总显示：

- 稳定触发 True 的任务包括 balance_hard/simple、basketball、bookshelf_hard/simple、insert_small、kitchen、maze、package、pole、sit_simple、slide、spoon、stand、truck 等；
- 从不触发 False 的任务包括 crawl、cube、door、hurdle、insert_normal、reach、window 等；
- 混合任务包括 walk、powerlift、push、stair、cabinet 等，但未确认存在同一 run 内 True→False 的 Mode A；
- 很多 True/False 差异可能来自早期重跑、步数不足或参数差异。

结论：

- 目前没有可信 Mode A 实例；
- “触发过 gate”不能等价于 Mode A；
- Mode A 必须包含 gate 自动关闭。

### 2. 对 Mode A 缺失的解释转向 HumanoidBench 结构局限

今天形成的新论文叙事：

> HumanoidBench 在全局阈值下呈现结构性 FTC 分布：高 FTC 任务从不触发 gate，低 FTC 任务触发后又达不到 `τ_high`，导致 gate 锁定。因此缺少“短暂跌破 `τ_low` 后自然恢复到 `τ_high` 以上”的任务类型。

任务分布被理解为近似双峰：

- 高 FTC/稳定任务：
  - FTC 通常高于 `τ_high=0.42`；
  - 从不跌破 `τ_low=0.35`；
  - 对应 [[Mode C]]；
  - 例如 walk、stair、bookshelf 等。
- 低 FTC/不规则接触任务：
  - FTC 接近或低于 0.35–0.42；
  - 一旦触发 gate，很难恢复到 `τ_high=0.42`；
  - 对应 [[Mode B]]；
  - 例如 truck。
- 缺失的任务类型：
  - 正常稳定 FTC > 0.42；
  - 训练中短暂跌破 0.35；
  - 随后自然恢复到 0.42 以上；
  - 这正是 Mode A 所需的“瞬态失稳—恢复”场景。

这不是简单的实验设置错误，而是 [[HumanoidBench]] 对接触动力学覆盖范围的结构性限制。论文中应把它写成 [[FTC Metric]] 的诊断性发现，而不是硬声称已经验证了 detect→stabilize→recover 的完整闭环。

### 3. insert_small Mode A 实验复盘：阈值灰区而非触发

Server C 正在跑 `modeA_insert_small_seed*.log`，共 17 个 seeds 左右。检查时：

- 多数 seed FTC 在 **0.41–0.46**；
- seed12 最低约 **0.385**；
- 仍未跌破 `τ_low=0.35`。

触发逻辑确认：

```text
if FTC < τ_low:
    stabilizing = True

if FTC >= τ_high 连续 W 次:
    stabilizing = False
```

因此 insert_small 的状态是：

- FTC 多数位于 `τ_low=0.35` 和 `τ_high=0.42` 之间或略高；
- 不够低，无法触发 stabilizing；
- gate 未打开时，`τ_high` 无实际作用；
- 所以仍属于 [[Mode C]]，不是 Mode A。

关键结论：

- `τ_low=0.35, τ_high=0.42` 不是随意错误设置，而是来自此前的物理/崩溃前区间标定；
- 若强行改为如 `τ_low=0.40, τ_high=0.44` 来触发 insert_small，会显得像为触发 Mode A 而调参，审稿风险很高；
- 不建议继续追加“找 Mode A”实验。

## 新洞察

### 1. Mode A 不是“还没找到”，而是当前 benchmark 分布中结构性缺席

此前的思路一直是：继续找更可能触发 Mode A 的 HumanoidBench 任务，或者增加并行环境/训练步数。今天的复盘表明，这条路大概率已经走到尽头：

- 已尝试更高并行数，如 64 env；
- 已尝试长训练，如 100M；
- 已尝试被认为最可能触发的任务；
- 仍然没有 True→False 的 gate 序列。

这说明问题不在于算力不够或任务没选对，而是 HumanoidBench 在全局阈值下缺少“瞬态失稳后恢复”的任务类型。

这对论文非常关键：  
[[TC-PPO Algorithm]] 的 gating 机制若要证明“有益”，最需要 Mode A；但当前实验证据只覆盖：

- [[Mode B]]：gate 激活后锁定，可能有害；
- [[Mode C]]：gate 不触发，等价于 Vanilla 或近似零损耗。

因此论文主贡献必须重新定位：

- 不应把 TC-PPO gating 的 detect→stabilize→recover 作为已实证核心贡献；
- 应强调 [[FTC Metric]] 作为诊断指标，揭示 HumanoidBench 接触动力学结构；
- 网络容量 `[512×4]` 是性能提升的主要来源；
- gating 是一个可解释但尚需更合适 benchmark/per-task calibration 验证的机制。

### 2. truck 是最清晰的 Mode B 证据

truck 的结果很有价值，但不是 Mode A：

- `τ_low=0.35` 附近触发 gate；
- 激活后 FTC 卡在 0.344–0.352；
- 永远达不到 `τ_high=0.42`；
- gate 无法关闭。

这恰好说明：

- 全局阈值可能对低 FTC 天花板任务过于苛刻；
- `τ_high` 若高于任务自然 FTC 上限，会导致永久保守更新；
- [[Mode B]] 是实际风险，不是理论边角情况。

论文中应更坦诚讨论 Mode B，而不是把它包装成小限制。

### 3. insert_small 揭示了阈值灰区问题

insert_small 的 FTC 多数在 0.41–0.46，最低也只到约 0.385。这类任务暴露了 hysteresis gate 的另一个区域：

- FTC 低于 `τ_high` 时不代表会触发；
- 只要没跌破 `τ_low`，gate 就保持关闭；
- 因而存在一个“灰色地带”：看起来接近边界，但机制上仍是 Mode C。

这说明论文中解释 Mode C 时不能只写 “FTC > τ_high”；更严格地说，Mode C 是 **gate never fired**，原因可以是 FTC 始终未跌破 `τ_low`，即使有些时刻低于 `τ_high`。

### 4. argparse 白名单不是环境真实性问题

reach/room/push 的事件提醒：入口脚本的白名单限制不能直接等价为 benchmark 不支持。

正确判断顺序应该是：

1. 先查 `gymnasium` registry 是否存在环境；
2. 再查训练脚本是否允许该 `--env`；
3. 如果 registry 存在但 argparse 拒绝，则只是脚本入口限制；
4. 修改 choices 不改变环境、奖励、动力学或训练逻辑，因此可视为真实任务运行。

这对后续复现实验很重要，避免把工程限制误写成科学结论。
