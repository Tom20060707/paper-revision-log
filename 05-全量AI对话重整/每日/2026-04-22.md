   今天最重要的新洞察是：TC-PPO 不应只被叙述为“预警器/保护器”，还可以通过 [[Potential-Based Reward Shaping]] 变成“助推器”。具体做法是引入 **FTC-guided Progress-Based Reward Shaping（PBRS）**：在 FTC 稳定但任务进度停滞时进入 Push mode，用任务进度势函数提供密集学习信号，尝试突破类似 `kitchen` 这种 Type-II Mode C 高难任务。

## 关键对话与决策

### 1. HumanoidBench 27 vs 24 任务：不是遗漏，而是机器人配置差异

最早的问题是：官方 [[HumanoidBench]] 说有 27 个任务，而本文似乎只完成了 24 个任务，是否存在缺失。

今天确认：

- 官方共有 **27 个任务**。
- 本文基于标准 `h1` 配置评估 **24 个任务**。
- 真正没有纳入的是：
  - `door`
  - `window`
  - `cube`

排除原因：

- 这三个任务需要 **`h1hand` 变体**，即 H1 + Shadow Hand。
- 需要 21-DoF 灵巧手参与精细操作。
- 标准 H1 无手配置无法合理执行这些任务。
- 若切换到 `h1hand`：
  - 动作空间从约 19 维扩展到约 61 维；
  - observation space 会变化；
  - 接触拓扑从足地接触扩展到手指精细接触；
  - 需要重新验证 [[FTC Metric]] / [[GCTC]] 在灵巧手接触模式下是否有效。

**决策：**  
当前论文不补做 `door/window/cube`，而是在 Experimental Setup / Limitations 中说明排除原因，并把 dexterous manipulation 留作 future work。

### 2. 19 维动作空间与 6 个 FTC 通道的关系

今天澄清了一个重要概念误区：

- **19 维动作空间**：H1 标准机器人被控制的关节自由度，是 PPO 每一步输出的 action。
- **6 个 FTC 通道**：FTC 使用的接触观测点，不是控制自由度。

19 维动作空间大致对应：

| 部位 | 关节数 |
|---|---:|
| 双髋关节 | 6 |
| 双膝 | 2 |
| 双踝 | 2 |
| 腰部 | 1 |
| 双肩 | 6 |
| 双肘 | 2 |
| 合计 | 19 |

6 个 FTC 通道为：

- 2 个足部接触；
- 2 个手部/末端接触；
- 2 个膝部接触。

两者关系是：

> 关节 action 驱动机器人运动 → 身体部位产生接触事件 → FTC 从接触时间序列中计算接触稳定性。

**决策：**  
论文中补充说明：6 通道是针对双足/全身接触任务的物理启发式设计，而不是机器人控制自由度。自动通道选择可行，但不作为本文贡献，放入 Limitations / Future Work。

### 4. 扰动实验规划：失败标准、公平性、极限测试

今天用户指出实验规划中三个关键漏洞：

#### 4.1 “失败”定义是否有依据？

原先规划中使用“reward 下降超过 20%”作为失败标准。今天明确：

- 20% 不是物理定律，而是经验阈值；
- 但论文中已有 Eq.(Collapse Depth) 使用类似定义；
- 因此 Phase 1 的失败标准应与 Collapse Depth 保持一致。

**决策：**  
论文和规划中统一失败标准：以峰值后 reward 下降超过 20% 作为 collapse / failure 判据，并说明这是与 Collapse Depth 对齐的 operational criterion。

#### 4.2 对照实验公平性问题

原规划中：

- Phase 1：只用外力脉冲扫描 Force Scale；
- Phase 2：正式对照时加入四种噪声。

这可能导致不公平：单外力下确定的 F\*，在四噪声组合下可能不是同一个难度水平。

**新增修复：Phase 1 fairness validation**

- 单噪声确定 F\* 后；
- Phase 2 前先跑一次全噪声 + F\* 快速验证；
- 若 Vanilla PPO 在全噪声下退化过强，则回退到前一个扫描点；
- 目标是保证 Phase 2 是“受控困难”而不是“噪声堆叠导致无意义崩溃”。

规划中写入：

- 快速验证约 1M steps；
- 若全噪声组合导致退化超过约 30%，则降低 F\*。

#### 4.3 新增 Phase 3 极限测试

新增 robustness boundary experiment：

- 在 TC-PPO 已能承受 F\* 的前提下；
- 继续提高外力扰动，如 F\* × 1.5 / 2.0 / 3.0；
- 观察 TC-PPO 何时彻底失败；
- 得到鲁棒性边界 F_max。

目的：

- 不只证明“TC-PPO 比 Vanilla 好”；
- 还回答“TC-PPO 的鲁棒性上界在哪里”。

**规划更新结果：**

- 时间线：约 10 天 → 12 天；
- GPU·h 估算：约 213h → 约 250h；
- 实验数量估算：约 49 个；
- 实际等待时间因多 GPU 并行约为 40–45 小时，而不是 250 小时。

### 5. 四阶段算法演化叙事写入论文

今天还重构了论文叙事，把算法从零散模块整理为四阶段演化：

1. **Stage I：过大的 τ 值**  
   初始阶段没有充分考虑人形机器人和人的接触差异，阈值设定过大，导致机制不稳定或过度敏感。

2. **Stage II：硬门控 + 生物力学参考**  
   采用固定阈值，例如 0.35 / 0.42，并结合生物力学文献理解 CV 相关非线性，从而降低 τ 并引入 hysteretic hard gating。

3. **Stage III：软门控 + 自适应阈值**  
   τ 随 FTC 分布变化而变化，进入 soft gate / adaptive thresholds 的方向。

4. **Stage IV：CUSUM + GAE + 噪声扰动**  
   引入 CUSUM 异常积累、FTC-weighted GAE，并在扰动环境下验证鲁棒性。

**论文修改：**

- Abstract 中加入四阶段演化；
- Introduction 贡献列表前加入 four-stage algorithm evolution；
- Paper organization 改成按四阶段组织；
- Section 5 子标题加 Stage I / Stage II；
- Limitations 中相关段落加 Stage III 指向。

### 10. Mode C 被拆成 Type-I / Type-II，并导向 PBRS

今天的重要理论推进之一是发现：同为 Mode C，其实有两种本质不同的情况。

#### Type-I Mode C：任务本身接触稳定，不需要保护

例如：

- `balance_simple`
- `walk`

特点：

- FTC 天然高；
- 接触质量好；
- 门控不触发；
- TC-PPO ≈ Vanilla PPO；
- 这类任务不是本文算法的主要用武之地。

一句话：

> Type-I Mode C 是“不需要我”。

#### Type-II Mode C：任务极难，从未进入有效学习阶段

例如：

- `kitchen`
- 可能还有部分 long-horizon manipulation 任务。

特点：

- reward 长期为 0；
- 没有“学会后崩溃”的过程；
- FTC 不是明显下降，而是缺少有效探索；
- 原 TC-PPO 的保护机制帮不上忙。

一句话：

> Type-II Mode C 是“我现在帮不了，但值得升级后去帮”。

这直接引出今天最大的算法升级方向：  
将 TC-PPO 从 **warning/protection mechanism** 扩展成 **push/booster mechanism**。

### 11. 新算法升级：FTC-guided Progress-Based Reward Shaping

今天搜索和讨论了控制理论、信息论、动力学系统、自适应控制、神经科学、优化理论、安全 RL 等方向，最后决定不把所有机制都堆进论文，而是收敛到一条最干净的主线：

> **FTC-guided Progress-Based Reward Shaping（PBRS）**

核心思想：

- 原 TC-PPO：FTC 低 → 保护，防止训练崩溃；
- 新扩展：FTC 稳定但任务进度停滞 → push，提供 progress shaping；
- FTC 不直接作为任务 reward，而作为“阶段选择器”。

必须加入的公式：

\[
r'_t = r_t + \lambda_{\text{prog}} \left(\gamma \Phi_{\text{prog}}(s_{t+1}) - \Phi_{\text{prog}}(s_t)\right)
\]

其中：

- \(\Phi_{\text{prog}}(s)\)：任务进度势函数；
- 对 `kitchen` 来说，可设为已完成子任务的加权和；
- \(\lambda_{\text{prog}}\)：shaping 权重；
- 该形式来自 [[Potential-Based Reward Shaping]]，可引用 Ng et al. 1999；
- 理论优势是：在标准条件下不改变最优策略，只改善学习信号密度。

新的三种操作模式：

1. **Protect mode**  
   FTC < \(\tau_{\text{low}}\)，触发保守更新，收紧 KL / entropy 等，防 collapse。

2. **Push mode**  
   FTC 稳定，但 return slope / progress slope 长期停滞，激活 progress shaping，推动探索和任务进展。

3. **Normal mode**  
   FTC 正常且任务仍有进展，执行标准 PPO / TC-PPO 更新。

平台期检测：

- 用最近 N 个 episode 的 return slope；
- 若 return slope < \(\epsilon_{\text{progress}}\)，判定 plateau；
- 若同时 FTC 稳定，则进入 Push mode。

**决策：**  
把 PBRS 写入当前论文，而不是作为下一篇论文。实验上优先冲 `kitchen`，验证能否从 0 推到完成 1–2 个子任务。

## 新洞察

### 1. TC-PPO 的核心定位应从“更强 PPO”转向“事件驱动训练控制器”

今天明确：TC-PPO 如果被包装成“又一个更强的 PPO”，就必须在 benchmark 分数上显著胜出；但当前数据并不够戏剧化。

更合理的定位是：

> FTC 是 PPO 训练崩塌的领先指标，TC-PPO 用它进行事件驱动的 trust-region 控制。

这让评价标准从“最终 reward 最高”转为：

1. FTC 是否比 reward collapse 更早预警；
2. 门控是否把不可恢复 collapse 变成可恢复；
3. 这种控制是否跨任务、跨容量、跨扰动可重复。

这与 [[Mode A-B-C Taxonomy]] 更匹配。

---

### 2. Type-I / Type-II Mode C 是适用边界，也是新贡献入口

原先 Mode C 只是“门控不触发”。今天拆解后，发现：

- Type-I Mode C：任务稳，不需要 TC-PPO；
- Type-II Mode C：任务难，TC-PPO 保护机制帮不上，但可能通过 PBRS 升级为助推机制。

这让失败不再只是 negative result，而变成了下一步算法设计的入口。

---

### 3. PBRS 是最适合当前论文的“助推器”升级路线

跨学科搜索后，虽然有很多方向可选：

- Lyapunov / CLF / CBF；
- empowerment / curiosity；
- attractor dynamics；
- variable impedance；
- contact mode segmentation；
- natural gradient；
- safety shield；

但当前论文最适合的不是堆机制，而是选择最小充分扩展：

> FTC-guided PBRS = 保留原 protect 机制，只在 plateau 时激活 progress potential shaping。

优点：

- 理论上有 Ng et al. 1999 的 PBRS 支撑；
- 工程改动比迁移框架小；
- 论文叙事清晰；
- 能直接瞄准 `kitchen` 这类 Type-II Mode C 任务。

---

### 4. mjlab 的价值是未来平台，不是当前救命方案

mjlab 的价值主要在：

- GPU 并行仿真；
- manager-based API；
- contact sensor / terrain / curriculum / event 模块化；
- 可复现性和工程结构。

但它当前不能直接救这篇论文：

- 平台不一致；
- 任务不一致；
- 机器人不一致；
- 数据不能混用；
- 迁移成本高。

当前最优策略是：

> 当前论文继续 HumanoidBench 路线；mjlab 写入 Related Work / Future Work；下一篇或扩展版再考虑迁移。

---

### 5. 当前论文最缺的是“硬证据”，不是更多概念

今天多次比较后确认，TC-PPO 当前短板不是 idea，而是 evidence。

最需要补的是：

- 扰动下 Vanilla 崩溃、TC-PPO 存活的曲线；
- FTC 领先 reward collapse 的统计证据；
- CUSUM / GAE / PBRS 的最小消融；
- Kitchen 从 0 到非零子任务完成数的突破证据。

---

## 待办与风险

### 明日优先待办

1. **检查 PBRS 论文修改质量**
   - 确认新增 PBRS 段落没有破坏原有四阶段叙事；
   - 检查 Protect / Push / Normal 与旧 Mode A/B/C 是否表述一致；
   - 避免“控制器”措辞引来过强稳定性证明要求。

2. **确认 HumanoidBench `kitchen` 的进度信号接口**
   - 是否能直接读取 microwave、kettle、burner、light switch 等子任务完成状态；
   - 若不能，需要从 env info / reward components 中提取；
   - 确认 \(\Phi_{\text{prog}}(s)\) 的实现方式。

3. **实现最小 PBRS wrapper**
   - 不改 PPO 主循环；
   - 先在 env reward 层加入 progress shaping；
   - 增加 push mode detector；
   - 记录：
     - return slope；
     - progress count；
     - FTC mean / variance；
     - push mode activation ratio。

4. **先跑 Kitchen 快速验证**
   - Vanilla PPO；
   - TC-PPO protect-only；
   - TC-PPO + PBRS；
   - 至少 seed 0 快速判断是否有从 0 到非零 progress 的迹象。

5. **扰动实验继续按 v2 规划推进**
   - stair / slide 优先；
   - Phase 1 扫描；
   - Phase 1 fairness validation；
   - Phase 2 正式对比；
   - Phase 3 极限测试。

6. **增加 FTC 领先性统计脚本**
   - 计算 `T_FTC`、`T_reward`、`lead_time`；
   - 输出跨 seed 汇总；
   - 后续可画成机制证明图。

---

### 主要风险

1. **PBRS 可能把论文变复杂**
   - 风险：审稿人觉得从预警、门控、CUSUM、GAE 又加 PBRS，模块过多。
   - 缓解：强调 PBRS 是 Type-II Mode C 的 extension，不是所有实验的必要模块。

2. **Kitchen 可能仍然没有突破**
   - 风险：PBRS 给了进度信号，但 H1 仍学不会复杂长时程操作。
   - 缓解：先以“完成 1 个子目标”作为成功迹象，不要求完整 kitchen success。

3. **PBRS 的 progress potential 可能依赖任务特定信息**
   - 风险：被质疑不够通用。
   - 缓解：表述为 task-progress potential，可由环境提供或由检测器估计；当前 kitchen 是 proof-of-concept。

4. **Potential shaping 理论条件需要谨慎**
   - 风险：PBRS 最优策略不变性需要严格形式与相同 discount 条件。
   - 缓解：论文中写“inherits the standard policy-invariance property under the PBRS assumptions”，不要过度声称。

5. **mjlab 速度诱惑可能干扰当前主线**
   - 风险：临时迁移导致 3–4 周工程重写。
   - 缓解：当前不迁移，只引用，保持 HumanoidBench 数据一致性。
