### 3. 鲁棒性协议：six-component → five-component

根据 `噪声+扰动.docx`，今天做出重要实验设计决策：

**正式鲁棒性协议删除 observation noise。**

理由：

- [[FTC Metric]] 基于接触时间戳 / 接触节律；
- observation noise 对 FTC 的因果链较远；
- 若把 observation noise 纳入核心协议，容易削弱论文主张；
- 更合适的扰动是直接影响接触动力学或控制时序的因素。

正式保留的 five-component protocol：

1. 外力脉冲；
2. 摩擦随机化；
3. 关节力矩噪声；
4. 动作延迟 / control latency；
5. 关节阻尼随机化。

新增但降级为 P2 可选：

- 质量随机化 ±15%；
- 理由：顶刊 sim-to-real 标配，但它主要改变接触力分布，对 FTC 时序影响更间接。

同时确定实验有效性不能只看 reward 是否更高，而要满足机制条件：

1. Vanilla 在扰动下出现明确退化；
2. FTC 时序显示 Mode A 真的触发；
3. gate ON / OFF 与 reward 稳定或恢复阶段对齐；
4. 无扰动或清洁对照下 TC-PPO 与 Vanilla 不应出现无关差距；
5. 对 control latency，最好看到延迟步数增加时 Vanilla 下降更明显，而 TC-PPO 下降更小。

### 5. 服务器资源利用率：确认 CPU-bound 与 EGL 限制

今天对训练性能的理解明显推进。

结论：

- 当前 SB3 + MuJoCo + MLP PPO 的主要瓶颈是 **MuJoCo env.step() 的 CPU 物理仿真**。
- GPU 只在 policy update 和少量推理中使用，因此 5–15% GPU 利用率是正常现象。
- 当前模式并不是“显卡没调好”，而是 **技术栈与租用硬件类型不匹配**。

估算逻辑：

- rollout 阶段占总时间约 90–95%；
- policy update 约占 5–10%；
- 因此 GPU 利用率天然较低。

硬件选择决策：

- 对当前 TC-PPO 论文，不建议中途切换到 MJX / Isaac Lab，因为会破坏已有可复现基线与任务结果。
- 更实际的方案是：继续 MuJoCo，但选择 **大核 CPU + 单张中等 GPU** 的机器，而不是多张高端 GPU。
- 当前 Server B 的 3×4090 对 MuJoCo PPO 来说 GPU 过剩，CPU 核心才是真正有价值的部分。

同时纠正并行策略：

- 旧规则“最多 6 进程”不再作为固定限制。
- 新规则应按服务器规格动态估计：
  - Server B：256 核，理论可尝试 30+ 进程，但必须受 EGL context、内存、任务一致性限制；
  - Server C：112 核，理论可尝试 10+ 进程。

但实践中发现：盲目扩到 20 进程 × 4 env 会触发 EGL framebuffer/context 错误。说明并行上限不仅由 CPU 决定，也受渲染/仿真 context 限制。

在扩进程时，为绕过 EGL 错误曾尝试将部分任务改为 `--envs 2`，但随后意识到这会破坏实验可比性。

关键原则：

- 同一噪声、同一算法、不同 seed 之间，`n_envs` 必须一致。
- 同一噪声下不同算法对比，`n_envs` 也应一致。
- 不同论文部分可以使用不同 n_envs，但必须清楚标注。
- 当前主线实验沿用 HumanoidBench / 历史基线的 `n_envs=4`，不能随意混入 `n_envs=2`。

- 用户提出可以系统研究 number of environments 的 sweet point，类似此前已经做过网络容量 512 sweet point、P5/P75 sweet point。
- 论文中已出现 N=4 / N=64 的相关内容，N=256 记录待进一步从服务器日志确认。

## 新洞察

### 1. FTC 鲁棒性实验必须证明“机制链条”，不是只证明“存活”

今天明确了鲁棒性实验的核心不是简单地让机器人在噪声下不倒，而是要证明：

扰动 → FTC 下降 → gate 触发 Protect → 接触恢复 → gate 退出或转入 Push → reward / stability 改善。

如果只有 reward 更高，而 FTC 没有进入 Mode A，则不能作为 [[FTC Metric]] 的机制证据。

---

### 2. Observation noise 不适合作为 TC-PPO 的核心鲁棒性噪声

尽管 observation noise 是常见 sim-to-real 扰动，但它与 FTC 的接触时序主张关联弱。把它放进主协议会让审稿人误以为论文在证明“通用鲁棒性”，而不是“接触时序鲁棒性”。

因此删除 observation noise 是正确的“叙事收束”，不是削弱实验。

---

### 3. Control latency 是最能证明 FTC leading indicator 的扰动

动作延迟会直接破坏控制时序，使接触事件相对动作滞后或错位。若延迟步数增加时，Vanilla 性能单调下降，而 TC-PPO 下降更缓，就能非常直接地支持 “FTC 作为领先预警信号” 的核心主张。

---

### 4. Kitchen 的 reward=0 不是简单失败，而是 PBRS 的实验入口

Kitchen 中 Vanilla / 原 TC-PPO 的 reward 长期为 0，说明：

- 单靠 contact-timing regulation 无法解决 sparse reward / long-horizon composition；
- FTC 高并不代表任务成功；
- 这正好定义了 [[Mode A-B-C Taxonomy]] 中与 task progress 脱钩的盲区；
- PBRS 的意义是为 kitchen-class 任务引入 progress signal，而不是替代 FTC。

---

### 5. 当前训练浪费的根源是“技术栈—硬件经济性不匹配”

不是单纯 GPU 利用率调不起来，而是：

- MuJoCo 物理仿真在 CPU；
- SB3 PPO rollout 主要 CPU-bound；
- MLP policy update 太小，无法吃满 GPU；
- 多 GPU 对当前任务价值有限；
- 大核 CPU + 单张中等 GPU 更适合当前论文阶段。

---

### 6. 并行度不能只看 CPU，还要看 EGL/context 和实验一致性

今天的扩量尝试说明：

- CPU 还有余量不代表可以无限加进程；
- 每个 env 可能消耗 EGL / OpenGL context；
- 大量 SubprocVecEnv 会触发 framebuffer/context 错误；
- 降低 `n_envs` 虽然能缓解 context 压力，但会破坏实验一致性；
- 正确做法是按任务分批、保持同组 `n_envs` 一致，并监控 CPU load、内存、GPU context、日志错误。

---

### 7. n_envs 可能本身就是一个值得系统研究的 sweet point

已有类似经验：

- 网络容量约 512 附近可能是 sweet point；
- P5 / P75 也有 sweet point；
- n_envs 可能同样存在“先升后降”或“收益递减”的训练尺度规律。

这可能成为补充分析点，但必须基于已有 N=4 / N=64 / N=256 记录核实，不能凭印象写进论文。

---

### 风险

1. **扰动强度不足**
   - 若 FTC 不跌破 τ_low=0.35，则 Mode A 不触发，实验机制失败。

2. **扰动过强**
   - 若 Vanilla 和 TC-PPO 都直接崩溃，则无法证明 TC-PPO 的恢复能力。

3. **Kitchen PBRS 仍无法突破 0**
   - 若 PBRS shaping 只产生小额 shaping reward，但不推动真实 task completion，需要重新检查势函数设计。

4. **并行实验污染**
   - 混用 `n_envs`、不同 wrapper 版本、不同 commit/脚本版本，会导致结果不可比。

5. **服务器资源误判**
   - CPU load 低并不等于系统稳定；
   - EGL/context、文件 I/O、SubprocVecEnv 通信都可能成为隐藏瓶颈。

6. **论文叙事超前**
   - 当前 robustness 部分仍需谨慎，未完成实验不能写成 completed validation。
