### 5. TikZ 图、外部 PDF 图与投稿兼容性

围绕“为什么论文里有两种图表形式”做了澄清：

- TikZ 图：
  - 适合流程图、状态机、架构图；
  - 图本身由 LaTeX 代码绘制；
  - 可直接在 `main.tex` 中改节点、箭头、颜色、间距。
- 外部 PDF 图：
  - 适合训练曲线、柱状图、分布图；
  - 通常由 Matplotlib/seaborn 脚本生成；
  - `main.tex` 只通过 `\includegraphics` 插入；
  - 若要改图本体，需要修改对应 Python 脚本并重新生成 PDF。

明确结论：

- 混用 TikZ 与外部 PDF 完全正常；
- CoRL/IEEE/NeurIPS 等投稿系统最终只接收编译后的 PDF，审稿人不关心图的生成方式；
- 没必要强行统一成一种形式。

### 8. Kitchen 任务：是否值得做、如何定位

下午对 Kitchen 任务做了多轮技术与叙事分析。核心背景：

- Kitchen 是 HumanoidBench 中极稀疏、离散完成条件的 manipulation task。
- 任务成功看子任务是否真正完成，而不是“离目标近不近”。
- 论文当前已观察到：
  - FTC 稳定；
  - contact 不明显崩坏；
  - 但 task reward 仍为 0；
  - shaped reward 可能上涨，但不能等同于真实任务完成。

DeepSeek 初步给出过一个关于 PBRS 触发点负奖励的推导，但后来重新计算发现该推导有误：

原错误结论称跨越触发阈值会产生约 `-0.69λ` 的负奖励。重算后：

```text
γ × Φ(s') − Φ(s)
= 0.99 × [(c+1) − 1.0] − [c − 0.3]
= 0.99c − (c − 0.3)
= −0.01c + 0.3
```

当 `c=0` 时，shaped reward 是 `+0.3λ`，不是负的。

但更重要的问题仍成立：

- 不是“跨越被惩罚”；
- 而是“跨越阈值的信号太弱、太平”；
- agent 容易学会接近目标，却缺少足够动力跨过离散完成线；
- sigmoid-like boundary shaping 的作用是放大“刚好没跨过”和“跨过去了”的差异。

形成的真实判断：

- Kitchen 值得做，但只限时 1 周；
- 目标不是解决 manipulation；
- 只作为 [[Mode A-B-C Taxonomy]] 中 Push mode 的验证点；
- 若出现 `task_reward > 0`，再扩展训练；
- 若 1 周仍为 0，则写成 limitation / boundary characterization。

推荐执行节奏：

1. Day 1：修复 reward logging bug，确保 `task_reward` 与 `shaped_reward` 分开记录。
2. Day 1：将纯距离势函数改为边界增强型势函数。
3. 短跑验证：看是否出现 `task_reward > 0`。
4. 若仍为 0：尝试课程学习，例如先训单个子任务。
5. 一周截止，不无限调参。

## 新洞察

### 1. 图表优化不能只看内容，还要单独做“视觉评估”

今天最重要的图表洞察是：内容正确不等于图好。DeepSeek 更擅长判断图是否讲清了方法逻辑；Kimi 在视觉层面补充了很多关键问题：

- 文字和图形是否可能重叠；
- 节点是否贴边；
- 子图比例是否不均；
- 坐标系是否混用；
- 注释是否靠硬编码坐标；
- 图例是否压住标签；
- caption 是否重复正文。

这说明图表审阅至少需要两层：

1. 内容层：是否支撑论文论点；
2. 视觉层：审稿人是否能一眼读懂。

对于 TC-PPO 这种机制型论文，方法图尤其要少字、少公式、少颜色，把 [[FTC Metric]]、scheduler、single-policy PPO 主干之间的关系讲清楚即可。

### 2. Kitchen 的价值是“边界诊断”，不是“额外主贡献”

Kitchen 如果成功，容易诱人地写成 manipulation breakthrough；但这会破坏论文主线。更稳的定位是：

- 主线：[[TC-PPO Algorithm]] 通过 [[FTC Metric]] 稳定 humanoid locomotion 训练；
- Kitchen：当 contact 已稳定但 reward 仍为 0 时，Push mode 是否能提供额外探索推动；
- 成功：说明 Push mode 有潜力；
- 失败：说明纯距离 PBRS 在极稀疏离散 manipulation 上存在边界。

这让 Kitchen 不论成败都能服务论文，而不是拖垮主线。

### 3. PBRS 的问题不是“符号错”，而是“边界信号不够尖”

重新计算后发现，DeepSeek 关于触发点负奖励的数学推导是错的；但它指出的方向有启发：纯距离势函数在 Kitchen 这类离散任务上无法强区分“接近”和“完成”。

更准确的缺点：

- 距离势函数能鼓励靠近；
- 但不能保证跨越离散完成阈值；
- agent 可能停在阈值附近；
- shaped reward 上升可能是假进步；
- 真正需要的是 object-state-aware / boundary-enhanced potential。

这与 [[FTC Metric]] 的作用边界一致：FTC 能判断 contact 稳不稳，但不能替代 manipulation reward 的语义条件。

### 4. GPT 作为 Claude Code 主模型目前不适合端到端论文修改

即使已经配置好代理并提高系统提示强度，GPT 在 Claude Code 中仍常常表现为：

- 只给建议；
- 不主动读写文件；
- 不持续推进；
- 工具调用不可靠。

因此实际工作流应是：

- Claude/Codex：主控、读写文件、跑编译、执行多步任务；
- DeepSeek：代码/技术分析、替代方案；
- Kimi：中文写作、长文本润色、视觉审阅；
- GPT：可做对照咨询，但不应作为论文工程主力。

### 5. Kimi 长任务必须显式调高 `max_tokens`

今天重复验证了一个工程经验：

- Kimi 在长 prompt / LaTeX / 多图评估任务中，默认输出容易空返回或截断；
- `max_tokens=8000+` 后明显稳定；
- LaTeX 写作建议 `12000`；
- 中文 prompt 比英文 prompt 更稳定。

这个经验应固化到后续国产模型调用流程中。
