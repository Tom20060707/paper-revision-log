# 2026-04-04 研究日记：FTC 阈值校准、三模式门控与 Mode A 实验启动

## 核心进展

今天真正推进了 [[TC-PPO Algorithm]] 的核心逻辑闭环：从“τ_high=0.50 是 bug”进一步发展为一套可解释、可复现的 [[FTC Metric]] 阈值校准方法，并把它写进论文与实验规划中。

最重要的进展有四点：

1. **明确了 τ_low 与 τ_high 的物理含义**
   - τ_low 不再被简单解释为“稳定 FTC 的最低值”，而是解释为 **pre-collapse 危险窗口的上界**。
   - τ_high 不再用“均值/中位数”含糊解释，而是定义为 **健康训练时 FTC 能自然到达的典型值**。
   - 这解决了审稿人可能质疑的核心问题：为什么不是取 0.30、0.50 或稳定区最高值。

2. **形成了 [[Mode A-B-C Taxonomy]]**
   - Mode A：门控先打开后关闭，是真正想证明的理想工作模式。
   - Mode B：门控一直打开，说明 τ_high 过高或任务自然 FTC 天花板太低。
   - Mode C：门控一直关闭，说明任务天然稳定，TC-PPO 退化为 Vanilla PPO。
   - 该分类解释了 walk/truck 结果，也指导了下一步实验任务选择。

3. **把固定阈值方案升级为“自适应校准协议”**
   - 当前 τ_low=0.35、τ_high=0.42 被定位为 H1/HumanoidBench 上的一次手动实例化。
   - 更通用的方案是 warm-up 阶段先测量 FTC 分布，再用百分位数自动设阈值：
     - τ_low = FTC warm-up 分布的低百分位，例如 5th percentile
     - τ_high = FTC warm-up 分布的典型值，例如 50th percentile
   - 这使得 TC-PPO 从“手动调参”转向“任务自适应校准”。

4. **启动了 bookshelf_hard 上的 Mode A 验证实验**
   - Server C 上已启动 6 个 4-envs 实验：
     - TC-PPO × seeds 0/1/2，τ_low=0.35，τ_high=0.42
     - Vanilla PPO × seeds 0/1/2
   - bookshelf_hard 的稳定 FTC 约 0.514，高于 τ_high=0.42，因此理论上比 walk/truck 更可能出现 Mode A。

## 对话精炼

### 1. 关于“10M 步是否合适”的判断

今天确认了一个重要事实：目前还没有真正观察到完整的“触发 → 保护 → 恢复 → 退出”循环。

已有实验对应关系大致如下：

| 实验 | 现象 | 是否完整循环 |
|---|---|---|
| walk, τ=0.35 | FTC 约 0.41，从不触发 | 否，Mode C |
| truck, τ=0.35/τ_high 旧设定 | 会触发但关不掉 | 否，Mode B |
| walk, τ=0.8, 100M | 全程触发、全程过约束 | 否，Mode B |

结论：
- 若只看最终 reward，5M 可能够。
- 若要看完整崩溃—保护—恢复曲线，10M 是合理选择。
- 若要看长期稳定性或 100M 级退化，需要更长训练。

因此，10M 被认为适合下一步 Mode A 探索，但任务必须换成更可能触发并恢复的 bookshelf_hard、powerlift 或 insert-normal，而不是 walk/truck。

### 2. τ 与 FTC 的关系：从“互为因果”修正为“迭代校准”

今天讨论中澄清了一个误区：FTC 不会因为随机种子求均值而自然趋向 0.5。FTC 不是均匀随机变量，而是由机器人接触动力学、任务结构和策略状态共同决定。

更准确的关系是：

```text
第一轮：不知道 FTC 分布 → 设一个初始 τ → 跑实验 → 观察 FTC 分布
第二轮：根据实测 FTC 分布 → 重新校准 τ_low / τ_high
```

这不是“FTC 与 τ 在同一实验内互相决定”，而是跨实验迭代的校准过程。

### 3. τ=0.8 的真实解释

今天决定不再把 τ=0.8 包装成“为了捕捉完整 FTC 范围的探索性设计”，而是更诚实地解释为：

- FTC ∈ [0,1]，初始直觉上误以为“0.8 以下就是不好”。
- 没有充分意识到 FTC = 1 - CV² 是平方型指标。
- τ=0.8 对应 CV 约 44.7%，这个阈值过于严苛。
- 这揭示了一个普遍陷阱：**用线性直觉设计非线性指标阈值**。

这个解释反而增强了论文的方法论价值：阈值不能凭直觉，必须由实测分布校准。

### 4. 两个门控阈值的完整逻辑

今天把 τ_low 和 τ_high 的作用拆开：

- **τ_low：什么时候出问题？**
  - 触发条件。
  - 应设在 pre-collapse 区间上界。
  - 如果不稳定 FTC 区间是 0.30–0.35，那么 τ_low=0.35 表示在 FTC 刚进入危险窗口时立刻干预。
  - 为什么不用 0.30？因为 0.30 是危险窗口下界，等到这里再触发太晚，策略可能已经接近崩溃。

- **τ_high：什么时候恢复正常？**
  - 退出条件。
  - 应设为健康训练时 FTC 能自然到达的典型值。
  - 如果 τ_high 太高，例如 0.50 或 0.52，很多任务永远达不到，门控会一直打开，形成 Mode B。
  - 如果 τ_high 太低，例如 0.36，可能刚有一点恢复就过早关闭保护。

因此今天形成的关键表述是：

```text
开门标准：进入危险区。
关门标准：恢复到正常区。
```

### 5. Mode A/B/C 分类成为论文核心解释框架

今天将两阈值门控的运行模式归纳为三类：

| 模式 | 门控行为 | 含义 | 对应实验 |
|---|---|---|---|
| Mode A | 先打开后关闭 | 理想模式，证明 TC-PPO 真正有效 | 尚需实验证明 |
| Mode B | 一直打开 | τ_high 过高或任务自然 FTC 天花板低，导致过约束 | truck、τ=0.8 实验、部分 sit_simple |
| Mode C | 一直关闭 | 任务天然稳定，TC-PPO 自动退化为 Vanilla PPO | walk |

这个分类今天被写进论文 Section 3.2，成为解释 walk/truck 结果与后续实验选择的重要框架。

### 6. 关于并行环境数量的判断

今天讨论了 n_envs 对结果的影响：

- 并行环境越多，batch 更大，梯度方差更小，Vanilla PPO 可能更稳定。
- 如果大并行下 Vanilla 也不崩，TC-PPO 会趋向 Mode C。
- 如果大并行下 Vanilla 仍崩，而 TC-PPO 能恢复，则会大幅增强论文主张。
- 因此，并行数消融本身是有价值的。

但实际资源限制很关键：

- Server C A800 显存 80GB，当前 6 个 4-envs 实验已占约 28.8GB。
- 每个 MuJoCo EGL env 子进程约占 1,047MiB GPU 显存。
- 在当前实验未结束时，额外启动 64 envs 会超过剩余显存。
- 当前更实际的方案是：
  - 等当前 6 个实验完成后再跑 32/64 envs。
  - 或现在只启动单个 32-envs 探路实验。

## 新洞察

### 1. pre-collapse 窗口是 TC-PPO 的真正理论支点

今天最关键的洞察是：TC-PPO 不是在“完全崩溃后救火”，而是在 pre-collapse 危险窗口中提前干预。

```text
稳定训练：FTC 高于 0.354 左右
危险窗口：FTC 下滑到 0.30–0.35
完全崩溃：FTC 继续跌破 0.30
```

τ_low=0.35 的意义不是“最低稳定值”，而是：

```text
当 FTC 从上向下穿过 0.35 时，说明系统刚进入危险窗口；
此时干预最早、最有恢复余量。
```

这比“取稳定区最低值”更有说服力。

### 2. τ_high 的本质是“可达性”而不是“越高越安全”

τ_high 不是奖励更高稳定性的目标值，而是退出保护的恢复判据。它必须是任务自然能达到的健康水平。

如果设得太高：
- 门控永远关不上。
- 策略长期处于保守更新。
- TC-PPO 变成 Mode B。
- truck 中 τ_high=0.50 的失败就是典型例子。

因此，τ_high 的核心约束是：

```text
τ_high 必须低于或等于该任务健康训练时可自然达到的 FTC 典型值。
```

### 3. 固定阈值只能是实例化，自适应阈值才是通用算法

今天意识到：τ_low=0.35 / τ_high=0.42 不是普适常数，而是 H1/HumanoidBench 上的经验实例化。

真正有通用意义的是校准方法：

```text
warm-up → 测量 FTC 分布 → 用百分位数自动生成 τ_low / τ_high
```

这个思想让 [[TC-PPO Algorithm]] 从“一个带阈值的 PPO 变体”提升为“带自校准稳定性门控的 PPO 框架”。

### 4. walk/truck 不是失败，而是揭示了适用模式

walk 和 truck 不再只是“TC-PPO 没赢”的尴尬结果，而是对应了两种退化模式：

- walk：FTC 天然高，门控不触发，Mode C。
- truck：FTC 天花板低，触发后关不上，Mode B。

它们帮助证明：
- TC-PPO 不应在所有任务上都强行生效。
- 门控行为取决于任务 FTC 分布。
- 任务自适应阈值是必要的。

### 5. 并行环境数量可能成为额外贡献点，但要谨慎

大并行实验不一定直接证明 TC-PPO 更强，但可以揭示：

```text
collapse frequency / gate activation rate 随 n_envs 增加而变化
```

如果 4 → 32/64 envs 后 Vanilla 崩溃减少，TC-PPO 触发减少，这也能支持论文对“资源受限 on-policy regime”的定位。

但硬件现实是：MuJoCo EGL 子进程显存开销很大，不能只按神经网络显存估计。
