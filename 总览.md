# 总览

## 这篇论文现在想解决什么

人形机器人很难学，不只是因为动作维度高，还因为它会不断和地面、物体、环境发生接触。

一旦接触节奏乱掉，PPO 训练很容易进入坏状态：reward 还没明显掉下去，身体姿态和接触模式可能已经开始退化。

这篇论文现在最想讲清楚的是：

> 能不能用接触一致性提前发现这种退化，并用 PPO 稳定机制把训练拉回来？

## 从旧思路到新思路

### 早期：TC-PPO

最开始的想法是做一个 PPO 变体，用接触时间一致性调节训练。

这个阶段的重点是：

- PPO 会在接触密集任务中不稳定；
- 接触指标可能比 reward 更早暴露问题；
- 门控机制可以在某些任务中保护训练。

问题是，如果只写“TC-PPO 提升性能”，负结果会很难解释。

### 中期：FTC / CTS

后来开始强调接触时序稳定性。

FTC/CTS 的直觉是：

> 好的支撑接触通常有一定规律；如果接触节奏突然变乱，策略可能正在走向崩溃。

这对走路、上楼、支撑、恢复平衡这类任务比较自然。

但它也有硬伤：很多任务不是周期性的。比如 kitchen、insert、push、truck，接触本来就是阶段化、事件驱动的，不能要求它们像走路一样有稳定步频。

### 当前：CSC + 接触域稳定性

所以现在更合理的总框架是 CSC：

> Contact-Surface Consistency，接触面一致性。

它不强迫任务周期化，而是看当前接触事件分布是否偏离稳定参考模式。

这样旧的 CTS/FTC 可以保留，但只作为支撑接触或准周期接触的特例。

## 当前核心关系

```text
接触域退化
  ↓ 被观测
接触面一致性下降
  ↓ 触发/校准
PPO 稳定机制
  ↓ 改善
接触域稳定性
```

## 最容易误解的点

### 1. 一致性不是最终目标

一致性是诊断信号。它回答：

```text
接触行为是不是开始变乱？
```

稳定性才是目标。它回答：

```text
训练和策略是否还可靠？
```

### 2. Mode A/B/C 不是任务分类

它们不是说某个任务好或坏，而是说门控和稳定域处在什么状态。

- Mode A：退化能被发现，也能被拉回来。
- Mode B：门控或接触结构不匹配，保护可能锁死。
- Mode C：接触看起来稳定，但任务不一定成功。

### 3. Kitchen 不是“已经解决”

Kitchen 的价值是暴露边界：接触稳定不等于任务成功。稀疏奖励、阶段化操作、物体状态变化，都需要更细的任务进度或接触面建模。

## 现在怎么写最稳

可以说：

- 接触一致性可作为崩溃前诊断信号。
- CTS/FTC 在支撑接触上是可用实例。
- CSC 是更一般的框架。
- 非周期任务暴露了旧指标边界。
- PPO 稳定机制用于抑制接触域退化。

不要说：

- TC-PPO 普遍提升所有任务。
- `1 - CV²` 是通用接触指标。
- CTS/FTC 已经解释所有 manipulation 任务。
- PBRS 已经解决 kitchen。
- 触觉、软手、物体-环境接触都已实验验证。

## 下一步

最重要的不是继续堆缩写，而是把论文写清楚：

1. 标题和摘要统一到“接触域一致性与稳定性”。
2. Method 中分清 CSC 总框架、CTS/FTC 特例、PPO 稳定机制。
3. Experiments 中把 Mode A/B/C 写成稳定域状态。
4. Discussion 中诚实说明非周期和操作任务的边界。
