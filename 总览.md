# 总览

这份总览想做一件很简单的事：不先讲一堆缩写，先讲清楚这篇论文到底在关心什么。

如果你刚打开这个仓库，看到 TC-PPO、FTC、CTS、CSC 这些名字有点晕，很正常。它们不是一开始就排好队出现的，而是在反复写、反复被实验结果提醒、反复改口径的过程中慢慢分出来的。

## 这篇论文在看什么

人形机器人学 PPO 的时候，难点不只是“动作维度很高”。更麻烦的是，它的身体一直在接触世界。

比如脚要踩稳，身体要保持平衡，手可能还要推、拉、插入、搬东西。接触本身一乱，训练就容易往坏方向走。更微妙的是，reward 不一定第一时间告诉你“出事了”。有时候 reward 还没明显掉，接触模式已经开始变得不稳定。

这篇论文现在想抓住的，就是这个提前出现的信号。

可以把主问题理解成：

> 我们能不能在 reward 崩掉之前，先从接触模式里看见训练正在变坏？  
> 看见以后，能不能让 PPO 更新得更稳一点，而不是继续把策略推向坏接触状态？

## 想法是怎么变过来的

### 1. 先是 TC-PPO

最早的想法挺直接：既然 PPO 在人形接触任务里容易不稳，那就用接触时间的一致性来调节 PPO。

这个想法不是错的。它抓住了一个很有价值的现象：接触模式往往比 reward 更早暴露问题。

但如果论文只写成“我提出 TC-PPO，然后它提升性能”，就会有点单薄。因为真实实验不会永远配合漂亮故事，有些任务会提升，有些任务会失败，还有些任务会让人发现原来的解释不够用。

### 2. 然后转向 FTC / CTS

后来我们开始更认真地看接触时序：脚什么时候接触、接触多久、节奏是不是突然变乱。

这对上楼、走路、支撑、平衡恢复这类任务挺自然。稳定的策略不一定每一步完全一样，但接触节奏通常不会乱成一锅粥。

问题也在这里出现了：不是所有任务都有“节奏”。

kitchen、insert、push、truck 这类任务更像阶段化操作。它们的接触是事件驱动的，不是像走路那样一下一下周期重复。所以，把一个周期接触公式硬套到所有任务上，肯定会被挑出来。

### 3. 现在更接近 CSC

所以现在更舒服的说法是 CSC：Contact-Surface Consistency，接触面一致性。

它不再问：

```text
接触间隔是不是很均匀？
```

而是问：

```text
这个任务里真正重要的接触面，是否还保持着稳定、可解释、可比较的模式？
```

这样一来，早期的 CTS/FTC 不是被推翻，而是被放回合适的位置：它们适合支撑接触或准周期接触，是 CSC 下面的一类具体情况。

## 现在最该抓住的关系

可以先记住这个顺序：

```text
接触模式开始变差
  ↓
接触一致性先发出信号
  ↓
PPO 更新变得更保守或更有选择
  ↓
训练尽量回到稳定接触状态
```

这里有个很重要但容易被忽略的区别：

- 接触一致性是“提醒你哪里不对”的信号。
- 接触域稳定性才是“希望训练最后达到”的状态。

打个比方，温度计不是健康本身，但它能提醒你是不是发烧了。接触一致性在这里有点像这个温度计。

## 几个容易读歪的地方

### 接触一致性不是越高越神

论文不是想说“接触越一致，任务就一定越成功”。这太粗暴了。

更准确的是：当接触模式突然变得不稳定时，它可能是在提醒我们训练要出问题了。

### Mode A/B/C 不是给任务打分

Mode A/B/C 更像是在描述训练状态：

- Mode A：接触退化被发现了，保护机制也把训练拉回来了。
- Mode B：看见问题了，但没拉回来，可能阈值、接触面定义或任务结构不匹配。
- Mode C：接触看起来没崩，但任务也不一定真的完成。

所以 Mode C 不是“好消息”。它有时候反而提醒我们：只看接触还不够，还要看任务进度和物体状态。

### Kitchen 不是已经解决

Kitchen 的价值在于暴露边界。它告诉我们：接触稳定不等于任务成功。

这类任务需要更细的任务阶段、物体状态、进度奖励或接触面建模。这个边界说清楚，论文反而更可信。

## 现在写论文时的口径

比较稳的写法是：

- 接触一致性可以作为 PPO 崩溃前的诊断信号。
- CTS/FTC 是支撑接触或准周期接触里的有效实例。
- CSC 是更宽的接触面一致性框架。
- 非周期操作任务暴露了旧指标的边界。
- PPO 稳定机制利用这些信号，让训练不要轻易滑进坏接触状态。

需要收住的写法是：

- 不把 TC-PPO 写成所有任务都赢的万能算法。
- 不把 `1 - CV²` 写成所有接触任务通用公式。
- 不把 kitchen 写成已经被 PBRS 解决。
- 不把还没验证的手-物体、物体-环境、触觉接触写成已有结果。

## 下一步该往哪里走

接下来最值得做的不是继续造新缩写，而是把现有故事写得更干净：

1. 摘要里先讲接触退化，而不是先推销算法名。
2. Method 里分清 CSC、CTS/FTC 和 PPO 稳定机制各自负责什么。
3. Experiments 里把 Mode A/B/C 写成训练状态和方法边界。
4. Discussion 里诚实讲非周期操作任务为什么更难。

这样写出来的论文会更像一个研究问题自然展开，而不是一个算法名字硬撑全场。
