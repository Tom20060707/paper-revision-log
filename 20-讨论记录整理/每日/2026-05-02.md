## 新洞察

### 1. 图表质量不能只靠编译成功判断

今天最大的教训是：LaTeX 编译成功不等于图合格。

必须增加视觉验收层：

```text
脚本生成 → LaTeX 编译 → 渲染最终论文页 → 肉眼检查缩放后的可读性
```

尤其 IEEE/CoRL 类双栏论文中，单图预览看起来可读，不代表嵌入论文后仍可读。

### 2. 概念曲线图的关键不是“线存在”，而是“连续曲率表达机制”

Fig. 6 / Fig. 7 的问题说明：机制图中的曲线如果像“直线段拼接”，读者会感觉粗糙、不自然。

更好的设计原则是：

- 先定义完整趋势函数；
- 再按阶段分色；
- 不要分段画完再硬接；
- 不要在小图里塞未解释分支线；
- 曲线要体现连续的训练动态，而不是几何拼图。

这对 [[Mode A-B-C Taxonomy]] 和 reward staging 图尤其重要，因为这些图不是实验数据图，而是论文叙事中的“机制直觉图”。

### 3. [[FTC Metric]] 阈值的核心争议不是 P5/P75 概念，而是估计稳定性

今天厘清了三层问题：

1. `P5/P75` 的统计定义：  
   它们是分位边界，不是尾部均值。

2. `P5/P75` 的机制语义：  
   - `P5`：低尾异常触发；
   - `P75`：恢复到任务可达健康区。

3. 用户真正关心的问题：  
   单次 warm-up 的分位点是否偶然性太大。

初步建议是 multi-seed percentile calibration：

```text
每个 seed 单独算 P5/P75
再对 seed-wise 阈值取 median/mean
报告 std 或 CI
```

但用户指出这会显著增加计算量，不符合最简原则。这个洞察很重要：阈值稳定性方案必须在“统计稳健”和“最简计算预算”之间折中。

### 4. 自动化日志应整理“完整自然日”，不是“当前日期”

日志机制问题的根因不是某次漏扫，而是任务语义错了。

正确机制：

```text
00:30 整理昨天完整自然日
```

错误机制：

```text
23:55 整理 current local date
```

这个经验也适用于实验日志、训练日志和论文修改日志：凡是按天归档的内容，最终版都应该在自然日结束后生成。

## 明日待办

### A. 阈值实验方案收敛

继续回答用户最后提出的问题：

> 多 seed calibration 会显著增加计算量，也不符合最简原则，那怎么改？

需要设计一个更轻量的方案，可能方向：

1. **单 seed + 多窗口分位数稳定性检查**
   - 不增加训练 seed；
   - 在同一个 `1M` warm-up CTS history 内做 block bootstrap 或 sliding-window percentile；
   - 报告 `P5/P75` 的窗口方差。

2. **使用足够密集 CTS 采样 + 插值分位数**
   - 强调不是单 checkpoint；
   - 从完整 history 估计分位点；
   - 用 `np.percentile` 插值降低单样本离散性。

3. **winsorized / smoothed percentile**
   - 仍保留 P5/P75 边界语义；
   - 对 CTS history 做轻度 smoothing 或去掉极端异常值；
   - 避免变成尾部均值。

4. **最小额外计算的 sanity check**
   - 不重跑 full training；
   - 只在 warm-up log 上做 bootstrap；
   - 报告 threshold CI / sensitivity，而不是新增大量 PPO runs。

明天要把这套变成论文中可写的 protocol，而不是空泛解释。
