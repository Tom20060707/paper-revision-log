## 今日主线

今天的主线是把 [[TC-PPO Algorithm]] 从“机制提出 + 部分诊断”推进到“核心闭环行为已有实证”的状态。

最重要的进展有四个：

1. **论文中 [[FTC Metric]] 自适应阈值逻辑基本理顺**  
   明确区分了“跨任务自适应”和“任务内静态”，并把 `P5/P45` 重新校准为 `P5/P75` 的过程写入论文。论文现在能解释为什么 `τ_low=P5` 锚定异常触发语义、`τ_high=P75` 锚定可达恢复语义，而窗口宽度 `Δτ` 是二者共同约束的结果，不是任意调参。

2. **[[Mode A-B-C Taxonomy]] 的证据链显著增强**  
   Stair 上 soft gating + adaptive τ 已经稳定观测到 Mode A：3/3 seeds 出现 gate ON → recovery → gate OFF，reward 约 `39.4 / 38.9 / 40.5`，均值约 `39.6`，接近 vanilla stair seed0 的 `40.3`。这使论文从“Mode A 理论上期望但未观察到”转为“Mode A 已在 Stair 实证”。

3. **软/硬门控实现与论文公式对齐**  
   发现 Section 3.2 里 soft gating 公式仍写成单变量 sigmoid：`σ(-k(FTC-τ_low))`，但代码实际是三维特征：
   `z=[FTC, H_t, \bar T_t]`，固定权重 `w=[-2.0,-0.5,1.0]`，在线 Welford 归一化后做 sigmoid。今天已把正文与 Appendix 统一，并修正了 Welford 引用和 label 问题。

4. **服务器实验状态与数据使用边界更清楚**  
   Server C 已完成并关机，数据保存在服务器本地临时存储中；Server B 仍有 vanilla stair / vanilla insert_small 等实验接近完成。今天也确认了部分中间值不能直接写成最终结果，例如 vanilla insert_small 曾用 `26.0` 是 8.7M/10M 的 preliminary 值，后续需要等最终 10M 结果再更新论文。

## 关键对话与决策

### 1. P45 → P75 的论文逻辑重构

最初论文和代码存在不一致：部分论文位置仍写 `P45`，代码已转向 `P75`。今天做了系统梳理：

- 实验历史中固定阈值曾使用 `(0.35, 0.42)`。
- 自适应 `P5/P75` 方法在 H1 上典型输出约 `(0.35, 0.41)`。
- 因此论文应区分：
  - **实际历史实验固定值**：`τ_high=0.42`，不能误改。
  - **自适应方法典型输出**：`τ_high≈0.41`，与固定值相差约 `0.01`，可解释为一致性验证。

决策：  
不把所有 `0.42` 机械替换为 `0.41`。只有描述 adaptive percentile 输出时使用 `0.41`；描述历史实验条件时保留 `0.42`。

### 2. 三种在线 τ 方案的引入方式

用户质疑 Scheme A/B/C “凭空冒出来”。今天将其改为从 2×2 设计空间自然推出：

| 设计维度 | 不追踪 FTC 上升 | 追踪 FTC 上升 |
|---|---|---|
| 无 warmup 下界 | Scheme A | Scheme B |
| 有 warmup 约束 | 当前静态设计 | Scheme C |

这让三种动态方案和当前静态设计覆盖完整设计空间，不再像随意枚举。

相关概念：[[Mode A-B-C Taxonomy]]、[[TC-PPO Algorithm]]

### 3. “位置 vs 宽度”审稿风险

用户提出一个重要审稿问题：Mode A 触发到底是因为窗口宽度变大，还是因为阈值绝对位置合适？是否需要固定 `Δτ` 平移窗口做消融？

讨论后决策：**暂不做该消融**。原因是 P5/P75 设计本身不是单纯调宽度，而是同时定义语义位置：

- `P5`：异常低尾，决定 gate 何时打开。
- `P75`：正常可达恢复区间，决定 gate 何时关闭。
- `Δτ≈0.07` 是结果，不是独立自由参数。

论文中已补充解释，说明固定 `Δτ` 平移窗口等价于破坏 P5/P75 的语义锚点，不能回答核心设计问题。

### 4. Truck 窄窗口数据的定位

- 但窄窗口数据仍有价值：可作为 “窗口过窄 → gate chattering / Mode B / 性能下降” 的反面证据。

## 新洞察

### 1. Mode A 的关键不是“阈值越宽越好”，而是“触发点和恢复点都有语义”

P5/P75 的意义在于：

- P5 将 gate activation 绑定到低尾异常；
- P75 将 gate deactivation 绑定到任务内可达的正常恢复区间；
- 如果 `τ_low` 太高，会正常波动也触发 gate；
- 如果 `τ_high` 太高，会 gate 永远关不上；
- 如果二者太近，会 chattering；
- 因此窗口宽度只是统计结果，不能单独作为调参旋钮。

这为 [[FTC Metric]] 的阈值选择提供了比“经验上 P75 好”更强的解释。

### 2. Truck 的失败数据不是废数据，而是 Mode B 与窗口校准的反证

Truck 的窄窗口实验说明：

- 窄窗口无法解决 Mode B，甚至会恶化；
- P40/P45 的具体差异不重要，应合并为 “narrow-window failure”；
- 不应把 P40 包装成设计，而应诚实说明是旧 warmup fallback/旧格式导致的意外窄窗口；
- 这类失败数据可以增强论文可信度，因为它证明 TC-PPO 不是所有任务都 magically work，而是受 FTC dynamics 约束。

### 3. Soft gating 的价值比 hard gating 更明确

Stair 与 insert_small 都显示 hard gating 风险较大：

- Stair soft：稳定接近 vanilla，并实现 Mode A。
- Stair hard：目前完整/partial 数据显示方差更大、reward 较低。
- Insert-small hard：约 `13.5`，明显低于 soft `23.1`。

Soft gating 不只是工程 trick，而是让约束从离散开关变成连续调节，减少对 PPO 更新的破坏。

### 4. 论文必须严格区分“真实数据”“preliminary”“示意图”

今天发现若图表更新太快，很容易把中间值写成最终值，例如 vanilla insert_small `26.0`。后续所有图表都应：

- 从最终 `summary.json` 或明确日志读取；
- 对 partial 数据标注 preliminary；
- 对手工插值/示意曲线在 caption 中说明；
- 不把正在跑的 8.7M/10M 或 9.0M/10M 数字写成 final。

### 5. GPU0 显存高不是训练负载不均，而是 MuJoCo EGL 渲染行为

Server B 上 GPU0 显存高的原因是 MuJoCo 并行环境 worker 的 EGL/OpenGL 渲染进程会绑定物理 GPU0，不受 `CUDA_VISIBLE_DEVICES` 控制。GPU1/2 主要承担 PyTorch 计算，显存占用低是正常的。这解释了此前“为什么压力都在 GPU0”的疑惑。
