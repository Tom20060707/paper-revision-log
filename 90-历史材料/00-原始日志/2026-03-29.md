## 新洞察

### 1. 实验规划必须从“论文缺口”而非“想跑什么”出发

今天的对话强调：下一步实验不是孤立决定的，而应从主论文与技术报告的当前状态倒推。应先确认论文中已经声称了什么、图表展示了什么、缺哪些关键证据，再决定实验优先级。

这对 [[TC-PPO Algorithm]] 项目尤其重要，因为算法论文通常需要同时满足：

- 主结果是否超过 baseline；
- 是否有消融支持关键模块；
- 是否有稳定性或泛化分析；
- metric 是否与论文主张一致；
- 实验是否覆盖不同 task/mode。

### 2. 服务器 A / B 数据可能是规划的关键约束

用户特别要求比较服务器 A 和服务器 B 的数据集，说明这两个目录很可能分别保存了不同阶段或不同机器上的实验结果。下一步规划不能直接假设“实验未完成”，必须先盘点：

- 哪些实验已经跑完；
- 哪些只有部分 seed；
- 哪些只有日志没有 checkpoint；
- 哪些结果可直接入论文；
- 哪些需要复跑或补 seed；
- A/B 数据是否存在命名不一致、配置不一致或重复实验。

### 3. 工具链稳定性本身成为实验风险

今天真正暴露出的风险不是算法问题，而是自动化读取与远程协作链路不稳定。连续 API 400 表明当前 Claude Code 调用层可能存在消息格式或工具接口问题。若不先解决，后续很难可靠地让助手读取论文、整理数据、生成脚本或远程执行实验。

因此明天第一优先级应不是立即跑实验，而是恢复“可审查、可执行、可复现”的工作流。
