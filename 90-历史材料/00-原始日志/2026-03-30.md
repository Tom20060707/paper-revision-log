#### 阶段 1：0.1B 长时间稳定性实验

| 项目 | 设定 |
|---|---|

## 新洞察

### 1. 引用错误不一定来自当前源码

今天一个重要发现是：PDF 中出现的参考文献信息错误，不一定意味着当前 `.tex` 源码仍然错误。实际检查后发现 HumanoidBench、WoCoCo、FastTD3 在源码里已正确，只有 PPO-BR 作者需要修。

这提示后续论文版本管理要区分：

- 源码版本；
- 编译后的 PDF；
- 上传版本；
- 本地旧渲染版本。

否则很容易重复修正已经正确的 BibTeX 条目。

### 2. 0.1B 实验的科研价值在于“机制证据”，不是只看最终分数

100M steps 长训练的关键价值，是让 [[FTC Metric]] 有足够长的时间展现动态作用：

- 何时检测到不稳定；
- 是否触发保守更新；
- 触发后 KL / entropy / reward 是否恢复；
- TC-PPO 是否相较 vanilla PPO 更少发生崩溃或能更快恢复。

因此这组实验应优先设计可解释曲线，而不仅是最终 average return。

### 3. 与现有结果对齐比盲目扩大并行更重要

当前阶段决定 0.1B 长训练先使用 `n_envs=4`。这虽然可能不是最高吞吐设置，但好处是：

- 与已有实验可比；
- 可以直接判断“训练更久”是否改善或暴露稳定性问题；
- 避免同时改变训练步数、硬件、并行环境数造成归因混乱。

大规模并行可以作为后续独立实验，不应混入第一轮 0.1B 稳定性验证。

### 4. A800 环境准备是当前阻塞点

实验规划已经较清晰，但真正开跑前仍卡在工程环境：

- 是否已安装正确 CUDA / PyTorch；
- MuJoCo / HumanoidBench 是否可运行；
- 从 Server A 复制到 Server C 的代码与数据是否完整；
- SSH 是否稳定可用；
- 日志目录、checkpoint 保存路径、显存与磁盘空间是否足够。

明天的实际优先级应是先把 Server C 变成可复现实验环境。
