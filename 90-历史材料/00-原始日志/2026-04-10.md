### 2. Section 7 新增两项机制：自适应阈值与软门控

今天围绕 Section 7 写入了四类内容：

1. **总起段落**  
   说明 TC-PPO 的两项核心改进：
   - 自适应阈值：参数层改进，解决任务间 FTC 分布差异。
   - 软门控：机制层改进，解决任务内非周期接触造成的硬门控误激活。

2. **自适应阈值校准**  
   逻辑链确定为：
   - 固定 `τ_low/τ_high` 在不同任务间不可泛化。
   - `truck` 是典型 Mode B 问题：自然 FTC 分布低于固定阈值，gate 容易永久激活。
   - 线性 `μ ± kσ` 方法失败，因为 [[FTC Metric]] 定义为 `1 - CV²`，非线性、有界且分布可能偏斜。
   - 最终采用百分位数法：
     - warm-up 使用 Vanilla PPO，避免循环依赖；
     - 收集 FTC 集合 `F`；
     - `τ_low = P5(F)`；
     - `τ_high = P45(F)`。
   - `P5/P45` 被确定为全文一致口径，清除了 `P50` 和 `(5, 50)` 残留。

3. **软门控机制**  
   逻辑链确定为：
   - 原始 hard gating 是二进制门控，容易在 `package`、`push` 等非周期任务中误激活。
   - 软门控使用连续门控信号 `g_t`。
   - 特征向量包括：
     - `FTC_t`
     - 接触熵 `H_hat_t`
     - 接触间隔特征 `T_hat_t` / `T_bar_t`（具体符号以论文最终版本为准）
   - 正则化系数随 `g_t` 连续调节：
     - `λ_t = λ_max · g_t`
     - `β_t = β_0(1 + g_t)e^{-t/4M}`
   - 为避免符号冲突，sigmoid 不再写作 `σ`，改为 `\mathrm{sigm}`，因为论文中 `σ` 已用于标准差。

4. **总结与局限性**  
   明确写入：
   - 自适应阈值主要缓解任务间分布差异导致的 Mode B。
   - 软门控主要缓解任务内非周期接触导致的误激活。
   - 两者都没有改变 FTC 的周期性本质。
   - 极端非周期任务如 `kitchen` 仍可能效果有限。
   - 后续方向包括 [[GCTC]] 扩展、更多实验、真实机器人部署。

### 4. Introduction 研究叙事重排

用户强调，`τ=0.8` 的来源和失败逻辑必须讲清楚。今天重排后的叙事顺序为：

1. 研究初衷是解决人形机器人训练中的崩溃问题。
2. 观察到崩溃可能与接触时序不规律、变异系数有关。
3. 因为人形机器人与人类步态有相似性，所以参考生物力学期刊中的人类 CV。
4. 在 [[TC-PPO Algorithm]] 框架已经存在的前提下，将 `τ` 作为算法中的待定参数，用生物力学启发给出初始 `τ=0.8`。
5. 后续发现 `τ=0.8` 失败：
   - 线性直觉误导；
   - `FTC = 1 - CV²` 中存在平方非线性；
   - `τ=0.8` 对应 `CV = sqrt(0.2) ≈ 44.7%`，远高于正常人类步态 CV。
6. 再查阅机器人与 PPO 相关研究，发现机器人步态 FTC 分布与人类差异很大。
7. 重新校准到 `τ_low=0.35`、`τ_high=0.42`。
8. 分析 24 个任务后提出 [[Mode A-B-C Taxonomy]]：
   - Mode A：目标模式，短暂激活后恢复；
   - Mode B：gate 锁死；
   - Mode C：gate 从不触发。
9. 由 Mode B 与非周期任务误激活进一步引出：
   - 自适应阈值；
   - 软门控。

这个顺序修复了此前“好像先查文献才提出 TC-PPO”的叙事错误。

## 新洞察

### 1. TC-PPO 的核心问题不只是阈值大小，而是“任务分布—门控机制”的匹配问题

今天的讨论使 [[Mode A-B-C Taxonomy]] 更清楚：

- 固定阈值如果设得过高，会导致任务天然落入 Mode B。
- 固定阈值如果设得过低，可能导致 gate 无法及时介入。
- 即使阈值自适应，hard gating 在非周期任务上仍可能误判任务本身的不规则接触为“不稳定”。

因此 TC-PPO 的泛化问题分成两层：

1. **任务间分布差异**：由自适应阈值解决。
2. **任务内非周期结构**：由软门控缓解。

这一区分非常关键，能防止论文把所有问题都归结为 `τ` 调参。

---

### 2. `τ=0.8` 的失败故事可以成为论文可信度的一部分

原本 `τ=0.8` 只是一个失败参数。今天将它重写为一个有解释力的研究过程：

- 从人形机器人崩溃问题出发；
- 借鉴生物力学 CV；
- 在线性直觉下设定较高阈值；
- 通过 `FTC = 1 - CV²` 发现非线性误差；
- 再回到机器人任务分布重新校准。

这个故事比单纯说“我们调参发现 0.35/0.42 更好”更可信，也更适合 IEEE 期刊论文。

---

### 3. “当前贡献但验证进行中”是比 future work 更准确的表述

自适应阈值与软门控已经形成了明确算法方案，也可在当前代码/论文框架中实现，因此不适合放在 future work。

但由于完整跨任务实验尚未完成，也不能声称实验部分已经验证。

最终采用：

- 方法是当前贡献；
- full cross-task validation ongoing。

这比“future work”或“已完成验证”都更诚实。

---

### 4. 图表 caption 的措辞必须避免把“风险”写成“事实”

今天多次发现 Mode 分类错误并非来自公式，而来自 caption 和 summary 中的过度简化。例如：

- `sit_simple` 不是稳定 Mode C，而是 Mode B risk。
- `truck` 是 confirmed observed Mode B。
- `powerlift` 不能因为属于 Mode C 就被写成 `FTC > 0.42` 组。

这提醒后续写图表时必须区分：

- observed mode
- distributional risk
- theoretical classification
- transient behavior

---

### 5. `ceiling` 与 typical range 不能混用

`truck` 的典型稳定范围约 `0.36–0.37`，但 FTC ceiling / upper whisker 应写 `≈0.39`。

今天多处错误正是因为把这两个概念混用了。

后续论文中应保持：

- typical stable range：可写 `0.36–0.37`
- ceiling / upper whisker：写 `≈0.39`

   安装了较多 Claude Code 插件，可能增加自动触发复杂度。后续若响应变慢或误触发，需要精简。
