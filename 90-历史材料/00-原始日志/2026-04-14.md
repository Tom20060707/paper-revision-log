# 2026-04-14 研究日记

## 今日主线

今天围绕 [[TC-PPO Algorithm]] 的论文叙事、实验优先级和网络容量关系做了大幅推进。核心问题从“TC-PPO 在 [[512×4]] 大网络上收益有限”转向一个更清晰的科学问题：

> [[Network Capacity]] 决定 PPO collapse 属于哪一层失败；[[TC-PPO Algorithm]] 只在 Layer-2 优化动态失稳仍然存在、但 Layer-1 表达能力已经基本满足的区间发挥最大价值。

由此形成了新的论文主线：  
**不是简单地证明 TC-PPO 总是优于 Vanilla PPO，而是证明 TC-PPO 在合适容量区间内能以更小网络实现稳定训练，具备“算力效率 / 轻量部署”价值。**

今天还启动了 [[Slide]] 任务上的 [[128×2]] / [[256×2]] sweet point 实验；但由于并行进程过多导致服务器 C 短暂无响应，最终确认有效保留下来的主要是 Vanilla 结果，TC-PPO 部分需要谨慎重跑。

---

## 关键对话与决策

### 1. 论文整体水平重新评估

对当前 TC-PPO 论文进行了客观定位：

- 总体判断：**CoRL/IROS 边缘，IEEE T-RO / RA-L 长文方向更合适**。
- 优势：
  - [[FTC Metric]] 作为 contact-interval 统计量有物理意义。
  - 三幕叙事诚实：承认早期阈值设定错误，而不是事后美化。
  - 实现细节较完整：FTC、GCTC、Welford、ring-buffer、伪代码等。
  - 网络容量控制实验有价值：用 [[512×4]] 排除小网络表达能力瓶颈。
- 弱点：
  - [[Mode A-B-C Taxonomy|Mode A]] 正面证据偏弱。
  - [[512×4]] Vanilla 已解决不少 collapse，导致 TC-PPO 的边际增益变小。
  - 缺少 PPO-BR、REPPO 等竞品直接对比。
  - Mode B / Mode C 数据更充分，Mode A 数据最稀疏。

形成的论文定位是：

> 这不是一篇“方法显著碾压所有 baseline”的论文，而是一篇“发现接触驱动 PPO collapse 的可诊断机制，并提出选择性稳定框架”的论文。

---

### 2. TC-PPO 训练步数预算：5M 合理，10M 冗余

讨论并写入了训练步数预算分析：

- 论文原始设置中：
  - warm-up：**1M steps**
  - FTC 检查间隔：**100K steps**
  - hysteresis 确认窗口：**W=2**
- 理论最短完整 Mode A 周期：
  - warm-up 1M + 第一次触发检查 0.1M + 关闭确认 0.2M ≈ **1.3M steps**
- 实际推断：
  - Stair 类早期不稳定任务，TC-PPO 机制大约在 **2.5–3M steps** 完成主要作用。
  - 推荐预算：**5M steps**
  - 10M steps 在很多任务上主要是 HumanoidBench 传统设置带来的冗余。

今天确认了新的实验原则：

> 对 sweet point 实验，跑 **5M steps** 即可：1M warm-up + 4M 正式训练；无需继续沿用 10M，除非为了和 HumanoidBench 官方表格严格对齐。

---

### 3. 为什么不同任务中 TC-PPO 效果不同

形成并写入任务差异解释框架。TC-PPO 的效果差异不主要由“任务难度”决定，而由三个因素共同决定：

1. **接触拓扑结构**
   - Walk 类任务天然周期性强，FTC 稳定，通常为 [[Mode A-B-C Taxonomy|Mode C]]。
   - Truck 类任务需要持续接触或非周期接触，FTC 结构性偏低，容易成为 [[Mode A-B-C Taxonomy|Mode B]]。
2. **学习动态**
   - Stair 类任务早期学习阶段可能出现有限时间的不规则接触窗口，后期恢复稳定，适合 TC-PPO。
3. **FTC 分布与阈值窗口是否对准**
   - Mode A 需要 FTC 轨迹跨越 τ_low / τ_high 区间。
   - 如果始终高于阈值，TC-PPO 无需触发。
   - 如果始终低于阈值，TC-PPO 会长期正则化，可能有害。

核心结论：

> TC-PPO 的适用条件是：任务存在由学习过程引发的、有限时间的接触不规则窗口，且该窗口落入 [[FTC Metric]] 的阈值响应区间。

---

### 4. 提出并强化 Two-layer Failure Decomposition

今天最重要的概念推进是进一步巩固 [[Two-layer Failure Decomposition]]：

#### Layer-1：Representational Capacity Failure

- 网络太小，根本无法表达任务所需策略。
- 典型对应 [[64×2]] PPO 在 HumanoidBench 接触丰富任务上的 collapse。
- 这类失败不是 TC-PPO 的靶点。
- 解决方式是增加网络容量，例如 [[512×4]]。

#### Layer-2：Optimization Dynamics Failure

- 网络容量基本够，但接触事件造成梯度 spike / trust-region break。
- FTC 下降可作为早期 warning signal。
- [[TC-PPO Algorithm]] 通过 adaptive trust-region / soft gating 缓解这类失稳。

由此得到新的论文防御逻辑：

> [[512×4]] 的强效果不是对 TC-PPO 的否定，而是说明 Layer-1 表达能力瓶颈长期被低估；TC-PPO 解决的是在 Layer-1 基本满足后暴露出来的 Layer-2 优化动态失稳。

---

### 5. 网络容量与 TC-PPO 效果：倒 U 型关系

今天进一步分析了 [[Network Capacity]] 与 TC-PPO 相对收益之间的关系，并写入论文：

不是“网络越大，TC-PPO 越有效”，而更可能是**非单调倒 U 型**：

```text
TC-PPO 相对提升
↑
|        sweet point
|          /\
|         /  \
|        /    \______
|_______/
+----------------------→ network capacity
  64×2   128×2/256×2   512×4   1024+
```

三个区间：

1. **小网络区：Layer-1 主导**
   - 网络表达不了策略。
   - TC-PPO 即使收紧更新，也只是“沿错误方向走慢一点”。
   - 预测：TC-PPO ≈ Vanilla，甚至更差。

2. **中等网络区：sweet point**
   - 表达能力基本够。
   - 接触扰动仍未被过参数化稀释。
   - TC-PPO 有明确靶点。
   - 预测：TC-PPO 相对 Vanilla 提升最大。

3. **大网络区：Layer-2 自愈**
   - 大量参数维度稀释 gradient spike。
   - 高维参数空间提供绕行路径。
   - Vanilla PPO 自身已较稳定。
   - 预测：TC-PPO 边际收益趋近于 0。

论文中新增了梯度信噪比视角：

- 定义一个概念性 SNR：
  - 有效策略梯度 / 接触扰动梯度。
- Layer-1 区：有效梯度接近不可用，SNR 低但无可保护对象。
- Sweet point：有效梯度与接触扰动量级相当，TC-PPO 最有用。
- 大网络区：扰动被稀释，SNR 高，TC-PPO 作用变小。

---

### 6. [[128×2]] vs [[64×4]] 的机制解释

讨论了为什么优先推荐 [[128×2]] 而不是 [[64×4]]：

- 二者参数量可能接近，但结构不同：
  - [[64×4]]：窄而深，更多串行非线性变换。
  - [[128×2]]：宽而浅，更强并行特征表示。
- 对接触丰富 humanoid 任务：
  - 宽度更直接缓解 Layer-1 表达能力不足。
  - 深度增加反向传播路径，可能放大接触导致的 gradient spike。
- 因此 [[128×2]] 更可能处于：
  - Layer-1 基本解决；
  - Layer-2 仍然存在；
  - TC-PPO 有发挥空间。

同时澄清了网络架构记号：

- [[128×2]] 表示“2 个隐藏层，每层 128 个神经元”，不是普通算术乘法。
- `[512, 256, 128, 64]` 这类可以表示非均匀架构。
- 本文优先使用 `[width × depth]` 的均匀 MLP，是为了：
  - 与 HumanoidBench PPO baseline 可比；
  - 降低超参数搜索空间；
  - 保持实验解释清晰。

### 并行调度教训

今天出现一次明显调度错误：只看 GPU 显存剩余，忽略了 CPU / 内存 / MuJoCo 环境仿真负载。

错误做法：

- 试图同时跑过多训练进程。
- 每个进程又包含 `--envs 4` 个 MuJoCo 并行环境。
- 结果等效环境数过大，服务器 C 出现无响应。

经验教训已更新：

> A800 80GB 显存不是唯一瓶颈；PPO + MuJoCo 的 CPU / 内存压力很大。以后服务器 C 单机同时训练进程数应保守控制，优先稳定完成，不追求显存填满。

今日形成的运行原则：

- `--envs 4` 是每个 run 内部并行环境数，不等于训练进程数。
- 后续不应随意叠加 12–18 个训练进程。
- 服务器 C 上建议最多约 **6 个训练进程**，具体仍需看 CPU、内存和负载状态。
- 如果已有训练在跑，追加任务前必须先检查：
  - GPU 显存；
  - CPU load；
  - 内存；
  - 当前进程数；
  - 是否有关键 run 即将完成。

## 新洞察

### 1. TC-PPO 的核心价值不是“永远比大网络更强”

今天最重要的认知转变是：

> TC-PPO 不应该被包装成“在所有网络规模上都显著优于 PPO”的算法，而应被包装成“在轻量网络 / 中等容量网络上，用接触一致性先验补偿过参数化不足”的稳定化方法。

这与机器人部署场景高度相关：

- 大网络训练或许可以缓解 collapse；
- 但真实机器人控制关心推理延迟、控制频率和部署成本；
- 如果 [[128×2]] + TC-PPO 能接近 [[512×4]] Vanilla，就具备明确工程和科学价值。

---

### 2. 网络容量不是 TC-PPO 的 competitor，而是 gate

两者不是简单相辅相成，也不是谁完全主导，而是存在条件关系：

- [[Network Capacity]] 决定训练失败属于哪一层：
  - Layer-1：表达能力失败；
  - Layer-2：优化动态失稳。
- [[TC-PPO Algorithm]] 只解决 Layer-2。
- 因此容量太小或太大都会让 TC-PPO 看起来“没用”，但原因完全不同：
  - 太小：靶点还没出现；
  - 太大：靶点被大网络自愈。

这使 sweet point 实验成为论文当前最关键的实验。

---

### 3. Mode A/B/C 不是坏消息，而是选择性算法的证据

过去 Mode A/B/C 可能被看成 TC-PPO 不稳定或不一致。今天明确：

- Mode C：不触发，说明算法不干预正常训练。
- Mode A：有限时间触发并恢复，是理想工作区间。
- Mode B：长期触发说明阈值或任务结构不匹配，是诊断信号。

因此 [[Mode A-B-C Taxonomy]] 应被写成“选择性稳定机制的可解释行为”，而不是方法泛化失败。

---

### 4. 5M 实验预算有论文意义

之前 10M 很大程度上是为了遵循 HumanoidBench 传统与保险心理。今天进一步确认：

- 对 TC-PPO 机制本身，5M 足以观察主要作用。
- 这不仅节省算力，也支撑“TC-PPO 在早中期稳定训练中发挥作用”的机制叙事。
- 后续论文中可以将 10M 表格作为 benchmark 对齐，将 5M sweet point 实验作为机制验证 / efficiency study。
