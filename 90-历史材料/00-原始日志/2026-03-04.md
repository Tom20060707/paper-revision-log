这一步的意义在于：为后续批量整合论文图形修改建立了一个可复用流程，尤其适合当前论文中多张 TikZ/LaTeX 图需要逐块替换、但又不能引入额外文本变动的场景。

## 关键对话与决策

### 1. 明确采用“只替换变更片段”的保守策略

用户说明：Claude 和 Gemini 已经修改了论文中绘图相关的一些代码，现在需要在完整论文代码文件夹下，根据提示与锚点执行替换。

今日确认的操作原则：

- 以用户提供的三类信息为准：
  1. 图片锚点；
  2. 原始代码片段；
  3. AI 修改后的新代码片段。
- 优先通过 LaTeX 标签锚点定位，例如 `\label{fig:teaser_overview}`。
- 若后续没有锚点，则使用“原代码片段 → 新代码片段”的精确匹配方式。
- 每次只替换当前目标图块，不主动改动其他论文内容。
- 不对图意、实验结论或论文叙述进行额外发挥，避免引入幻觉。

该决策对后续很重要：论文修订阶段最怕“顺手优化”导致不可追踪改动，因此今天确立的是一种更接近补丁式编辑的流程。

### 2. 第一个图块作为流程验证样例

本次处理对象：

- 文件：`nips2026_revised_v4.tex`
- 图锚点：`\label{fig:teaser_overview}`
- 图类型：Figure 1 / teaser overview，首页概览图
- 原代码特征：包含 TikZ 绘图环境、`resizebox`、`figure` 环境与 caption/label。
- 操作目标：用用户提供的新代码替换旧图代码，其余论文内容保持不变。

实际执行中先尝试定位锚点，随后调整为固定字符串搜索以避免 LaTeX 反斜杠在正则中的转义问题。这是一个小但重要的工程细节：对于 LaTeX 文件，固定字符串搜索通常比正则更安全，尤其是处理 `\label{...}`、`\begin{figure}` 这类语法时。

### 3. 完成替换后做局部复核

- 当前仅处理该图块；
- 未报告对其他图、正文或实验段落的改动。

这一步确认了本轮整合没有扩大编辑范围，后续可以继续按同样协议处理第二张图。

## 新洞察

### 1. LaTeX 图代码整合应采用“锚点优先，片段兜底”

今天验证了一个实际可行的论文图整合流程：

1. 有锚点时，用 `\label{...}` 精确定位；
2. 找到标签附近完整 `figure` 环境；
3. 只替换对应环境；
4. 替换后复核 label/caption 是否仍在；
5. 每次只处理一张图，避免批量误替换。

这种方法比直接全文搜索大段旧代码更稳健。因为 TikZ 代码通常很长，且包含大量重复样式、节点、箭头命令，直接匹配大段文本容易因空格、换行或注释差异失败；而锚点是论文结构中相对稳定、语义明确的定位单元。

### 2. 固定字符串搜索比正则更适合初始定位 LaTeX 标签

处理中出现了一个小调整：定位 `\label{fig:teaser_overview}` 时，固定字符串搜索更安全。原因是 LaTeX 命令中的反斜杠、花括号等字符在正则中具有额外含义，若不仔细转义容易定位失败或误匹配。

后续建议：除非确实需要复杂模式匹配，否则对于 LaTeX 锚点应优先使用 literal search。

### 3. 图修订的核心风险不是“改不动”，而是“多改了”

今天的流程强调“仅替换被更改过的片段”，这其实是论文修订中非常关键的控制点。尤其是当前论文可能已经经过多轮 Claude/Gemini/人工改写，如果在整合图代码时同时改动正文、caption、引用或宏定义，会导致版本差异难以追踪。

因此后续每个图块都应保留类似记录：

- 图编号或功能；
- 锚点；
- 替换范围；
- 是否复核 label/caption；
- 是否涉及正文同步修改；
- 是否需要重新编译确认。

### 4. Teaser 图是方法故事线的压缩表达

当前 teaser overview 的标题线索为：

> Problem → Signal → Adaptive Control → Stable Outcome

这与 TC-PPO 论文的核心叙事相当贴合：从任务中的不稳定或失败现象出发，通过可观测信号与代价指标进入自适应策略控制，最终获得更稳定的学习或执行结果。后续若继续优化该图，应重点检查它是否准确承载了 [[FTC Metric]]、[[TC-PPO Algorithm]] 与 [[Mode A-B-C Taxonomy]] 的关系，而不是仅仅追求视觉复杂度。
