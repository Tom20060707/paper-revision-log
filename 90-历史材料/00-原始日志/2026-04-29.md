# 2026-04-29 研究日记

## 今日主线

今天的核心推进是：**把论文从一个过窄的 `CTS = 1 - CV²` 指标论文，重构为“人形机器人接触域一致性与稳定性研究”的框架论文/方法论文**。

主要完成了四个方向的统一：

1. **概念主线重构**
   - 区分了“接触时间一致性”和“接触域稳定性”：
     - 一致性：诊断信号 / 过程指标 / 可观测量。
     - 稳定性：目标性质 / 训练与策略是否不崩。
   - 明确写法：
     - `Contact timing consistency` 不是最终目标，而是接触域退化的可观测诊断信号。
     - `Contact-domain stability` 是 [[TC-PPO Algorithm]] 希望改善的系统级性质。

2. **公式体系重构**
   - 原本 `CTS = 1 - CV²` 被认为过于依赖周期性接触，难以解释 kitchen、truck、push、insert、短程加速、楼梯平台等非周期任务。
   - 因此把旧 CTS 公式降级为：
     - [[FTC Metric]] / CTS 在准周期支撑接触上的轻量实例。
   - 新增更一般的 [[Contact-Surface Consistency]] / CSC 框架：
     - 比较当前接触事件分布与稳定参考分布。
     - 非周期任务不再强行要求接触间隔等长，而是看接触占用、驻留时间、接触面转移、进度条件接触等模式是否一致。

3. **机制链条重构**
   - 原来容易被理解成：
     - “稳定了就 Push”
     - 或 “Protect 和 Push 是两个并列模块”
   - 今天明确改成：
     - 崩溃苗头出现 → 崩溃前预警 → 预警后保护 → 形成稳定平台期 → 谨慎推动 → 高回报 / 成功。
   - 但也进一步修正：**保护不是推动的普遍前提，稳定平台期才是推动的前提**。
     - Mode A：保护后进入稳定平台，再 Push。
     - Mode C-E：天然稳定但进展停滞，可以不经过 Protect，直接从稳定平台进入 Push。

4. **Mode A/B/C 重新解释**
   - 不再把 A/B/C 当成任务好坏分类。
   - 改成 [[Mode A-B-C Taxonomy]]：用于界定稳定域、保护门状态和是否形成稳定平台。
   - Mode C 进一步细分：
     - C-S：稳定且有进展，不需要干预。
     - C-E：稳定但探索/进展停滞，需要 Push。
     - C-M：门没触发但任务也没进入稳定有效状态，属于漏检/越界情况。

### 2. “Object” 不应作为 CTS 通道

早上围绕任务-通道适配图讨论时，一度提出在 manipulation 中加入 `Object` 或 `Object State`。随后用户指出：

> object 状态变化和算法、接触稳定性有什么关系？这不是算法直接控制或监测的量。

因此做出关键修正：

- 不把 `Object State` 写成 CTS 通道。
- `Object State` 只能是任务结果 / success metric / reward 相关内容。
- 如果要表达 manipulation 中的接触关系，应写：
  - `Hand-Obj Contact`
  - 而不是单独 `Object`
- 这帮助论文避免让审稿人误解为：
  - CTS 直接计算了物体状态变化。

### 3. “一致性”与“稳定性”的概念分工

今天最重要的概念之一是明确：

> 一致性是尺子，稳定性是目标。

形成的论文逻辑：

```text
接触时间一致性下降
↓
接触域退化
↓
训练不稳定 / 策略崩溃
↓
CTS-PPO 介入
↓
恢复接触域稳定性
```

对应概念分工：

| 概念 | 角色 | 含义 |
|---|---|---|
| 接触时间一致性 | 诊断信号 / 指标 | 接触事件是否呈现可重复模式 |
| 接触域稳定性 | 目标性质 | 策略学习是否在接触密集交互下保持鲁棒 |
| CTS / CSC | 观测与预警 | 发现接触域退化 |
| CTS-PPO | 稳定机制 | 抑制退化、改善训练稳定性 |

这直接推动了图 1 的框架改写。

### 4. 图 1 框架确定：崩溃前预警

图 1 被定义为：

```text
接触域研究
|
接触域退化
/              \
诊断信号        稳定机制
|               |
接触时间一致性   接触域稳定性
|               |
崩溃前预警       崩溃抑制
|               |

```

其中一个重要措辞从 `Early Warning` 改成：

- `Pre-Collapse Warning`

中文对应：

- “崩溃前预警”

理由是：核心贡献不是发现已经崩了，而是在 reward 明显下降或任务彻底失败前，通过接触一致性看到崩溃苗头。

### 5. 图 2 从三层改为四层研究边界

用户指出原来的三层接触建模 taxonomy 不符合想表达的意思。真正要表达的是**研究范式边界**：

- 本文只研究“接触面”的一致性与稳定性。
- 不研究手指弯曲、软体结构、触觉感知等更深层问题。

因此将原来的三层改成四层：

```text
Level I — Surface
Level II — Rigid digits
Level III — Soft digits
Level IV — Tactile
```

具体含义：

| Level | 含义 | 本文关系 |
|---|---|---|
| Level I — Surface | 拳头/脚底/接触面级，只看外部接触面什么时候接触 | 本文研究 |
| Level II — Rigid digits | 手展开但手指不弯曲，相当于多个刚性接触面 | 可作为未来扩展 |
| Level III — Soft digits | 手指/关节可弯曲、柔顺、变形 | 不涉及 |
| Level IV — Tactile | 触觉、压力、滑移、纹理、传感反馈 | 不涉及 |

图 2 也改为极简四层图，只保留：

```text
Surface → Rigid → Soft → Tactile
```

并且只在 `Surface` 下标注 `This work`。

### 7. 百分位阈值、自适应阈值、硬门控、软门控的层级关系厘清

围绕阈值产生了多轮讨论，最后形成清晰区分：

```text
CTS/CSC 指标：测量值
Threshold 阈值：判定边界
Hard gating：二值开/关干预
Soft gating：连续强度干预
```

进一步明确：

- 不应把“自适应阈值”和“百分位阈值”写成两个并列贡献。
- 更合理写法：
  - `percentile-calibrated thresholds`
  - 或 `warm-up percentile thresholds`
- 百分位阈值就是从 warm-up CTS 分布中直接计算：
  - `τ_low = P5`
  - `τ_high = P75`

其作用不是保证所有任务都进入 Mode A，而是排除“固定全局阈值不适配”这个混杂因素。

## 新洞察

### 1. `1 - CV²` 不能承担“接触域通用指标”的角色

今天最关键的理论洞察是：

> 如果把 `CTS = 1 - CV²` 写成所有接触密集任务的通用指标，论文会被轻易攻击。

原因：

- 它隐含“好接触 = 接触间隔近似周期”。
- 但大量人形机器人任务是非周期或阶段性的：
  - kitchen
  - truck
  - push
  - insert
  - 楼梯平台切换
  - 起跑 / 短跑 / 加速
- 因此旧公式只能作为准周期支撑接触实例，而不能作为总理论。

更稳妥的框架是：

```text
General metric: Contact-Surface Consistency
Evaluated instance: CTS for quasi-periodic support contact
Mechanism: CTS-PPO for stabilization
```

### 2. “保护成功”不等于“任务成功”

今天进一步明确：

- Protect 的目标是让策略不崩。
- Push 的目标是让策略继续取得任务进展。
- 不崩溃和成功是两个概念。

因此机制链必须写成：

```text
保护让策略进入稳定域
推动利用稳定域提升任务进展
```

而不能写成：

```text
保护成功 = 最终任务成功
```

### 3. Reward 可作为阶段轴，但不能作为唯一成功指标

用户提出可以用 reward 分阶段，但不能只用 reward 判断任务完成好坏。最终形成：

- reward / return 可以作为训练过程的外轴：
  - 深度负值
  - 小负值
  - 零附近
  - 小正值
  - 平台期
  - 上升期
- 但任务成功应尽量结合：
  - unshaped return
  - benchmark success flag
  - task-specific success metric
  - object-state success（若任务有）

这对 kitchen 类任务尤其重要。

### 4. Mode A/B/C 不是好坏分类，而是稳定域分类

今天对 [[Mode A-B-C Taxonomy]] 的理解发生明显升级：

旧理解容易变成：

```text
Mode A = 好
Mode B = 差
Mode C = 不重要 / 差
```

新理解：

```text
Mode A = 可恢复退化域
Mode B = 保护闭环未闭合域
Mode C = 未触发域，需要进一步细分
```

尤其 Mode C 不一定差：

- C-S：稳定且有进展，不需要干预。
- C-E：稳定但无进展，需要 Push。
- C-M：门没触发但任务失败，可能是漏检或越界。

这使 A/B/C 不再是结果排名，而是机制诊断工具。

### 5. Mode B 的意义分阶段变化

今天厘清了 Mode B 与阈值的关系：

- 固定阈值阶段：
  - Mode B 可能是全局阈值不可达。
  - 百分位阈值 / warm-up percentile thresholds 的作用就是排除这个问题。
- 百分位阈值之后：
  - 若仍出现 Mode B，就不应再简单归因于阈值设错。
  - 更可能说明当前接触面信号没有形成“退化 → 保护 → 恢复”的闭环。

这使百分位阈值的论文价值更清楚：

```text
它不是万能修复器，而是用于排除全局阈值不适配这个混杂因素。
```

### 6. “稳定平台期”比“保护”更适合作为 Push 的必要前提

一度将链条写成：

```text
Push ⇒ Hold ⇒ Protect
```

但这会无法解释 Mode C-E：

- 有些任务天然稳定，不需要 Protect。
- 但仍可能奖励停滞，需要 Push。

最终改成：

```text
Push ⇒ Hold
```

也就是：

- 推动必须发生在稳定平台期之后。
- 稳定平台期可以来自：
  - Protect 后恢复；
  - 或天然稳定。

这是今天机制逻辑最重要的修正之一。

   - CSC 总框架、非周期 CSC、四层接触边界、Mode C 细分都增强了论文叙事。

     - Protect 是稳定器；
     - Push 是稳定域上的进度利用；
     - 二者不是松散拼接。
   - Push 的位置应服务于 kitchen / sparse-progress 任务，而不是让论文变成另一个 reward shaping 论文。

3. **图太多可能造成叙事分散**
   - 当前至少涉及：
     - 图 1 总框架
     - 图 2 四层边界
     - 图 3 状态机
     - 奖励阶段曲线
     - 多轨迹组合图
   - 明天需要检查是否有重复表达。

4. **Mode C 细分容易复杂化**
   - C-S / C-E / C-M 是合理的，但需要简洁说明。
   - 如果写得太细，会让审稿人觉得 taxonomy 过度设计。

5. **“崩溃后预警”与“崩溃前预警”措辞需统一**
   - 今天最终倾向 “崩溃前预警” / `Pre-Collapse Warning`。
   - 但用户口述中曾出现“崩溃后预警”的链条描述。
   - 论文中应统一使用：
     - collapse symptoms emerge
     - pre-collapse warning
   - 避免给人“已经崩了才预警”的印象。
