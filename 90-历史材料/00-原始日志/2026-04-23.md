## 算法/公式变化

新增 PBRS 公式（已写入论文 Eq.10）：

r'_t = r_t + λ_prog × (γ × Φ_prog(s_{t+1}) - Φ_prog(s_t))

- Φ_prog(s) = 已完成子任务加权和（kitchen：4个子任务各权重1.0）
- λ_prog 默认 0.2，扫描范围 0.05/0.1/0.2/0.5
- 理论保证：Ng et al. 1999 PBRS 定理，最优策略不变

新增平台期检测公式（Eq.11）：
- 用 OLS slope 检测最近 N=20 个 episode 的 return 斜率
- slope < ε_progress = 0.01 时判定为平台期，激活 push mode

三模式框架（替换原 Mode A/B/C 的叙事角色）：
- Protect mode = 原 Mode A：FTC低，踩刹车
- Push mode = 新增：FTC稳定但进度停滞，激活PBRS
- Normal mode = 原 Mode C：标准PPO

## 新洞察

1. mjlab 的实质：MuJoCo Warp 把物理仿真从 CPU 搬到 GPU，单卡同时跑4096个环境。你现在实际速度约250 steps/秒（5M steps≈5.5小时反推），mjlab 可达50,000-200,000 steps/秒，差距200-800倍。迁移成本5-7周，不值得在这篇论文里做。

2. Type-I vs Type-II Mode C 是真正的理论贡献：
   - Type-I：接触质量天然好，FTC从不低，任务能学（walk、balance_simple）
   - Type-II：FTC不低但任务极难，reward=0（kitchen、maze）—— FTC盲区边界的精确定义

3. PBRS 关键设计：FTC 做"阶段选择器"而非势函数本身。FTC稳定时才激活push mode，push mode 里用子任务完成度做Φ_prog。两个信号独立不混淆。

4. 竞品格局：ASAP 88/100 > H2O 80/100 > HumanoidVerse 73/100 > TC-PPO 69/100。差距主要在：无真机（-10分），Mode A 触发率低（7/24任务）。

## 修正的旧认知

1. 旧：mjlab 对论文有"工程系统化"价值 → 修正：唯一实质价值是训练速度，迁移成本太高无法在本论文使用
2. 旧：GPU 在跑仿真 → 修正：仿真在 CPU，GPU 大部分时间在等待，实际利用率10-30%；mjlab 才能让GPU达到80-95%
3. 旧：Mode C 是单一的"安全保证" → 修正：Mode C 分 Type-I/II，Type-II 是PBRS的真正目标
