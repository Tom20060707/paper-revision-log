# 2026-04-01 研究日记

## 今日主线

今天的核心工作是围绕 [[TC-PPO Algorithm]] 的真实训练状态做了一次较彻底的排查与纠偏。

一开始计划是：

- A800（Server C）跑 0.1B / 100M long-horizon pilot；
- Server B（三卡 4090）跑补 seed、消融实验；
- Server A/B 历史数据作为已有主实验基础。

但在排查过程中发现一个关键风险：Server A/B 历史实验虽然文件名里有 `tc_ppo_*`，实际运行脚本很可能只是标准 SB3 PPO。随后进一步确认：

- Server A/B 的 CSV/log 是真实 HumanoidBench 仿真训练数据；
- 但这些数据不是 TC-PPO，而是 vanilla PPO，可作为 PPO baseline；
- A800 上真正正在运行的是 `train_real.py`，包含真实环境、[[FTC Metric]]、entropy/KL 自适应逻辑，正在跑 walk/truck 的 TC-PPO vs vanilla PPO 100M 对照实验；
- A800 上旧的 `run_100M.sh` / `train.py` 报错记录是旧脚本，不代表当前运行失败。

今天最大的推进是把“哪些数据真实、哪些算法真实、哪些实验还缺失”重新理清了。

## 关键对话与决策

### 1. 明确三台服务器角色

最终统一命名：

| 服务器 | 硬件 | 今日确认状态 |
|---|---:|---|
| Server A | 单卡 4090 | 历史数据真实，但属于 vanilla PPO baseline |
| Server B | 三卡 4090 | 历史数据真实，但属于 vanilla PPO baseline；当前无训练进程 |
| Server C / A800 | A800 80GB | 正在运行真实 TC-PPO + vanilla PPO 100M 长时序对照 |

这一点非常重要，因为早些时候混淆了 Server A/B/C 的身份，导致对数据身份的判断一度摇摆。

### 2. Server A/B 历史数据的身份被重新定义

Server B 的 Claude Code 检查 `/root/humanoid-bench/ppo/run_ppo_paper.py` 后反馈：

- 使用 `stable_baselines3.PPO("MlpPolicy", ...) + model.learn()`；
- 没有 FTC 计算；
- 没有接触状态逻辑；
- 没有 entropy coefficient 动态调整；
- 没有 KL penalty 自适应；
- callback 只记录 return，不改变训练逻辑。

因此结论是：

> Server A/B 中以 `tc_ppo_*` 命名的历史实验，实际是 vanilla PPO 在真实 HumanoidBench 环境上的训练结果。

这不是“假数据”，而是“算法标签错误”。后续论文中不能把它们作为 TC-PPO 结果，只能作为 PPO baseline 使用。

### 3. A800 误判被纠正

早些时候查看到 `~/autodl-tmp/run_100M.sh` 调用 `python train.py` 并报错：

- `python: can't open file '/root/autodl-tmp/train.py'`
- `conda activate` 初始化失败

一度误判 A800 100M 实验没有跑起来。

随后通过 `nvidia-smi` 和 `ps aux` 确认 A800 实际有 4 个训练进程正在运行，且命令行使用的是：

- `train_real.py`
- `h1hand-walk-v0`
- `h1hand-truck-v0`
- `tc_ppo` 与 `vanilla_ppo`
- 100M steps

并且 GPU 利用率约 89%，显存占用约 19GB，说明当前 A800 确实在训练。

最终确认：

> A800 当前真实运行的是 `train_real.py`，旧 `run_100M.sh` 报错只是旧脚本残留，不代表当前 4 个后台训练失败。

### 4. 确认 `train_real.py` 是真实 TC-PPO 实现

A800 上读取 `~/autodl-tmp/tc_ppo/train_real.py` 后确认该文件共 492 行，结构完整：

| 模块 | 内容 |
|---|---|
| imports / 环境设置 | 设置 `MUJOCO_GL=egl`，加入 humanoid-bench 与 tc_ppo 路径 |
| `FTCTracker` | 计算 per-limb contact intervals 和 FTC |
| `ContactWrapper` | 从 MuJoCo contact / geom 数据提取 6 路接触状态 |
| `TCPPOCallback` | FTC < 0.8 时进入 stabilization：entropy ×2，`target_kl=0.05`；FTC ≥ 0.9 且连续 2 次后恢复 |
| `ResultCallback` | 保存 `summary.json`，每 20 rollout 写一次 |
| `make_env` | `gym.make` + `ContactWrapper` + seed |
| `train()` | SB3 PPO，网络 `[512, 512, 512, 512]`，learning rate `3e-5`，checkpoint 每 500k steps |
| `main()` | argparse 入口 |

这表明 A800 的 `train_real.py` 不是 mock 随机数训练，而是真实 HumanoidBench + TC-PPO 逻辑。

### 5. 重新定义实验缺口

根据今天排查，当前实验缺口不是“所有数据都是假的”，而是：

1. PPO baseline：Server A/B 已有，真实可用；
2. TC-PPO 100M long-horizon：A800 正在跑 walk/truck seed 0；
3. TC-PPO 27 任务 10M 主实验：尚未完成，需要用 `train_real.py` 在真实环境上跑；
4. 大并行环境实验：需要另行设计，不能直接假设 HumanoidBench 原版支持 4096/8192 envs。

## 新洞察

### 1. “真实数据”和“真实算法”必须分开审计

今天最重要的方法论教训是：训练数据可能是真实环境生成的，但算法标签可能是错的。

Server A/B 的数据不是 mock，也不是伪造，而是真实 HumanoidBench + SB3 PPO。但它们并没有执行 [[TC-PPO Algorithm]]，因此不能因为目录名叫 `tc_ppo_*` 就认为是 TC-PPO 结果。

后续所有实验数据应至少记录：

- 运行脚本文件名；
- git commit 或脚本 hash；
- 关键参数；
- 是否包含 FTC 计算；
- 是否包含动态 entropy/KL；
- 训练命令；
- 日志中是否记录 FTC/stabilization 状态。

### 2. A800 长时序实验的定位是正确的

用户一度怀疑 A800 100M vanilla PPO 对照没有意义，但最终澄清：

- 100M long-horizon 不是为了跑 27 任务主结果；
- 它是为了验证 collapse-recovery / 长时序稳定性；
- 因此必须同时有 TC-PPO 和 vanilla PPO；
- 没有 vanilla PPO，就无法证明 TC-PPO 更稳定。

所以 A800 当前 walk/truck × TC-PPO/vanilla PPO × 100M 的设计是合理的。

### 3. 27 任务主实验不需要 100M

主实验应遵循 HumanoidBench 原文设置：

- 27 tasks；
- 10M steps；
- 4 parallel envs；
- seeds 0/1/2；
- learning rate `3e-5`；
- PPO baseline 对照。

100M 只用于少数代表任务的长时序 pilot，不应扩展到全部 27 个任务，否则成本过高且不符合主实验定位。

### 4. 大并行环境实验是潜在亮点，但需要谨慎落地

用户提出 4096/8192 并行环境，这与 [[TC-PPO Algorithm]] 的动机高度相关：并行 PPO 在大并行 rollout 下更容易发生 temporal inconsistency / collapse，TC-PPO 应更有优势。

但现实约束是：

- 当前 `train_real.py` 基于 MuJoCo + SB3 + `SubprocVecEnv`；
- 4096/8192 envs 更像 Brax/JAX 并行仿真设置；
- HumanoidBench 原版未确认支持如此规模；
- 实际可先做 4/16/32/64 envs 的 scaling ablation，待确认性能与可行性。

### 5. 论文叙事需要从“已有 TC-PPO 全结果”改成“baseline 已有，TC-PPO 正在补齐”

当前论文不能写成“27 任务 TC-PPO 已经完成”，因为真实 TC-PPO 主实验还没跑完。更稳妥结构：

- PPO baseline：已有真实多任务数据；
- TC-PPO main results：待跑 / pending；
- long-horizon stability：A800 正在跑；
- ablation：pending；
- large-parallel scaling：pending / future work，视实验完成情况决定。
