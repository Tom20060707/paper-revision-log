### 4. 明确 Table 4 / Figure 5 的真正漏洞

用户指出：表 4 看起来 Mode C 很多，会让人误以为 TC-PPO 大多数时候不触发，A/B 很少。但实际上很多任务没有用新版 `adaptive threshold + soft gating` 重新跑过。

经过讨论，形成了更准确的判断：

- 当前问题不是“Mode C 多一定错误”。
- 真正问题是：**这些 Mode C 是否是在新版机制下测出来的？**
- 如果很多 Mode C 来自旧版固定阈值，那么它们不能作为“新版 TC-PPO 不触发”的证据。
- 对审稿人来说，这会削弱 [[Mode A-B-C Taxonomy]] 的可信度，也会削弱 “selective stabilization” 的 generality。

重要修正：

- 不应声称 Mode A 必须占多数。
- 更严谨的说法是：  
  **对于 Vanilla 崩溃、高方差或 FTC 有明显下降的任务，新版 TC-PPO 应该能触发 Mode A；对于天然稳定任务，Mode C 是合理的零干扰结果。**

### 5. 下一轮实验策略：保底 + 核爆

最终形成的实验策略不是重跑所有 24 个任务，而是做最小高冲击力闭环。

#### 第一优先级：`push`

理由：

- 当前 FTC 约 `0.374`，接近 `τ_low=0.35`。
- warmup 阶段很可能出现更低 FTC。
- 自适应 `P5/P75` 阈值可能更容易捕捉到不稳定阶段。
- 是从 Mode C 转 Mode A 的保底候选。

#### 第一优先级：`balance_hard`

理由：

- FTC 分布方差大。
- 适合验证 soft gating 是否能在高波动任务中稳定触发。
- 若转为 Mode A，可以直接增强 Table 4 / Figure 5 的正证据。

#### 高风险高回报：`package`

已知：

- Table 4 中 `package` 的 Vanilla 约 `-8992±813`，接近完全崩溃。
- 但 FTC 约 `0.447`，高于固定 `τ_high=0.42`。

这产生一个重要矛盾：

- 如果 reward 崩溃但 FTC 仍高，说明 package 的失败可能不经过 FTC 下降通道。
- 这意味着 package 可能不是 [[FTC Metric]] 能检测的 Layer-2 collapse，而可能是 Layer-1 或其他失败模式。
- 因此 package 是“核爆任务”，但不是稳赚任务。

决策：

- 仍然要跑 package，因为成功会极大增强 necessity 证据。
- 但必须完整记录 FTC 时间序列。
- 如果失败，也可以诚实写成 FTC 的适用边界，而不是硬解释成失败。

今天对服务器 B/C 做了环境梳理和实验部署。

- 服务器 B 并非“没有代码”，而是代码在不同目录中。后来发现至少有多个 `train_real.py` 版本：
  - `/root/autodl-tmp/tc_ppo/train_real.py`

  - `/root/train_real.py`
- 旧版和新版 `train_real.py` 行数和 md5 不同，但核心算法逻辑一致。
- 差异主要是 GPU 指定方式：
  - 旧版支持 `--device cuda:X`
  - 新版使用 `CUDA_VISIBLE_DEVICES`
- [[FTC Metric]]、soft gating、自适应阈值、Mode 检测等核心逻辑判断为一致。

自适应阈值确认：

已查到部分任务实际自适应阈值不同于固定值：

| 任务 | `τ_low` / P5 | `τ_high` / P75 | 固定参考 |
|---|---:|---:|---:|
| stair | `0.446` | `0.468` 或附近批次值 | `0.35 / 0.42` |
| slide | `0.375` | `0.477` | `0.35 / 0.42` |
| sit-hard | `0.406` | `0.450` | `0.35 / 0.42` |

这支持了一个关键解释：

- 论文中的 `0.35 / 0.42` 不应被表述成所有任务永久固定阈值。
- 更准确的写法应是：固定阈值是默认/早期设置；新版方法使用 warmup FTC 分布的 P5/P75 做任务自适应校准。

## 新洞察

### 1. Mode C 不是坏结果，但“旧机制下的 Mode C”不能证明新版机制不触发

今天最关键的概念澄清：

- Mode C 本身可以是好事：说明稳定任务上 TC-PPO 不干扰 PPO。
- 但如果 Mode C 是旧硬门控 + 固定阈值下的结果，就不能拿来支撑新版 soft gating + adaptive threshold 的 generality。
- 对论文来说，必须明确哪些 Mode 是“旧阈值诊断”，哪些是“新版实测”。

建议论文中引入更清晰的标注，例如：

- `Mode-C_old`: fixed-threshold diagnostic
- `Mode-A_adaptive`: rerun with P5/P75 + soft gating
- `pending`: not yet rerun under adaptive setting

---

### 2. 自适应阈值不是小修小补，而是 TC-PPO 的核心机制之一

今天对 stair/slide/sit-hard 的阈值检查说明：

- 自适应阈值可以明显偏离固定 `0.35/0.42`。
- 这不是异常，而是设计目标。
- 固定阈值更像全局默认或早期诊断工具。
- 新机制的核心应该表述为：**用 warmup 分布定义任务局部门控边界**。

这会影响论文写法：

- 不应把 `τ_low=0.35, τ_high=0.42` 写得像所有任务统一使用的最终阈值。
- 应强调 `P5/P75` 的任务自适应性，并在实验表中记录实际 calibrated thresholds。

---

### 3. package 是双刃剑：成功是核爆，失败也是边界证据

`package` 的矛盾非常有价值：

- reward 崩溃到约 `-8992±813`
- 但 FTC 约 `0.447`，高于 `τ_high=0.42`

这说明它可能不是 FTC-leading collapse。若 TC-PPO 救不回来，不应硬说失败，而应把它定义为：

> FTC 无法检测所有失败；它主要针对 foot-contact temporal inconsistency 驱动的 Layer-2 optimization collapse。

这反而可以让论文更诚实、更像成熟的机器人学习论文。

---

### 4. 下一轮实验的成败标准应从“分数提升”扩展到“机制闭环”

不要只看 final reward。更重要的是：

1. warmup FTC 分布是否支持自适应阈值。
2. gate 是否在 FTC 下降时触发。
3. gate 是否随后关闭。
4. reward 是否稳定或恢复。
5. 是否从旧 Mode C 转为新版 Mode A。
6. 若失败，FTC 是否确实没有提前下降。

这才是 [[TC-PPO Algorithm]] 的 mechanism validation。
