# 2026-04-29 研究日记

## 核心进展

今天真正推进的是：把论文主线从“一个过窄的 CTS 指标”重构为**人形机器人接触域的一致性与稳定性研究**。

核心转向：

- 不再把 `CTS = 1 - CV²` 当成所有接触任务的通用指标。
- 将其降级为 [[FTC Metric]] / CTS 在**准周期支撑接触**上的一个轻量实例。
- 引入更上层的 **CSC / Contact-Surface Consistency** 框架，用于覆盖非周期、阶段化、事件驱动的接触任务。
- 重新解释 [[Mode A-B-C Taxonomy]]：A/B/C 不是任务好坏分类，而是**稳定域状态分类**。
- 明确 [[TC-PPO Algorithm]] 的完整链条：

```text
崩溃苗头出现
→ 崩溃前预警
→ 预警后保护
→ 保护产生稳定域
→ 稳定保持一段时间
→ 谨慎推动
→ 得分提高并成功
```

今天的关键认知是：**保护不是最终目的，保护是推动的前提；不崩溃和任务成功不是一回事。**

### 2. 图 2：三层改为四层研究边界

原来的三层接触建模 taxonomy 不符合真实意图。今天明确为四层研究范式边界：

```text
Level I — Surface
Level II — Rigid digits
Level III — Soft digits
Level IV — Tactile
```

对应解释：

- **Level I / Surface**：只看拳头、脚底、手掌等外部接触面何时接触。本文主要研究这一层。
- **Level II / Rigid digits**：手指展开，但作为刚体多接触面，不考虑弯曲。
- **Level III / Soft digits**：考虑手指/关节弯曲、柔顺、软体变形。
- **Level IV / Tactile**：加入触觉、压力、滑移、纹理、传感反馈。

重要边界：

> 本文研究的是接触面之间的一致性与稳定性，不进入灵巧手弯曲、软体结构和触觉感知。

图 2 也被压缩成极简图：

```text
Surface → Rigid → Soft → Tactile
```

仅 `Surface` 标为 `This work`。

### 4. 百分位阈值、自适应阈值、软/硬门控的层级被理清

今天明确了几个概念不是重复，而是机制链上的不同层级：

```text
CTS / CSC 指标：测什么
Threshold 阈值：何时算异常
Hard gating：是否干预
Soft gating：干预多强
```

比喻：

```text
CTS/CSC = 温度计
Threshold = 发烧线
Hard gate = 超线就开药
Soft gate = 轻烧少干预，高烧强干预
```

同时明确：

- “自适应阈值”不要作为抽象贡献单独啰嗦写。
- 百分位阈值就是自适应阈值的具体计算方式。
- 更干净的写法是：

```text
warm-up percentile thresholds
```

或：

```text
percentile-calibrated thresholds
```

当前具体设定保留：

```text
τ_low = P5
τ_high = P75
```

来自 warm-up 阶段的 CTS/CSC 分布。

## 新洞察

### 1. `1 - CV²` 不是错，而是被放错了位置

今天最关键的洞察：

> `CTS = 1 - CV²` 不能作为接触域任务的总公式，它只适合准周期支撑接触。

原因：

- 很多任务天然非周期。
- 楼梯任务上平台时节奏会变。
- 短跑有起跑、加速、冲刺阶段。
- 推箱子、厨房、插入类任务是阶段化/事件驱动接触。
- 物体交互接触不是稳定步态周期。

因此，旧公式必须降级为：

```text
CSC 总框架下的一个支撑接触实例
```

而不是总理论。

### 2. 一致性不应等同于周期性

新的定义应更宽：

```text
一致性 = 在相似任务阶段，接触事件分布是否稳定、可重复、不过度漂移
```

而不是：

```text
接触间隔是否等长
```

这为非周期任务打开空间。非周期任务可考虑：

- contact dwell time
- contact order
- contact coverage
- phase-conditioned contact presence
- 当前接触事件分布与参考分布的距离

### 3. A/B/C 不是好坏榜，而是稳定域划分

今天纠正了一个叙事误区：不能默认为 Mode A 好、Mode C 差。

更合理解释：

```text
Mode A：可恢复退化域
Mode B：当前信号下不可闭合的退化域
Mode C：未触发域
```

其中 Mode C 还需要细分：

```text
C-S：stable contact，稳定且无需干预
C-M：missed instability，可能漏检
C-E：stable but exploration-limited，稳定但探索/进展停滞，需要 Push
```

这避免了把 Mode C 简单写成失败或无效。

### 4. Mode B 的意义要和百分位阈值结合

Mode B 最初的重要意义不是“方法失败”，而是解释为什么需要百分位阈值。

逻辑应写成：

```text
固定全局阈值
→ 出现 Mode B
→ 可能是阈值不可达
→ 用 warm-up percentile thresholds 排除全局阈值错配
→ 若仍 Mode B，则说明当前接触一致性信号没有形成可恢复闭环
```

所以百分位阈值的价值不是强行让所有任务进入 Mode A，而是：

> 排除阈值错配这个混杂因素，使剩余 Mode B 更具有诊断意义。

### 5. 保护和推动不是并列，而是前后依赖

今天最后形成了最重要的算法叙事：

```text
保护是推动的前提
推动是保护后稳定域上的谨慎利用
```

不能在深度负 reward 崩溃区直接 Push。必须先：

```text
CTS/CSC 恢复
保护门关闭
稳定保持一段时间
unshaped return 超过阶段地板
进入平台期
```

才允许推动。

### 6. 奖励可作为阶段外轴，但不能单独评估成功

奖励曲线可以用于划分训练阶段：

```text
深度负值崩溃区
→ 小负值/零附近恢复区
→ 小正值稳定区
→ 平台期
→ 推动后增长区
```

但任务是否成功不能只看 reward，还要看：

- unshaped return
- benchmark success flag
- object-state success
- 任务完成指标

这点对 kitchen 类任务尤其重要。
