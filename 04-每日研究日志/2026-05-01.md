# 2026-05-01 研究日记

今天这篇是补全版。早上那一版是在 09:14 左右生成的，只记录了“准备跑日志同步”这件事；但 10:00 以后其实发生了很多真正重要的讨论和论文修改。重新扫本地记录后，5 月 1 日 10:00 到 24:00 之间能读到 147 条去重后的 AI 对话事件，核心都围绕 TC-PPO 的算法解释、状态机扩展和论文重构展开。

## 今天真正推进了什么

今天最大的变化，不是又加了一个小模块，而是把原来比较散的“CTS 触发保护 + Push 推动探索”整理成了一个更完整的接触健康调节框架。

原来的思路大概是：

- 接触不稳时，打开 Protect；
- 接触稳定但任务不进步时，打开 Push；
- 用 CTS 作为主要诊断信号。

今天讨论后，论文口径被推进成：

- 先识别状态；
- 再判断这是不是偶然波动；
- 接近危险时先预警；
- 真正不稳时进入保护；
- 保护救不回来时考虑挽救机制；
- 稳定以后，如果卡在平台期，再小剂量推动；
- 推动过程还要监控，避免“练过头”反而把策略推坏。

这比“一个指标触发两个动作”更像一个系统，也更符合接触类机器人训练里的真实情况：策略不是简单地好或坏，而是在 normal、warning、protect、hold、push、rescue 这些状态之间移动。

## 上午到下午：先把 PPO 和门控逻辑讲透

下午先回到 PPO 的基础。用户提到“clip 函数忘了”，于是重新梳理了 PPO clipped objective。

关键点是：

```text
r_t(theta) = pi_theta(a_t | s_t) / pi_old(a_t | s_t)
```

它表示新策略相对旧策略，对同一个动作的概率放大或缩小了多少。PPO 的 `clip` 会把这个比例限制在：

```text
[1 - epsilon, 1 + epsilon]
```

如果 `epsilon = 0.2`，也就是 `[0.8, 1.2]`。这相当于告诉策略：可以学习，但一步别迈太大。

今天重新讲清楚了几个例子：

- 好动作，`advantage > 0`，策略想提高概率；如果比例已经超过 1.2，PPO 不再继续奖励这种过猛更新。
- 好动作，但比例只到 1.1，说明更新幅度正常，可以接受。
- 坏动作，`advantage < 0`，策略应该降低概率；这里同样要防止策略一下子改得过激。

这段复盘很有用，因为后面讨论 Protect、Push、Rescue 时，实际上都绕不开 PPO 的“更新幅度控制”这个底层问题。TC-PPO 不是另起炉灶，而是在 PPO 原本的保守更新机制之外，再加一层基于接触健康状态的调度。

## 稳定域、W 和 c：不是看一次，而是看连续恢复

接着讨论“如何判断进入稳定域”。

最后整理出来的口径是：稳定不是 reward 高，也不是 CTS 偶然高一次，而是接触信号已经连续恢复。

一个更清楚的判断方式是：

```text
active = True 时，
如果 CTS_t >= tau_high，就让 c += 1；
如果 c >= W，就关闭保护；
如果中间 CTS_t < tau_high，c 归零。
```

这里几个符号今天都讲清楚了：

- `CTS_t`：第 t 个检查点的 Contact Timing Stability，接触时序稳定性。
- `tau_low`：保护触发线，低于它说明接触节律已经危险。
- `tau_high`：恢复确认线，高于它才开始考虑退出保护。
- `W`：恢复确认窗口，是超参数，不是实时算出来的统计量。
- `c`：连续恢复计数，真正随训练过程变化。

一个例子很能说明问题：

```text
tau_high = 0.42
W = 2

0.43 -> c = 1
0.39 -> c = 0
0.44 -> c = 1
0.45 -> c = 2 -> 关闭 active
```

这里第 1 次和第 3 次不能加起来算两次，因为中间断了。`c` 统计的是连续恢复，不是累计恢复。

这个细节很小，但对算法解释很关键。否则审稿人会误以为只要一段时间里总共有两次超过 `tau_high` 就能退出保护，那样状态机会很容易抖。

## tau_low 和 tau_high：从固定数值转向数据分位数

今天也重新解释了为什么早期的 `0.35 / 0.42` 不应该被写成拍脑袋的常数。

更稳的写法是：

```text
tau_low  = P5(F)
tau_high = P75(F)
```

其中 `F` 是 warm-up 阶段收集到的一组健康 CTS 值。

这个设计背后的意思是：

- `P5(F)` 是健康训练分布的低尾。只有 CTS 掉到健康低尾之外，才认为真的危险。
- `P75(F)` 是比较稳定的健康上半区。保护开启后，不能刚刚脱离危险就关门，要等恢复到更可靠的稳定水平。

为什么不用一个阈值？因为只用一个阈值会导致门控抖动：

```text
0.419 -> 开保护
0.421 -> 关保护
0.418 -> 又开保护
0.423 -> 又关保护
```

所以需要滞回区间：

```text
tau_low 负责开门
tau_high 负责关门
```

这也让论文里 `0.35` 和 `0.42` 的关系更自然：它们是经验例子，真正原则是从任务自己的 warm-up 分布中估计阈值。

## Push domain：不是“越推越好”

当天讨论了推动域，也就是 Push domain。

Push 的条件被拆成几条：

```text
active = False
CTS_t >= tau_high
R_bar_t >= R_floor
alpha_hat_rel_t < epsilon_prog
h >= H_push
```

它的意思是：

- 当前不在保护状态；
- 接触已经稳定；
- 未塑形回报没有深度崩溃；
- 回报增长趋势已经变慢；
- 稳定状态持续了足够久。

这就把 Push 和 Protect 分开了。Protect 是“先别崩”，Push 是“已经站稳但不进步，所以轻轻推一把”。

今天还特别补了 `W` 和 `H_push` 的区别：

- `W` 判断能不能退出保护；
- `H_push` 判断退出保护以后，稳定够久没有，能不能开始推动探索。

所以一般应该有：

```text
H_push >= W
```

因为 Push 比关保护更激进，应该更保守一点。当天明确提到可以把 `H_push` 设成 3，意思是稳定恢复以后还要再观察几次，不能刚关保护就急着推。

## Recovery 和 Rescue：低回报也要分情况

晚上讨论了一个很重要的边界问题：如果回报不好，该怎么办？

最后把它分成两种情况。

### Recovery：不好，但在涨

判断大概是：

```text
R_bar_t < R_floor
alpha_hat_rel_t >= epsilon_prog
```

这说明当前表现还不够好，但趋势在变好。它像一个人身体还没恢复到理想状态，但已经在慢慢变好。这个时候不要乱加 Push，也不要直接判死刑。

处理方式更像是：

- 继续普通 PPO；
- 不加额外 Push shaping；
- 继续监控 CTS；
- 只有接触稳定性跌破危险线时才进入 Protect。

一句话：Recovery 是低回报学习期，它差，但还活着，还在自己爬坡。

### Rescue：不好，而且不涨

判断大概是：

```text
R_bar_t < R_floor
alpha_hat_rel_t < epsilon_prog
```

这说明表现差，而且学不动。这个时候直接 Push 风险很大，因为策略可能连基本接触或姿态都没建立好。对这样的状态，更合理的是保守的 bootstrap 或部署安全扩展，而不是把它混进当前 Push 结果里。

论文里后来把这件事写成边界说明：Rescue 可以作为未来或部署扩展，但不能伪装成已经实验验证的新模块。

## 识别、判断、预警、保护、挽救：状态机被拉开层次

用户提出了一个很关键的观点：保护域和预警应该区分，顺序应该是：

```text
先识别 -> 再预警 -> 再保护
```

今天进一步把它扩展成更完整的链条：

```text
Detection -> Judgment -> Warning -> Treatment / Protection -> Emergency Rescue Pacer
```

这里每一步含义不同：

- Detection 只负责看指标，比如 CTS、未塑形回报、回报趋势。
- Judgment 不因为一次异常就下结论，而是用连续窗口、滞回阈值和 return trend 判断是不是小概率噪声。
- Warning 是确认有风险，但还没真正崩溃。
- Protection 是真正干预，比如关闭 Push、降低更新激进程度、等待 CTS 连续恢复。
- Rescue Pacer 是保护也救不回来时的紧急扩展，比如回滚 checkpoint、短促强干预、保守恢复。

这个分层让论文从“一个 if 条件触发保护”变成了一个更像真实控制系统的健康状态调节器。

## 人体健康类比：从接触稳定性升级到接触健康调节

晚上用户提出了一个很有启发性的类比：把机器人策略看成一个人的身体。

这个类比大概是：

- normal baseline：正常，但不一定很强；
- healthy / enhanced：健康、更强，任务表现更好；
- unhealthy / collapsed：不健康，策略崩溃或接触失稳。

两条路径也很清楚：

- normal -> healthy：像规律锻炼、健康饮食，让身体更强；
- normal -> unhealthy：像熬夜、抽烟、不运动，最后可能生病甚至崩溃。

对应到算法里：

- CTS、return、trend 是体检指标；
- Warning 是早期风险提示；
- Protect 像治疗和休息；
- Rescue Pacer 像急救或起搏器；
- Push 像训练和强化；
- Push 的停止规则像“不要过度训练”。

这个类比最后推动了论文标题和方法框架的变化。论文不再只是“研究接触稳定性”，而是更偏向：

```text
Contact-Health Regulation
```

也就是接触健康状态调节。

## 正向链条：好上加好也要有剂量控制

用户后来追问：“对于好上加好，有类似步骤吗？”

于是整理出正向链条：

```text
Detection
Readiness Judgment
Opportunity Judgment
Controlled Enhancement
Monitoring
Withdrawal / Consolidation
```

这里最关键的一点是：Push 不是越多越好。

如果策略还在自然变好：

```text
alpha_hat_rel_t >= epsilon_prog
```

那就不要 Push，因为 PPO 自己还能学。

只有策略已经稳定、回报没崩、但进步停住时，才考虑小剂量推动：

```text
R_bar_t >= R_floor
alpha_hat_rel_t < epsilon_prog
CTS_t >= tau_high
```

同时要有停止规则，比如 `H_max`。如果 Push 太久，或者真实未塑形回报没有改善，就应该撤回。否则就像过度锻炼，可能从健康变成不健康。

这一天把 Push 从“额外奖励”讲成了“剂量受控的增强机制”，这是一个很大的表达升级。

## 论文水平评估

晚上还对当前论文水平做了一个比较直接的评估。

结论大概是：这篇不是水文，它有真实问题、有方法、有失败边界，也有一定新意；但现在更像一篇很有潜力、还没完全收敛的长技术报告或期刊扩展稿。

当时给出的判断是：

```text
6.5/10 到 7/10
```

优点是：

- 抓住了 humanoid contact-rich 任务里 PPO 容易崩的真实问题；
- 不是只做 PPO 参数调参，而是用接触信号做在线诊断和调节；
- 承认六通道 support-contact CTS 的适用边界，没有盲目吹成全任务通用；
- soft gating、adaptive threshold、部分任务保护效果比较扎实。

主要风险是：

- 概念太多：CHR、CHS、CTS、CSC、Mode A/B/C、Protect、Hold、Push、Rescue、CUSUM、CTS-GAE、HTS、PBRS 等等都出现了；
- 实验支撑不够均匀；
- Push、Rescue、CUSUM、GAE 更像扩展或未来工作，不能放得比已验证模块还重；
- 正向结果还不够压倒性，需要更稳的叙事和更清楚的边界。

这个评估对后面很有价值：论文可以往“系统框架”走，但必须诚实区分已验证内容和概念扩展。

## 今天形成的术语和判断

| 术语 | 今天整理后的含义 |
|---|---|
| PPO clip | 限制新旧策略概率比，避免策略一步更新过猛 |
| `CTS_t` | 当前检查点的接触时序稳定性 |
| `tau_low` | 保护触发线，建议来自 warm-up CTS 分布的 P5 |
| `tau_high` | 退出保护确认线，建议来自 warm-up CTS 分布的 P75 |
| `W` | 退出保护所需的连续恢复窗口，是超参数 |
| `c` | 连续恢复计数，中间断一次就归零 |
| `H_push` | 进入 Push 前需要持续稳定的窗口数，一般应不小于 `W` |
| `R_floor` | 未塑形回报的最低可接受线，用于避免在深度崩溃时误开 Push |
| `epsilon_prog` | 判断回报趋势是否停滞的进步阈值 |
| Recovery | 回报低，但趋势在涨，应该继续观察和普通学习 |
| Rescue | 回报低，而且不涨，是保护失败或部署安全扩展的候选 |
| CHR | Contact-Health Regulation，接触健康调节框架 |
| CHS | Contact-Health State，接触健康状态变量 |
| Emergency Rescue Pacer | 保护无法恢复时的紧急挽救扩展，不属于当前已验证实验 |
